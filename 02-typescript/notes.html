Variable declarations
  • const = can't reassign (enforced). Use by default.
  • let = can reassign. Use when mutation is needed.
  • var = never use (legacy, broken scoping rules).

TypeScript infers types — annotate when it can't figure it out or for function signatures.

const name: string = "Alice"       // explicit annotation
const city = "Paris"                  // inferred as string — no annotation needed
let count = 0                        // inferred as number

Functions

TypeScript
// Arrow function — the standard
const greet = (
  name: string,
  age: number
): string => {
  return `Hello ${name}, you are ${age}`
}

// Optional param (note the ?)
const format = (
  first: string,
  last?: string
): string => {
  return last ? `${first} ${last}` : first
}
Python equivalent
# Regular function
def greet(
    name: str,
    age: int
) -> str:
    return f"Hello {name}, you are {age}"


# Optional param
def format(
    first: str,
    last: str | None = None
) -> str:
    return f"{first} {last}" if last else first
Key differences from Python
  • Backtick template literals: `Hello ${name}` = Python f-strings
  • ? marks optional params — no default value needed (becomes undefined)
  • No keyword arguments — use objects instead: fn({ name: "Alice", age: 30 })
  • Arrow => syntax is the standard (like Python lambdas, but can be multi-line)

Interfaces & Types

TypeScript — interface
interface User {
  name: string
  age: number
  email?: string  // optional
}
Python — TypedDict
class User(TypedDict):
    name: str
    age: int
    email: NotRequired[str]
// Type alias for unions — like Literal[...] in Python
type Status = "active" | "inactive" | "pending"
interface
  • For object shapes
  • Can be extended: interface Admin extends User
  • Use for most data structures
type
  • For unions, aliases, complex types
  • More flexible but can't be extended
  • Use for union types, utility types
Structural typing: TypeScript cares about shape, not the type name. If an object has all the right fields, it matches the type — even without explicitly declaring it. Python's protocols work similarly.

Arrays

const names: string[] = ["Alice", "Bob"]
const users: User[] = [alice, bob]

Array methods → Python equivalents

.map(fn)
→
[fn(x) for x in list]
.filter(fn)
→
[x for x in list if fn(x)]
.reduce(fn, init)
→
functools.reduce(fn, list, init)
.find(fn)
→
next((x for x in list if fn(x)), None)
// Array methods are chained (like Python list comprehensions but different syntax)
const upperNames = names.map((n) => n.toUpperCase())
const adults = users.filter((u) => u.age >= 30)
const totalAge = users.reduce((sum, u) => sum + u.age, 0)

Union Types & Narrowing

let value: string | number = "hello"

// Narrowing — TS tracks which type it is after a check
if (typeof value === "string") {
  value.toUpperCase()  // TS knows it's string here
}
Narrowing is TypeScript's killer feature vs Python's type system. The compiler tracks which branch you're in and restricts the type automatically. No casting needed.

Generics

TypeScript
type ApiResponse<T> =
  | { success: true; data: T }
  | { success: false; error: string }
Python
from typing import Generic, TypeVar
T = TypeVar("T")

class ApiResponse(Generic[T]):
    success: bool
    data: T | None
    error: str | None

Parameterize types for reuse. You'll see this everywhere in React: useState<User>(), Array<string>, etc.
