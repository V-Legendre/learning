15 — Build & Deploy

Building production bundles with Vite, environment variables, and deploying to Vercel, Netlify, or AWS.

The Build Process

A React app in development is a bunch of TypeScript/JSX files, CSS, and assets that a dev server transforms on the fly. For production, you build everything into optimized static files.

$ pnpm build
vite v5.x building for production...
transforming (142 modules)...
rendering chunks (8)...
computing gzip size (8)...

dist/index.html 0.46 kB | gzip: 0.30 kB
dist/assets/index-a1b2c3.css 14.21 kB | gzip: 3.87 kB
dist/assets/index-d4e5f6.js 148.32 kB | gzip: 47.21 kB
dist/assets/vendor-g7h8i9.js 82.14 kB | gzip: 26.43 kB
Python analogy: Building is like creating a Docker image. You bundle everything into a self-contained artifact that can be deployed anywhere. The dist/ folder is your "image" — just static files (HTML + JS + CSS) that any web server can serve.

Optimization What It Does Why It Matters
Tree shaking Removes functions/modules you imported but never called Only ships code you actually use
Minification Renames variables to single letters, removes whitespace/comments 30-50% smaller files
Code splitting Creates separate chunks for lazy-loaded routes Faster initial page load
Asset hashing Adds content hash to filenames: main.a1b2c3.js Browser caches aggressively. New hash = new file = cache invalidated
CSS extraction Moves CSS out of JS into separate .css files Browser can cache CSS independently, parallel loading

Preview Locally

Before deploying, test the production build locally:

$ pnpm build
$ pnpm preview
Local: http://localhost:4173/
Network: http://192.168.1.42:4173/
Why preview matters: The dev server (pnpm dev) and the production build can behave differently. Environment variables, code splitting, and path resolution may change. Always test with pnpm preview before deploying.

Environment Variables

Vite uses import.meta.env for environment variables. Only variables prefixed with VITE_ are exposed to your frontend code.

.env files

.env ← loaded in all environments
.env.local ← loaded in all, gitignored (your overrides)
.env.production ← loaded only during pnpm build
.env.development ← loaded only during pnpm dev
# .env
VITE_API_URL=http://localhost:3000/api
VITE_APP_NAME=MyApp

# .env.production
VITE_API_URL=https://api.myapp.com
// Usage in code
const apiUrl = import.meta.env.VITE_API_URL
const mode = import.meta.env.MODE     // 'development' or 'production'
const isDev = import.meta.env.DEV     // boolean
const isProd = import.meta.env.PROD   // boolean
NEVER put secrets in frontend env vars! They are embedded in the JavaScript bundle and visible to anyone who opens DevTools. API keys for third-party services, database passwords, auth secrets — all of these must stay on the backend. The VITE_ prefix is an intentional friction: it forces you to acknowledge this code is public.
Python analogy: Same concept as .env files with python-dotenv. The key difference: in Python, env vars stay on the server. In a frontend build, they get baked into the JS files at build time. There's no server to hide them.

Type-safe environment variables

// src/vite-env.d.ts (augment Vite's types)

/// <reference types="vite/client" />

interface ImportMetaEnv {
  readonly VITE_API_URL: string
  readonly VITE_APP_NAME: string
}

interface ImportMeta {
  readonly env: ImportMetaEnv
}

Deploy to Vercel (Recommended)

Vercel is the easiest path from code to deployed app. It was built for frontend frameworks.

1
Push to GitHub — your repo needs to be on GitHub (or GitLab/Bitbucket)
2
Import in Vercel — go to vercel.com → "New Project" → select your repo
3
Configure — Vercel auto-detects Vite. Set framework to "Vite" and build command to pnpm build
4
Set env vars — Add VITE_API_URL and others in the dashboard (Settings → Environment Variables)
5
Deploy — click Deploy. Done. URL: your-app.vercel.app

What you get automatically

Feature Details
Auto deploys Every push to main triggers a production deploy
Preview deploys Every PR gets a unique preview URL (great for code review)
SSL/HTTPS Automatic, free, always on
CDN Your static files are served from edge locations worldwide
SPA routing Automatically rewrites all routes to index.html
Rollbacks One-click rollback to any previous deployment
Python analogy: Vercel is like Heroku but for frontend apps. Push code, it builds and deploys. Preview deploys are like Heroku Review Apps. The big difference: your app is just static files on a CDN, not a running server process.

Other Hosting Options

Platform Strengths When to Use
Netlify Very similar to Vercel. Great forms, functions, identity. Personal preference — Vercel and Netlify are near-equivalent
Cloudflare Pages Edge deployment, free tier, fastest CDN, Workers for serverless Global audience, edge computing needs, cost-sensitive at scale
S3 + CloudFront Full AWS control. You already know this stack. Already on AWS, need fine-grained control, enterprise requirements
GitHub Pages Free, simple, built into GitHub Static docs, personal sites, learning projects (this repo!)
S3 + CloudFront tip: You already know this from DevOps. The only gotcha for SPAs: configure CloudFront to serve index.html for 404 errors (so client-side routing works). Set a custom error response: 404 → /index.html with 200 status.

GitHub Actions CI/CD

For any hosting platform, you'll want CI to run linting and tests before deploying.

# .github/workflows/ci.yml

name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - name: Install pnpm
        uses: pnpm/action-setup@v4
        with:
          version: 9

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'pnpm'

      - name: Install dependencies
        run: pnpm install --frozen-lockfile

      - name: Lint
        run: pnpm lint

      - name: Type check
        run: pnpm tsc --noEmit

      - name: Test
        run: pnpm test

      - name: Build
        run: pnpm build
Vercel integration: If you use Vercel, you don't need a deploy step in CI — Vercel handles it automatically on push. The CI workflow is just for linting, type checking, and testing. If a check fails, Vercel won't deploy the broken build.

Custom Domain

DNS setup for Vercel / Netlify
1
Buy a domain (Cloudflare Registrar, Namecheap, Google Domains)
2
In Vercel: Settings → Domains → Add myapp.com
3
Add DNS records at your registrar:
A record: @ → 76.76.21.21 (Vercel's IP)
CNAME record: www → cname.vercel-dns.com
4
SSL certificate is provisioned automatically (Let's Encrypt)
