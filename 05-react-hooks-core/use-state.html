05 — useState

React's state management primitive: data that changes over time and triggers re-renders.

What Is State?

State is data owned by a component that, when changed, causes the component to re-render (re-execute its function and update the DOM). Regular variables declared with const or let inside a component get destroyed and re-created on every render. State persists between renders.

Regular variable (broken)
const Counter = () => {
  let count = 0  // reset to 0 every render

  const increment = () => {
    count += 1  // mutates, but no re-render
    console.log(count) // logs 1, but UI stays 0
  }

  return <button onClick={increment}>
    {count}
  </button>
}
useState (correct)
const Counter = () => {
  const [count, setCount] = useState(0)

  const increment = () => {
    setCount(prev => prev + 1)
    // triggers re-render, UI updates
  }

  return <button onClick={increment}>
    {count}
  </button>
}
Python analogy
class Counter:
    def __init__(self):
        self.count = 0  # persists between method calls

    def increment(self):
        self.count += 1
        self.render()  # manually trigger UI update

In React, useState is like a class attribute, except changing it automatically triggers a re-render. You never call "render" yourself.

React equivalent
// useState replaces class attributes
// setCount replaces self.count = x + self.render()
const [count, setCount] = useState(0)

// Calling setCount does two things:
// 1. Updates the stored value
// 2. Schedules a re-render

Basic Usage & Array Destructuring

useState returns an array of exactly two elements: the current value and a setter function. Array destructuring gives you both at once.

// Syntax: const [value, setValue] = useState(initialValue)
const [count, setCount] = useState(0)
const [name, setName] = useState("")
const [isOpen, setIsOpen] = useState(false)
const [items, setItems] = useState<string[]>([])

// Without destructuring (never do this, but helps you understand):
const stateArray = useState(0)
const count = stateArray[0]    // current value
const setCount = stateArray[1] // setter function
Naming convention: Always name the pair [thing, setThing]. The setter is set + capitalized variable name. This is a universal React convention, not enforced by the language.

TypeScript Type Parameter

TypeScript infers the type from the initial value. When it can't (e.g., the initial value is null or an empty array), provide the type explicitly using the generic parameter.

CodeInferred TypeNeeds Generic?
useState(0)numberNo
useState("")stringNo
useState(false)booleanNo
useState([])never[]Yes
useState(null)nullYes
 
// When inference works fine:
const [count, setCount] = useState(0)         // number
const [name, setName] = useState("Alice")   // string

// When you NEED the generic type parameter:
const [items, setItems] = useState<string[]>([])
const [user, setUser] = useState<User | null>(null)

// Interface for the User type:
interface User {
  id: number
  name: string
  email: string
}

const [user, setUser] = useState<User | null>(null)
// Now TypeScript knows user is User | null
// You must null-check before accessing user.name

Updater Functions (Critical Pattern)

When the new state depends on the previous state, always pass a function to the setter. This avoids the "stale closure" trap.

Direct value (stale closure bug)
const handleClick = () => {
  setCount(count + 1)
  setCount(count + 1)
  setCount(count + 1)
  // Result: count increases by 1, not 3!
  // All three read the SAME stale `count`
}
Updater function (correct)
const handleClick = () => {
  setCount(prev => prev + 1)
  setCount(prev => prev + 1)
  setCount(prev => prev + 1)
  // Result: count increases by 3
  // Each reads the LATEST pending value
}
Stale closure trap: Inside an event handler, count is captured at the time the handler was created. It does not update mid-function. The updater pattern (prev => prev + 1) always receives the most recent pending value from React's internal queue.

Immutable Updates (Objects & Arrays)

React uses reference equality to detect changes. If you mutate an object/array in place, the reference stays the same and React sees no change. You must create a new object/array every time.

Python analogy: This is like creating a new dict with {**old_dict, "key": "new_value"} instead of old_dict["key"] = "new_value". Same concept, different syntax.

Updating Objects

Mutating (broken, no re-render)
// NEVER do this
user.name = "Bob"
setUser(user)
// Same reference! React ignores it.
Spread operator (new object)
// Create a new object with one field changed
setUser({ ...user, name: "Bob" })
// New reference, React re-renders

Updating Arrays

OperationAvoid (mutates)Use instead (new array)
Add item arr.push(item) [...arr, item]
Remove item arr.splice(i, 1) arr.filter(x => x.id !== id)
Update item arr[i].done = true arr.map(x => x.id === id ? {...x, done: true} : x)
Sort arr.sort() [...arr].sort() 
interface Todo {
  id: number
  text: string
  done: boolean
}

const [todos, setTodos] = useState<Todo[]>([])

// Add
const addTodo = (text: string) => {
  setTodos(prev => [...prev, { id: Date.now(), text, done: false }])
}

// Remove
const removeTodo = (id: number) => {
  setTodos(prev => prev.filter(t => t.id !== id))
}

// Toggle done
const toggleTodo = (id: number) => {
  setTodos(prev => prev.map(t =>
    t.id === id ? { ...t, done: !t.done } : t
  ))
}

State Batching

React batches multiple setState calls within the same event handler into a single re-render. This is a performance optimization — you don't pay for intermediate renders.

const handleSubmit = () => {
  setName("Alice")       // queued
  setAge(25)             // queued
  setIsValid(true)       // queued
  // All three are batched into ONE re-render
}
React 18+: Batching now works everywhere — event handlers, promises, setTimeout, native events. In React 17 and earlier, batching only happened in React event handlers. One less thing to worry about.

Common Mistakes

Mistake 1: Reading state right after setting it
const handleClick = () => {
  setCount(prev => prev + 1)
  console.log(count)  // Still the OLD value!
  // State updates are applied on the NEXT render.
  // `count` in this closure is a snapshot from THIS render.
}

If you need the new value, compute it first:

const handleClick = () => {
  const nextCount = count + 1
  setCount(nextCount)
  console.log(nextCount)  // Use the computed value
}
Mistake 2: Too many separate useState calls
// When you have 5+ related state variables, consider grouping:

// Instead of this:
const [firstName, setFirstName] = useState("")
const [lastName, setLastName] = useState("")
const [email, setEmail] = useState("")
const [age, setAge] = useState(0)

// Consider grouping into one object:
const [form, setForm] = useState({
  firstName: "",
  lastName: "",
  email: "",
  age: 0,
})

// Or use useReducer for complex state logic (covered later)
Mistake 3: Initializing with expensive computation
// BAD: createExpensiveData() runs on EVERY render
const [data, setData] = useState(createExpensiveData())

// GOOD: Pass a function — React calls it only on first render
const [data, setData] = useState(() => createExpensiveData())
// This is called a "lazy initializer"

When to Use useState

Use CaseExample
Simple valuesCounters, toggles, selected tab index
Form inputsText fields, checkboxes, dropdowns
UI stateModal open/closed, sidebar expanded, loading spinner
ListsTodo items, cart items, search results
Nullable objectsCurrently selected user, fetched data before it arrives
Rule of thumb: If it's a simple value or a flat object and the update logic is straightforward (just replace the value), use useState. If you have complex state transitions (multiple fields that must change together based on action types), consider useReducer instead.

Complete Example: Todo List

import { useState } from "react"

interface Todo {
  id: number
  text: string
  done: boolean
}

export const TodoList = () => {
  const [todos, setTodos] = useState<Todo[]>([])
  const [input, setInput] = useState("")

  const addTodo = () => {
    if (!input.trim()) return
    setTodos(prev => [
      ...prev,
      { id: Date.now(), text: input, done: false }
    ])
    setInput("")
  }

  const toggleTodo = (id: number) => {
    setTodos(prev =>
      prev.map(t => t.id === id ? { ...t, done: !t.done } : t)
    )
  }

  const removeTodo = (id: number) => {
    setTodos(prev => prev.filter(t => t.id !== id))
  }

  // Derived state — no need for another useState
  const remaining = todos.filter(t => !t.done).length

  return (
    <div>
      <h2>Todos ({remaining} remaining)</h2>
      <input
        value={input}
        onChange={e => setInput(e.target.value)}
        onKeyDown={e => e.key === "Enter" && addTodo()}
      />
      <button onClick={addTodo}>Add</button>
      <ul>
        {todos.map(todo => (
          <li key={todo.id}>
            <span
              onClick={() => toggleTodo(todo.id)}
              style={{ textDecoration: todo.done ? "line-through" : "none" }}
            >
              {todo.text}
            </span>
            <button onClick={() => removeTodo(todo.id)}>x</button>
          </li>
        ))}
      </ul>
    </div>
  )
}
Key takeaway: remaining is derived from todos — it's computed during render, not stored in its own useState. This is the correct pattern. Never store state that can be computed from other state.
