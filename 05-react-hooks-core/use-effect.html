05 — useEffect

Synchronize your component with external systems. Not a lifecycle method.

Mental Model

The correct way to think about useEffect: "Run this side effect whenever these dependencies change."

Wrong mental model (lifecycle)

"Run on mount, update, and unmount."

This comes from class components. It leads you to write effects that do too much, track irrelevant lifecycle events, and fight the framework.

Correct mental model (sync)

"Synchronize with this external thing whenever these values change."

Each effect has a purpose: keep some external system in sync with your component's state/props.

Python analogy
# Like a context manager that re-runs
# when inputs change

class Effect:
    def __enter__(self):
        # Setup (subscribe, connect, etc.)
        self.conn = connect_websocket(url)

    def __exit__(self, *args):
        # Cleanup (unsubscribe, disconnect)
        self.conn.close()
React useEffect
useEffect(() => {
  // Setup (__enter__)
  const conn = connectWebSocket(url)

  return () => {
    // Cleanup (__exit__)
    conn.close()
  }
}, [url]) // Re-run when url changes

Basic Syntax

import { useEffect } from "react"

useEffect(
  () => {            // Effect function (runs the side effect)
    // ... do something

    return () => {   // Cleanup function (optional)
      // ... undo what you did
    }
  },
  [dep1, dep2]       // Dependency array (when to re-run)
)

Dependency Array Rules

Dependency ArrayWhen Effect RunsUse Case
[a, b] After first render, then whenever a or b changes Most effects: sync when specific values change
[] After first render only One-time setup: connect to WebSocket, start animation
Omitted entirely After every render Rarely useful. Usually a mistake. 
// Runs when `userId` changes — fetches new user data
useEffect(() => {
  const fetchUser = async () => {
    const res = await fetch(`/api/users/${userId}`)
    const data = await res.json()
    setUser(data)
  }
  fetchUser()
}, [userId])

// Runs once after first render — set up event listener
useEffect(() => {
  const handleResize = () => setWidth(window.innerWidth)
  window.addEventListener("resize", handleResize)
  return () => window.removeEventListener("resize", handleResize)
}, [])

// Runs after every render (no dependency array) — almost never want this
useEffect(() => {
  console.log("Rendered!")
})
Dependency rule: Every reactive value (props, state, variables derived from them) used inside the effect must be in the dependency array. The eslint-plugin-react-hooks enforces this. Don't suppress the warning — fix the code instead.

Cleanup Function

Return a function from your effect to clean up. React calls this function:

  • Before the next time the effect runs (when deps change)
  • When the component unmounts (removed from DOM)
// Example: Chat room subscription
useEffect(() => {
  const connection = createConnection(roomId)
  connection.connect()

  return () => {
    // Runs before next effect OR on unmount
    connection.disconnect()
  }
}, [roomId])

// What happens when roomId changes from "general" to "random":
// 1. Cleanup: disconnect from "general"
// 2. Effect: connect to "random"

Common cleanup patterns

Event listeners
useEffect(() => {
  const handler = (e: KeyboardEvent) => {
    if (e.key === "Escape") onClose()
  }
  document.addEventListener("keydown", handler)
  return () => document.removeEventListener("keydown", handler)
}, [onClose])
Intervals / Timers
useEffect(() => {
  const id = setInterval(() => {
    setSeconds(s => s + 1)
  }, 1000)
  return () => clearInterval(id)
}, [])
Abort fetch on cleanup
useEffect(() => {
  const controller = new AbortController()

  fetch(`/api/data/${id}`, {
    signal: controller.signal
  })
    .then(r => r.json())
    .then(data => setData(data))
    .catch(e => {
      if (e.name !== "AbortError") throw e
    })

  return () => controller.abort()
}, [id])
Boolean flag pattern
useEffect(() => {
  let cancelled = false

  fetchData(id).then(data => {
    if (!cancelled) setData(data)
  })

  return () => { cancelled = true }
}, [id])

When NOT to Use useEffect (Critical)

The most common React mistake is overusing useEffect. Many things that feel like "effects" are actually better handled by other patterns.

1. Don't use for derived/computed state
Anti-pattern
const [items, setItems] = useState(list)
const [count, setCount] = useState(0)

// WRONG: syncing derived state
useEffect(() => {
  setCount(items.length)
}, [items])
Just compute it
const [items, setItems] = useState(list)

// RIGHT: compute during render
const count = items.length
2. Don't use for event handling
Anti-pattern
// WRONG: effect watches for state change
const [submitted, setSubmitted] = useState(false)

useEffect(() => {
  if (submitted) {
    sendToApi(formData)
  }
}, [submitted])
Use event handler
// RIGHT: just do it in the handler
const handleSubmit = () => {
  sendToApi(formData)
}
3. Don't use to "sync" two state variables
Anti-pattern
const [firstName, setFirstName] = useState("")
const [fullName, setFullName] = useState("")

useEffect(() => {
  setFullName(firstName + " " + lastName)
}, [firstName, lastName])
Derive during render
const [firstName, setFirstName] = useState("")

// Derive it, don't sync it
const fullName = firstName + " " + lastName
4. Don't use for data transformation
Anti-pattern
const [filtered, setFiltered] = useState([])

useEffect(() => {
  setFiltered(
    items.filter(i => i.active)
  )
}, [items])
Compute or useMemo
// Simple: just compute it
const filtered = items.filter(i => i.active)

// Expensive? Use useMemo
const filtered = useMemo(
  () => items.filter(i => i.active),
  [items]
)

Litmus test: "Am I synchronizing with something outside React?" If yes, useEffect is appropriate. If you're just transforming data or reacting to user events, you don't need it.

Common Mistakes

Infinite loop: effect updates its own dependency
// INFINITE LOOP: setData changes data, which re-triggers effect
useEffect(() => {
  setData(transform(data))  // data is a dependency AND is being set
}, [data])

// FIX: Use updater function or compute during render
const transformed = transform(data)  // compute during render instead
Object/array dependencies that change every render
// BUG: options is a new object every render
const options = { method: "GET", headers: {} }

useEffect(() => {
  fetchData(url, options)
}, [url, options])  // options is "new" every render = infinite loop!

// FIX 1: Move the object inside the effect
useEffect(() => {
  const options = { method: "GET", headers: {} }
  fetchData(url, options)
}, [url])

// FIX 2: Memoize with useMemo (if it must be outside)

Complete Example: Window Size Tracker

import { useState, useEffect } from "react"

export const WindowSize = () => {
  const [size, setSize] = useState({
    width: window.innerWidth,
    height: window.innerHeight,
  })

  useEffect(() => {
    const handleResize = () => {
      setSize({
        width: window.innerWidth,
        height: window.innerHeight,
      })
    }

    window.addEventListener("resize", handleResize)

    // Cleanup: remove listener when component unmounts
    return () => {
      window.removeEventListener("resize", handleResize)
    }
  }, []) // Empty deps = set up once, clean up on unmount

  return (
    <p>
      Window: {size.width} x {size.height}
    </p>
  )
}
Key pattern: The effect subscribes to an external system (browser resize events), and the cleanup unsubscribes. This is the canonical useEffect use case. Everything else should make you question whether you actually need an effect.

React Strict Mode & Double Effects

In development mode with <StrictMode>, React intentionally runs every effect twice (mount → unmount → mount) to help you find missing cleanup functions. If your effect breaks on the second run, you have a cleanup bug.

// If you see your effect running twice in dev, it's intentional.
// Don't remove StrictMode — fix your cleanup instead.
// In production, effects only run once per mount.
