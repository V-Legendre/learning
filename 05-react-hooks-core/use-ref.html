05 — useRef

Mutable container for values that persist between renders without triggering re-renders.

Two Distinct Use Cases

1. Access DOM elements

Get a direct reference to a DOM node. Use it to focus inputs, scroll to elements, measure dimensions, or integrate with non-React libraries.

const inputRef = useRef<HTMLInputElement>(null)

// Attach to element
<input ref={inputRef} />

// Use the DOM node
inputRef.current?.focus()
2. Mutable values without re-render

Store values that persist across renders but don't need to update the UI when they change. Like instance variables on a class.

const renderCount = useRef(0)
renderCount.current += 1
// No re-render when .current changes

const timerId = useRef<number | null>(null)
timerId.current = setInterval(tick, 1000)
Python analogy
class MyComponent:
    def __init__(self):
        # Instance variable — persists between
        # method calls, no "re-render" on change
        self.timer_id = None
        self.render_count = 0
        self.input_element = None

    def render(self):
        self.render_count += 1
        # No automatic UI update from this
React equivalent
const MyComponent = () => {
  // useRef = self.something
  // Persists between renders,
  // no re-render when .current changes
  const timerId = useRef<number | null>(null)
  const renderCount = useRef(0)
  const inputRef = useRef<HTMLInputElement>(null)

  renderCount.current += 1
}

Ref vs State

FeatureuseRefuseState
Persists between rendersYesYes
Triggers re-render on changeNoYes
MutableYes (.current = x)No (must use setter)
Available immediatelyYes (sync)No (next render)
Can hold DOM referencesYesNo
Use for displayed valuesNoYes
Decision rule: If the value is displayed in the UI, use state. If you need to track something internally (timer IDs, previous values, DOM nodes), use a ref.

DOM Access Patterns

Focus an input

import { useRef } from "react"

export const SearchBar = () => {
  const inputRef = useRef<HTMLInputElement>(null)

  const handleClick = () => {
    inputRef.current?.focus()
    // Optional chaining (?.) because ref is null before mount
  }

  return (
    <div>
      <input ref={inputRef} placeholder="Search..." />
      <button onClick={handleClick}>Focus</button>
    </div>
  )
}

Scroll to element

const sectionRef = useRef<HTMLDivElement>(null)

const scrollToSection = () => {
  sectionRef.current?.scrollIntoView({
    behavior: "smooth",
    block: "start",
  })
}

// In JSX:
<div ref={sectionRef}>Target section</div>

Measure element dimensions

const boxRef = useRef<HTMLDivElement>(null)

useEffect(() => {
  if (boxRef.current) {
    const rect = boxRef.current.getBoundingClientRect()
    console.log(`Width: ${rect.width}, Height: ${rect.height}`)
  }
}, [])

Mutable Value Patterns

Track previous value

const usePrevious = <T>(value: T): T | undefined => {
  const ref = useRef<T>()

  useEffect(() => {
    ref.current = value
  }, [value])

  return ref.current // returns old value (before effect runs)
}

// Usage:
const [count, setCount] = useState(0)
const prevCount = usePrevious(count)
// prevCount is the value from the LAST render

Store timer / interval IDs

const Stopwatch = () => {
  const [seconds, setSeconds] = useState(0)
  const [running, setRunning] = useState(false)
  const intervalRef = useRef<number | null>(null)

  const start = () => {
    setRunning(true)
    intervalRef.current = setInterval(() => {
      setSeconds(s => s + 1)
    }, 1000)
  }

  const stop = () => {
    setRunning(false)
    if (intervalRef.current) {
      clearInterval(intervalRef.current)
    }
  }

  // Cleanup on unmount
  useEffect(() => {
    return () => {
      if (intervalRef.current) clearInterval(intervalRef.current)
    }
  }, [])

  return (
    <div>
      <p>{seconds}s</p>
      <button onClick={running ? stop : start}>
        {running ? "Stop" : "Start"}
      </button>
    </div>
  )
}
Why ref for the interval ID? We need to access intervalRef.current in the stop function to clear the interval. If we used state, changing the timer ID would cause an unnecessary re-render. We need to track it, not display it.

forwardRef — Passing Refs to Child Components

By default, you can't pass a ref to a custom component. React doesn't know which DOM element inside the child you want. Use forwardRef to forward the ref to a specific element inside the child.

import { forwardRef, useRef } from "react"

// Child component: accepts a ref and forwards it to its input
interface FancyInputProps {
  label: string
  placeholder?: string
}

export const FancyInput = forwardRef<HTMLInputElement, FancyInputProps>(
  ({ label, placeholder }, ref) => (
    <div>
      <label>{label}</label>
      <input ref={ref} placeholder={placeholder} />
    </div>
  )
)

// Parent component: creates a ref and passes it to child
const Form = () => {
  const inputRef = useRef<HTMLInputElement>(null)

  return (
    <div>
      <FancyInput ref={inputRef} label="Name" />
      <button onClick={() => inputRef.current?.focus()}>
        Focus Input
      </button>
    </div>
  )
}
TypeScript generics order: forwardRef<RefType, PropsType>. The ref type comes first (opposite of what you'd expect). This is a common gotcha.
React 19 update: In React 19, forwardRef is no longer needed. You can accept ref as a regular prop. But if you're on React 18 or earlier, forwardRef is still required.

Quick Reference

PatternCode
DOM reference const ref = useRef<HTMLDivElement>(null)
Mutable value const count = useRef(0)
Timer ID const id = useRef<number | null>(null)
Previous value const prev = useRef(value) + effect to update
Read current value ref.current
Write value ref.current = newValue
Don't read or write ref.current during rendering (i.e., in the component body outside of event handlers or effects). Refs are an "escape hatch" from React's data flow. Reading them during render makes your component's output unpredictable. Use them in event handlers and effects only.
