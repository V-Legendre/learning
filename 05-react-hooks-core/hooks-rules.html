05 — Rules of Hooks

Why hooks have strict rules and what breaks if you violate them.

The Two Rules

Rule 1: Top level only

Never call hooks inside if/else, loops, nested functions, or after early returns. Hooks must always be called in the same order on every render.

Rule 2: Only in React functions

Call hooks only from React function components or custom hooks. Never from regular JavaScript/TypeScript functions, classes, or module scope.

Why: React Tracks Hooks by Call Order

React doesn't use the variable names you give to hooks. Internally, it maintains an ordered list (like a linked list) of hook values for each component instance. On each render, React walks through this list in order, matching each useState/useEffect/etc. call to its stored state.

Normal render (hooks in consistent order)

1
useState(0)
Slot 0: count = 0
→
2
useState("")
Slot 1: name = ""
→
3
useEffect(fn)
Slot 2: effect

On re-render, React expects the same order: slot 0 → slot 1 → slot 2.

Broken render (conditional hook)

Conditional hook (BROKEN)
const Profile = ({ user }: Props) => {
  const [name, setName] = useState("")

  // WRONG: hook inside a condition
  if (user.isAdmin) {
    const [role, setRole] = useState("admin")
  }

  const [email, setEmail] = useState("")
}
What React sees
// Render 1 (isAdmin = true):
// Slot 0: name
// Slot 1: role    <-- exists
// Slot 2: email

// Render 2 (isAdmin = false):
// Slot 0: name
// Slot 1: email   <-- WRONG!
//   React gives email the value of role
//   Everything after is shifted & corrupted
The result: State gets mixed up between different hooks. Your email variable receives the value stored for role. Effects run with wrong dependencies. The component behaves unpredictably.

How to Work Around It

Wrong: hook after condition
const Profile = ({ show }: Props) => {
  if (!show) return null
  // WRONG: hooks below an early return
  const [name, setName] = useState("")
  // ...
}
Right: hooks first, then conditions
const Profile = ({ show }: Props) => {
  // ALL hooks first, unconditionally
  const [name, setName] = useState("")

  // THEN conditionally return
  if (!show) return null

  return <p>{name}</p>
}
Wrong: conditional hook call
const Form = ({ advanced }: Props) => {
  const [name, setName] = useState("")

  if (advanced) {
    // WRONG: conditional useEffect
    useEffect(() => { validate() }, [name])
  }
}
Right: condition inside hook
const Form = ({ advanced }: Props) => {
  const [name, setName] = useState("")

  // Always call the hook, condition inside
  useEffect(() => {
    if (advanced) validate()
  }, [name, advanced])
}
Pattern: Always call all hooks unconditionally at the top of your component. Put conditional logic inside the hook body, never around the hook call itself.

Where Hooks Are Valid

LocationValid?Why
Top level of a function componentYesThis is the intended location
Top level of a custom hookYesCustom hooks are just functions that call hooks
Inside an if blockNoCall order may change between renders
Inside a for loopNoNumber of calls may change
Inside a callback / nested functionNoMay not run on every render
Inside a regular (non-React) functionNoNo component instance to attach to
Inside a class componentNoHooks are for function components only
After an early returnNoWon't execute when condition is met

ESLint Enforcement

Don't rely on memory for these rules. The eslint-plugin-react-hooks package catches violations automatically.

$ pnpm add -D eslint-plugin-react-hooks
// eslint.config.js (flat config)
import reactHooks from "eslint-plugin-react-hooks"

export default [
  {
    plugins: { "react-hooks": reactHooks },
    rules: {
      "react-hooks/rules-of-hooks": "error",        // Checks Rules of Hooks
      "react-hooks/exhaustive-deps": "warn",        // Checks useEffect deps
    },
  },
]
Note: If you set up your project with create-vite or any modern React template, this plugin is usually pre-configured. Verify it's active by intentionally writing a conditional hook and checking that your editor flags it.

Custom Hooks Follow the Same Rules

Custom hooks are just functions whose names start with use. They can call other hooks, and the same ordering rules apply.

// Custom hook — valid because it starts with "use"
const useWindowWidth = (): number => {
  const [width, setWidth] = useState(window.innerWidth)

  useEffect(() => {
    const handler = () => setWidth(window.innerWidth)
    window.addEventListener("resize", handler)
    return () => window.removeEventListener("resize", handler)
  }, [])

  return width
}

// Usage in a component:
const Header = () => {
  const width = useWindowWidth()  // calls hooks internally, same rules apply
  return <header>Width: {width}px</header>
}
Naming matters: If your function's name starts with use, React (and the ESLint plugin) treats it as a hook. It must follow hook rules. If a helper function does NOT call hooks, don't prefix it with use.
