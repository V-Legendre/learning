08 — useMemo & useCallback

Memoization hooks for performance optimization. The React equivalent of @functools.lru_cache — powerful but often overused.

useMemo: Cache Computed Values

useMemo caches the result of a computation. React only re-runs the function when its dependencies change. Without it, the computation runs on every render.

Python: @cached_property
from functools import cached_property

class Report:
    def __init__(self, data: list):
        self.data = data

    @cached_property
    def sorted_data(self):
        # Only computed once,
        # cached for subsequent access
        return sorted(self.data,
            key=lambda x: x["score"])
React: useMemo
const Report = ({ data }: Props) => {
  // Only recomputed when `data` changes
  const sortedData = useMemo(
    () => [...data].sort(
      (a, b) => a.score - b.score
    ),
    [data] // dependency array
  )

  return (
    <ul>
      {sortedData.map(item => (
        <li key={item.id}>{item.name}</li>
      ))}
    </ul>
  )
}

Syntax

const memoizedValue = useMemo(() => expensiveComputation(a, b), [a, b])
//                           ^--- function to compute     ^--- dependencies
// Re-runs ONLY when a or b change. Returns cached result otherwise.

useCallback: Cache Function References

useCallback caches a function itself (its reference), not its result. In JavaScript, () => {} creates a new function object on every render. useCallback returns the same function reference unless dependencies change.

The problem: New function reference on every render
const Parent = () => {
  const [count, setCount] = useState(0)

  // This function is re-created on every render
  // Even though its logic never changes
  const handleClick = () => {
    console.log("clicked")
  }

  // ExpensiveChild receives a new handleClick reference
  // on every render, so it always re-renders
  return <ExpensiveChild onClick={handleClick} />
}
The fix: useCallback stabilizes the reference
const Parent = () => {
  const [count, setCount] = useState(0)

  // Same function reference across renders (unless deps change)
  const handleClick = useCallback(() => {
    console.log("clicked")
  }, []) // empty deps = never changes

  // ExpensiveChild sees the same prop, skips re-render
  return <ExpensiveChild onClick={handleClick} />
}
Key insight: useCallback(fn, deps) is equivalent to useMemo(() => fn, deps). Both cache something — useMemo caches the return value, useCallback caches the function itself.

React.memo: Memoize Components

React.memo wraps a component to skip re-rendering if its props haven't changed (shallow comparison). This is where useCallback becomes useful — without stable prop references, React.memo can't work.

// Memoized component: skips re-render if props are shallowly equal
const ExpensiveList = React.memo(({ items, onSelect }: Props) => {
  console.log("ExpensiveList rendered")  // only logs when props change
  return (
    <ul>
      {items.map(item => (
        <li key={item.id} onClick={() => onSelect(item.id)}>
          {item.name}
        </li>
      ))}
    </ul>
  )
})

The trio: useCallback + React.memo + useMemo

How they work together
const SearchPage = () => {
  const [query, setQuery] = useState("")
  const [sort, setSort] = useState<"name" | "date">("name")
  const { data: allItems } = useQuery({ /* ... */ })

  // useMemo: cache expensive filter + sort (only recomputes when inputs change)
  const filteredItems = useMemo(() => {
    if (!allItems) return []
    return allItems
      .filter(item => item.name.toLowerCase().includes(query.toLowerCase()))
      .sort((a, b) =>
        sort === "name"
          ? a.name.localeCompare(b.name)
          : b.date.getTime() - a.date.getTime()
      )
  }, [allItems, query, sort])

  // useCallback: stable function reference for memoized child
  const handleSelect = useCallback((id: string) => {
    console.log("Selected:", id)
  }, [])

  return (
    <div>
      <input value={query} onChange={e => setQuery(e.target.value)} />
      {/* React.memo child: won't re-render unless filteredItems or handleSelect change */}
      <ItemList items={filteredItems} onSelect={handleSelect} />
    </div>
  )
}

// React.memo: skip re-render if props haven't changed
const ItemList = React.memo(({ items, onSelect }: ItemListProps) => {
  // ... render items
})

When They Actually Help

Worth using
  • Expensive computations: Sorting/filtering large arrays (1000+ items), complex math, tree traversals
  • Stable callbacks for memoized children: useCallback + React.memo together
  • Hook dependencies: When a value/function is used in another hook's dependency array
  • Referential equality: When you need the same object reference to prevent unnecessary effect re-runs
Not worth it (premature optimization)
  • Simple computations: useMemo(() => a + b, [a, b]) — the memoization overhead exceeds the computation cost
  • Functions that aren't passed to memoized children: No point stabilizing a reference nobody checks
  • Primitive props: React.memo already works with strings, numbers, booleans — they compare by value, not reference
  • "Just in case" optimization: If you haven't measured a problem, you're adding complexity for nothing
Pointless: Memoizing cheap operations
// Adding overhead to a trivial operation
const fullName = useMemo(
  () => `${firstName} ${lastName}`,
  [firstName, lastName]
)

// Callback that nobody checks with React.memo
const handleClick = useCallback(() => {
  setOpen(true)
}, [])
// If <button> isn't React.memo'd,
// this useCallback does nothing.
Worthwhile: Real performance wins
// Sorting 10K items — measurably expensive
const sorted = useMemo(
  () => [...largeArray].sort(compareFn),
  [largeArray]
)

// Callback for a React.memo'd child component
// that renders 500 list items
const handleSelect = useCallback(
  (id: string) => setSelected(id),
  []
)
return <MemoizedList onSelect={handleSelect} />

React Compiler (React 19+)

React 19 introduced the React Compiler (formerly "React Forget") which automatically applies memoization at build time. It analyzes your components and inserts useMemo and useCallback where needed.

What the React Compiler does
Before (manual)After (compiler)
You manually wrap values in useMemo Compiler inserts memoization automatically
You manually wrap callbacks in useCallback Compiler stabilizes references where needed
You manually wrap components in React.memo Compiler adds bailout logic per-component
Current status (2026): The React Compiler is stable and enabled by default in new React 19 projects. For existing projects, you can adopt it incrementally. When using the compiler, you can remove most manual useMemo/useCallback calls — the compiler handles them better than humans because it can memoize at a finer granularity.
Should you still learn useMemo/useCallback? Yes. (1) You'll encounter them in existing codebases everywhere. (2) Understanding what they do helps you understand React's rendering model. (3) The compiler doesn't cover every case — knowing when to apply manual optimization is still valuable.

The Rule of Thumb

1
Write code without memoization
React is fast by default. Don't optimize preemptively.
->
2
Measure with React DevTools Profiler
Identify which components re-render unnecessarily.
->
3
Apply targeted memoization
useMemo for expensive computations. useCallback + React.memo for component trees.
->
4
Measure again
Verify the optimization actually helped. Remove if it didn't.

Summary

ToolCachesUse whenPython equivalent

useCallback Function reference Passing callback to React.memo child or as hook dependency No direct equivalent (Python doesn't recreate closures on every call)
React.memo Component render output Component re-renders often with same props @functools.lru_cache on a function's output
React Compiler Everything above, automatically React 19+ projects No equivalent (static analysis at build time)
The biggest mistake: Wrapping everything in useMemo/useCallback "for performance." This adds complexity, can introduce bugs (stale closures from wrong dependencies), and the overhead of memoization can actually make things slower for cheap operations. Profile first. Optimize second.
