08 — useContext

Sharing state across the component tree without prop drilling. React's built-in dependency injection.

The Problem: Prop Drilling

Prop drilling = passing props through intermediate components that don't use them, just to reach a deeply nested component that does. It's tedious, fragile, and pollutes intermediate component APIs.

Prop drilling example: Theme through 4 levels
// Level 1: App owns the theme
const App = () => {
  const [theme, setTheme] = useState<"light" | "dark">("light")
  return <Layout theme={theme} setTheme={setTheme} />
}

// Level 2: Layout doesn't use theme, just passes it through
const Layout = ({ theme, setTheme }: Props) => (
  <div>
    <Sidebar theme={theme} setTheme={setTheme} />
    <Main theme={theme} />
  </div>
)

// Level 3: Sidebar doesn't use theme either
const Sidebar = ({ theme, setTheme }: Props) => (
  <nav>
    <ThemeToggle theme={theme} setTheme={setTheme} />
  </nav>
)

// Level 4: ThemeToggle finally uses it
const ThemeToggle = ({ theme, setTheme }: Props) => (
  <button onClick={() => setTheme(theme === "light" ? "dark" : "light")}>
    {theme === "light" ? "Dark mode" : "Light mode"}
  </button>
)

// Layout and Sidebar are polluted with theme props
// they don't care about. Now imagine 10 levels deep...
Python analogy: Prop drilling is like passing a database connection through every function in a call chain, even if most functions don't use it: def process(db, data) -> def validate(db, item) -> def save(db, item). You'd use dependency injection instead. Context IS React's dependency injection.

The Solution: Context

Context lets you "teleport" data from a provider directly to any consumer below it in the tree, skipping all intermediate components.

The Full Pattern (Production-Ready)

Step 1: Create the context and custom hook
// theme-context.tsx
import { createContext, useContext, useState } from "react"

// 1. Define the context value type
type Theme = "light" | "dark"

type ThemeContextType = {
  theme: Theme
  toggleTheme: () => void
}

// 2. Create context with null default
const ThemeContext = createContext<ThemeContextType | null>(null)

// 3. Custom hook with safety check
export const useTheme = (): ThemeContextType => {
  const context = useContext(ThemeContext)
  if (!context) {
    throw new Error("useTheme must be used within a ThemeProvider")
  }
  return context
}
Step 2: Create the Provider component
// theme-context.tsx (continued)
export const ThemeProvider = ({ children }: { children: React.ReactNode }) => {
  const [theme, setTheme] = useState<Theme>("light")

  const toggleTheme = () => {
    setTheme(prev => prev === "light" ? "dark" : "light")
  }

  return (
    <ThemeContext.Provider value={{ theme, toggleTheme }}>
      {children}
    </ThemeContext.Provider>
  )
}
Step 3: Wrap the tree
// App.tsx
import { ThemeProvider } from "./theme-context"

const App = () => (
  <ThemeProvider>
    <Layout />       {/* No theme props needed! */}
  </ThemeProvider>
)
Step 4: Consume anywhere below
// ThemeToggle.tsx — no prop drilling!
import { useTheme } from "./theme-context"

const ThemeToggle = () => {
  const { theme, toggleTheme } = useTheme()

  return (
    <button onClick={toggleTheme}>
      {theme === "light" ? "Switch to dark" : "Switch to light"}
    </button>
  )
}

// Layout and Sidebar no longer have theme props at all.
// They're completely unaware of theme.

TypeScript: Why null Default + Custom Hook

You might see tutorials use createContext({} as ThemeContextType). That's a lie to the type system. The custom hook pattern is safer.

Bad: Fake default value
// Lies to TypeScript — {} is NOT
// a valid ThemeContextType
const ThemeContext = createContext<ThemeContextType>(
  {} as ThemeContextType
)

// If someone uses useContext outside
// a provider, they get {} — no error,
// just a silent runtime crash later.
Good: null + custom hook
// Honest default — null means "no provider"
const ThemeContext = createContext<
  ThemeContextType | null
>(null)

// Custom hook throws immediately
// with a helpful error message.
const useTheme = () => {
  const ctx = useContext(ThemeContext)
  if (!ctx) throw new Error("...")
  return ctx  // narrowed to ThemeContextType
}

When to Use Context

Good use cases
  • Theme (light/dark mode)
  • Auth (current user, login/logout)
  • Locale / i18n (language, translations)
  • Feature flags
  • Toast/notification system
  • Values that change infrequently
Bad use cases
  • Frequently changing values (form inputs, scroll position, animation frames)
  • Large data sets (lists, tables)
  • Server state (use TanStack Query instead)
  • Everything (don't use Context as a global store)
Performance trap: When context value changes, every component that calls useContext(MyContext) re-renders — even if it only uses a part of the value that didn't change. There's no selector mechanism like in Zustand. This is why context is bad for frequently changing values.

Multiple Contexts

Don't create a single "god context" with everything. Split into focused contexts. Each provider manages one concern.

Bad: God context
type AppContextType = {
  theme: Theme
  user: User | null
  locale: string
  notifications: Notification[]
  cart: CartItem[]
  // ... everything in one blob
}

// Changing theme re-renders EVERY
// consumer, even those that only
// need `user` or `cart`.
Good: Split contexts
// Each concern in its own context
<ThemeProvider>
  <AuthProvider>
    <LocaleProvider>
      <NotificationProvider>
        <App />
      </NotificationProvider>
    </LocaleProvider>
  </AuthProvider>
</ThemeProvider>

// Changing theme only re-renders
// components using useTheme().
Python analogy: Like Flask's g object or Django's middleware system. Each middleware handles one concern (auth, session, CORS), and any view can access what it needs without passing it through the URL handler chain.

Real-World Example: Auth Context

Complete auth context implementation
// auth-context.tsx
import { createContext, useContext, useState, useCallback } from "react"

type User = {
  id: string
  name: string
  email: string
  role: "admin" | "user"
}

type AuthContextType = {
  user: User | null
  isAuthenticated: boolean
  login: (email: string, password: string) => Promise<void>
  logout: () => void
}

const AuthContext = createContext<AuthContextType | null>(null)

export const useAuth = () => {
  const ctx = useContext(AuthContext)
  if (!ctx) throw new Error("useAuth must be used within AuthProvider")
  return ctx
}

export const AuthProvider = ({ children }: { children: React.ReactNode }) => {
  const [user, setUser] = useState<User | null>(null)

  const login = useCallback(async (email: string, password: string) => {
    const response = await fetch("/api/login", {
      method: "POST",
      body: JSON.stringify({ email, password }),
    })
    const userData = await response.json()
    setUser(userData)
  }, [])

  const logout = useCallback(() => {
    setUser(null)
  }, [])

  return (
    <AuthContext.Provider value={{
      user,
      isAuthenticated: user !== null,
      login,
      logout,
    }}>
      {children}
    </AuthContext.Provider>
  )
}

Consuming the auth context

// Any component, anywhere in the tree
const NavBar = () => {
  const { user, isAuthenticated, logout } = useAuth()

  return (
    <nav>
      {isAuthenticated ? (
        <>
          <span>Welcome, {user!.name}</span>
          <button onClick={logout}>Log out</button>
        </>
      ) : (
        <Link to="/login">Log in</Link>
      )}
    </nav>
  )
}

Context Checklist

QuestionAnswer
What does Context solve? Prop drilling — sharing data without passing through every level
What are the 3 pieces? createContext, Provider (wraps tree), useContext (reads value)
What's the best default value? null, with a custom hook that throws if used outside provider
When does it re-render consumers? Every time the value prop of the Provider changes (shallow comparison)
Can I use selectors? No. Every consumer re-renders on any value change. Split contexts to avoid this
Should I put everything in Context? No. Use it for low-frequency global data. For server state, use TanStack Query. For complex client state, use Zustand
Mental model: Context = React's dependency injection. The Provider registers a service. useContext resolves it. The custom hook is the type-safe API consumers use. Keep providers focused on one concern each.
