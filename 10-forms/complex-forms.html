10 — Complex Form Patterns

Multi-step wizards, dynamic fields, watch, setValue, and real-world form architecture.

Multi-Step Forms (Wizards)

Long forms are overwhelming. Break them into steps. The key challenge: persisting state between steps without losing data when navigating back.

Strategy: One form, multiple views

Use a single useForm instance across all steps. Only validate the current step's fields. The form state persists as you navigate between steps.

Step Component Pattern

// types.ts — shared form data type
interface OnboardingForm {
  // Step 1: Account
  email: string
  password: string
  // Step 2: Profile
  name: string
  bio: string
  // Step 3: Preferences
  theme: "light" | "dark"
  notifications: boolean
}
// OnboardingWizard.tsx
import { useForm, FormProvider } from "react-hook-form"
import { zodResolver } from "@hookform/resolvers/zod"

const OnboardingWizard = () => {
  const [step, setStep] = useState(0)

  const methods = useForm<OnboardingForm>({
    resolver: zodResolver(schemas[step]),  // validate current step only
    mode: "onBlur",
  })

  const steps = [<AccountStep />, <ProfileStep />, <PreferencesStep />]

  const next = async () => {
    const valid = await methods.trigger()  // validate current step
    if (valid) setStep(s => s + 1)
  }

  const back = () => setStep(s => s - 1)

  const onSubmit = (data: OnboardingForm) => {
    console.log("Final data:", data)
  }

  return (
    <FormProvider {...methods}>
      <form onSubmit={methods.handleSubmit(onSubmit)}>
        {/* Progress indicator */}
        <div>Step {step + 1} of {steps.length}</div>

        {steps[step]}

        <div>
          {step > 0 && <button type="button" onClick={back}>Back</button>}
          {step < steps.length - 1
            ? <button type="button" onClick={next}>Next</button>
            : <button type="submit">Submit</button>}
        </div>
      </form>
    </FormProvider>
  )
}

Step Components Use FormProvider Context

// AccountStep.tsx — each step accesses the shared form
import { useFormContext } from "react-hook-form"

const AccountStep = () => {
  const { register, formState: { errors } } = useFormContext<OnboardingForm>()

  return (
    <div>
      <input {...register("email")} placeholder="Email" />
      {errors.email && <span>{errors.email.message}</span>}

      <input {...register("password")} type="password" />
      {errors.password && <span>{errors.password.message}</span>}
    </div>
  )
}
Python analogy: FormProvider is like Flask's g object or a shared context manager — it makes form state available to any nested component without prop drilling.

Dynamic Fields with useFieldArray

For lists of items where users can add/remove entries — think: "add another address", "add team member", "add line item".

import { useForm, useFieldArray } from "react-hook-form"

interface InvoiceForm {
  client: string
  items: {
    description: string
    quantity: number
    price: number
  }[]
}

const InvoiceEditor = () => {
  const { control, register, handleSubmit } = useForm<InvoiceForm>({
    defaultValues: {
      client: "",
      items: [{ description: "", quantity: 1, price: 0 }],
    },
  })

  const { fields, append, remove, move } = useFieldArray({
    control,
    name: "items",  // must match the array field name
  })

  return (
    <form onSubmit={handleSubmit(console.log)}>
      <input {...register("client")} placeholder="Client name" />

      {fields.map((field, index) => (
        <div key={field.id}>  {/* useFieldArray provides stable IDs */}
          <input {...register(`items.${index}.description`)} />
          <input {...register(`items.${index}.quantity`, { valueAsNumber: true })} />
          <input {...register(`items.${index}.price`, { valueAsNumber: true })} />
          <button type="button" onClick={() => remove(index)}>Remove</button>
        </div>
      ))}

      <button type="button" onClick={() => append({
        description: "", quantity: 1, price: 0
      })}>
        Add Item
      </button>

      <button type="submit">Save Invoice</button>
    </form>
  )
}
MethodWhat it doesExample
append(obj)Add item at endappend({ description: "", quantity: 1, price: 0 })
prepend(obj)Add item at startprepend({ description: "First" })
remove(index)Remove by indexremove(2)
move(from, to)Reorder itemsmove(0, 3)
swap(a, b)Swap two itemsswap(1, 2)
insert(index, obj)Insert at positioninsert(1, { ... })
replace(items[])Replace entire arrayreplace(newItems)
Key rule: Always use field.id as the key in your map — NOT the array index. useFieldArray generates stable IDs. Using index as key causes input values to shift when items are removed.

Watching Field Values

watch subscribes to field changes — useful for dependent fields, conditional rendering, and live previews.

const { register, watch } = useForm<ShippingForm>()

// Watch a single field
const country = watch("country")

// Watch multiple fields
const [firstName, lastName] = watch(["firstName", "lastName"])

// Watch all fields (re-renders on ANY change — use sparingly)
const allValues = watch()

Common Use Cases

Dependent Dropdowns
const country = watch("country")

// City options depend on country
const cities = useMemo(
  () => getCitiesFor(country),
  [country]
)

<select {...register("country")}>
  <option value="US">USA</option>
  <option value="FR">France</option>
</select>

<select {...register("city")}>
  {cities.map(c => (
    <option key={c} value={c}>{c}</option>
  ))}
</select>
Conditional Fields
const hasDiscount = watch("hasDiscount")

<label>
  <input
    type="checkbox"
    {...register("hasDiscount")}
  />
  Apply discount?
</label>

{hasDiscount && (
  <input
    {...register("discountCode")}
    placeholder="Enter code"
  />
)}
Performance tip: watch causes re-renders on every change of the watched field. For performance-sensitive cases (like a live character count), consider useWatch which can be isolated to a child component.

setValue, reset, and Programmatic Control

setValue — Set a Single Field

const { setValue } = useForm<ProfileForm>()

// Set a field value programmatically
setValue("name", "Alice")

// With options
setValue("name", "Alice", {
  shouldValidate: true,   // trigger validation after setting
  shouldDirty: true,      // mark field as dirty
  shouldTouch: true,      // mark field as touched
})

reset — Reset Entire Form

const { reset } = useForm<ProfileForm>()

// Reset to default values
reset()

// Reset to specific values (e.g., from API response)
reset({
  name: user.name,
  email: user.email,
  bio: user.bio,
})

// Common pattern: populate form from API data
const { data: user } = useQuery({ queryKey: ["user"], queryFn: fetchUser })

useEffect(() => {
  if (user) reset(user)  // populate form when data arrives
}, [user, reset])
Don't: manual state sync
// Manually setting every field
useEffect(() => {
  if (user) {
    setValue("name", user.name)
    setValue("email", user.email)
    setValue("bio", user.bio)
    // easy to miss fields...
  }
}, [user])
Do: reset with full object
// One call resets everything
useEffect(() => {
  if (user) reset(user)
}, [user, reset])

// Cleaner, can't miss fields,
// resets dirty/touched state too

Nested Objects

React Hook Form supports dot notation for nested data structures.

interface OrderForm {
  customer: {
    name: string
    email: string
  }
  shipping: {
    street: string
    city: string
    zip: string
  }
}

// Register with dot notation
<input {...register("customer.name")} />
<input {...register("customer.email")} />
<input {...register("shipping.street")} />
<input {...register("shipping.city")} />
<input {...register("shipping.zip")} />

// Access nested errors
{errors.shipping?.street && <span>{errors.shipping.street.message}</span>}

// Submit gives you the nested object:
// { customer: { name: "Alice", email: "..." }, shipping: { street: "...", ... } }
Python analogy: Like nested Pydantic models — class Order(BaseModel): customer: Customer; shipping: Address. The dot notation is how RHF flattens nested structures for form fields.

Dirty Checking and Navigation Guards

Prevent users from accidentally leaving a form with unsaved changes.

const { formState: { isDirty } } = useForm()

// Warn before browser navigation (refresh, close tab)
useEffect(() => {
  const handler = (e: BeforeUnloadEvent) => {
    if (isDirty) {
      e.preventDefault()
      // Browser shows "Leave site?" dialog
    }
  }
  window.addEventListener("beforeunload", handler)
  return () => window.removeEventListener("beforeunload", handler)
}, [isDirty])

// For in-app navigation (TanStack Router):
// Use the router's beforeLoad or navigation blocking API
Form StateWhat it tells youUse case
isDirtyAny field changed from default?Enable/disable save button, navigation guard
dirtyFieldsWhich specific fields changedPartial updates — only send changed fields to API
isSubmittingForm submission in progressDisable submit button, show spinner
isSubmitSuccessfulLast submit succeededShow success message, reset form
isValidAll validation passingEnable submit button (with mode: "onChange")
touchedFieldsWhich fields user interacted withShow errors only after user touches a field

File Uploads

interface UploadForm {
  avatar: FileList
  name: string
}

const AvatarUpload = () => {
  const { register, handleSubmit, watch } = useForm<UploadForm>()

  const avatar = watch("avatar")

  const onSubmit = (data: UploadForm) => {
    const formData = new FormData()
    formData.append("avatar", data.avatar[0])
    formData.append("name", data.name)
    // POST formData to API
  }

  return (
    <form onSubmit={handleSubmit(onSubmit)}>
      <input {...register("avatar", {
        validate: {
          size: (files) =>
            !files[0] || files[0].size < 5_000_000 || "Max 5MB",
          type: (files) =>
            !files[0] || files[0].type.startsWith("image/") || "Images only",
        }
      })} type="file" accept="image/*" />

      {avatar?.[0] && (
        <img src={URL.createObjectURL(avatar[0])} alt="Preview" />
      )}

      <button type="submit">Upload</button>
    </form>
  )
}

Real-World Example: Settings Page

Combining multiple patterns — API-populated defaults, nested objects, dirty checking, and partial updates.

const settingsSchema = z.object({
  profile: z.object({
    name: z.string().min(1, "Required"),
    bio: z.string().max(500),
  }),
  notifications: z.object({
    email: z.boolean(),
    push: z.boolean(),
    frequency: z.enum(["instant", "daily", "weekly"]),
  }),
})

type Settings = z.infer<typeof settingsSchema>

const SettingsPage = () => {
  const { data: settings } = useQuery({
    queryKey: ["settings"],
    queryFn: fetchSettings,
  })

  const mutation = useMutation({ mutationFn: updateSettings })

  const { register, handleSubmit, reset, formState: { isDirty, dirtyFields } } =
    useForm<Settings>({
      resolver: zodResolver(settingsSchema),
    })

  // Populate form when API data arrives
  useEffect(() => {
    if (settings) reset(settings)
  }, [settings, reset])

  const onSubmit = (data: Settings) => {
    // Only send changed fields (partial update)
    mutation.mutate(data, {
      onSuccess: () => reset(data),  // reset dirty state after save
    })
  }

  return (
    <form onSubmit={handleSubmit(onSubmit)}>
      <h2>Profile</h2>
      <input {...register("profile.name")} />
      <textarea {...register("profile.bio")} />

      <h2>Notifications</h2>
      <label>
        <input type="checkbox" {...register("notifications.email")} />
        Email notifications
      </label>

      <button type="submit" disabled={!isDirty}>
        Save Changes
      </button>
    </form>
  )
}
Pattern summary: API data → reset(data) on load. User edits → isDirty enables save. Submit → reset(data) after success clears dirty state. Navigation guard uses isDirty to warn about unsaved changes.
