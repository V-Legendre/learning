06 — Tailwind Component Patterns

How to use Tailwind effectively in React: taming long class strings and building reusable components.

The Problem: Long Class Strings

Once you start adding responsive, hover, focus, and dark mode variants, class strings get long. This is normal and not inherently bad, but it needs management.

{/* This is a real-world button. It's fine, but hard to scan. */}
<button className="inline-flex items-center justify-center rounded-lg bg-blue-500 px-4 py-2 text-sm font-medium text-white shadow-sm hover:bg-blue-600 focus:outline-none focus:ring-2 focus:ring-blue-500 focus:ring-offset-2 active:scale-95 disabled:opacity-50 disabled:cursor-not-allowed transition-all duration-150">
  Save Changes
</button>
The solution is NOT to go back to CSS files

In Tailwind, the answer to "this class string is too long" is extract a React component, not create a CSS class. The component encapsulates the styles and gives you a clean API through props.

The cn() Utility: Conditional Class Merging

Before building components, you need a helper to conditionally join class names and handle Tailwind conflicts. The standard pattern combines two libraries: clsx (conditional joining) and tailwind-merge (deduplication).

Installation

$ pnpm add clsx tailwind-merge

Create the helper

// src/lib/utils.ts
import { clsx, type ClassValue } from "clsx"
import { twMerge } from "tailwind-merge"

export const cn = (...inputs: ClassValue[]): string => {
  return twMerge(clsx(inputs))
}

What each library does

clsx: conditional joining
import { clsx } from "clsx"

clsx("px-4", "py-2")
// "px-4 py-2"

clsx("px-4", isActive && "bg-blue-500")
// "px-4 bg-blue-500" (if isActive)
// "px-4" (if !isActive)

clsx({
  "bg-red-500": hasError,
  "bg-green-500": isSuccess,
})
tailwind-merge: deduplication
import { twMerge } from "tailwind-merge"

twMerge("px-4 px-6")
// "px-6" (last one wins)

twMerge("bg-red-500 bg-blue-500")
// "bg-blue-500" (removes conflict)

// Without twMerge, both classes stay
// and the one later in CSS wins
// (unpredictable!)

Using cn() in practice

import { cn } from "@/lib/utils"

// Conditional classes
<div className={cn(
  "px-4 py-2 rounded-lg",          // always applied
  isActive && "bg-blue-500 text-white",  // only if active
  isDisabled && "opacity-50 cursor-not-allowed",
)}>

// Override defaults from parent
<div className={cn(
  "p-4 bg-gray-800",  // component defaults
  className,            // parent overrides — tailwind-merge resolves conflicts
)}>
Why cn() matters: Without tailwind-merge, if a parent passes className="p-6" to override p-4, both classes stay in the DOM and the result depends on CSS source order (unreliable). tailwind-merge removes the conflicting p-4, leaving only p-6.

The Variant Pattern: Building a Button

The most important Tailwind + React pattern: a component with typed variant props that map to different Tailwind classes.

// src/components/Button.tsx
import { cn } from "@/lib/utils"

interface ButtonProps extends React.ButtonHTMLAttributes<HTMLButtonElement> {
  variant?: "primary" | "secondary" | "danger" | "ghost"
  size?: "sm" | "md" | "lg"
}

const variantStyles: Record<string, string> = {
  primary: "bg-blue-500 text-white hover:bg-blue-600 focus:ring-blue-500",
  secondary: "bg-gray-700 text-gray-100 hover:bg-gray-600 focus:ring-gray-500",
  danger: "bg-red-500 text-white hover:bg-red-600 focus:ring-red-500",
  ghost: "bg-transparent text-gray-400 hover:text-white hover:bg-gray-800",
}

const sizeStyles: Record<string, string> = {
  sm: "px-3 py-1.5 text-xs",
  md: "px-4 py-2 text-sm",
  lg: "px-6 py-3 text-base",
}

export const Button = ({
  variant = "primary",
  size = "md",
  className,
  children,
  ...props
}: ButtonProps) => {
  return (
    <button
      className={cn(
        // Base styles (always applied)
        "inline-flex items-center justify-center rounded-lg font-medium",
        "focus:outline-none focus:ring-2 focus:ring-offset-2",
        "disabled:opacity-50 disabled:cursor-not-allowed",
        "transition-all duration-150 active:scale-95",
        // Variant + size
        variantStyles[variant],
        sizeStyles[size],
        // Allow parent to override
        className,
      )}
      {...props}
    >
      {children}
    </button>
  )
}

Usage

// Default: primary, medium
<Button>Save</Button>

// Danger variant, large size
<Button variant="danger" size="lg">Delete Account</Button>

// Ghost with custom override
<Button variant="ghost" className="mt-4">Cancel</Button>

// Disabled (native HTML attribute, passed via ...props)
<Button disabled>Loading...</Button>
Key pattern: extends React.ButtonHTMLAttributes<HTMLButtonElement> lets the component accept all native button props (onClick, disabled, type, etc.) without listing them manually. The spread {...props} passes them to the underlying <button>.

When to Extract a Component

Extract when:
  • You're copying the same class string 3+ times
  • The element has variants (sizes, colors, states)
  • It has complex conditional styling logic
  • It's a recognizable UI pattern (Button, Card, Badge, Input)
Don't extract when:
  • It's used only once or twice
  • It's just a layout wrapper (flex gap-4)
  • Extraction would create a component with only className and children props
  • You're trying to avoid long class strings (that's fine)

Common Component Examples

Card

interface CardProps {
  children: React.ReactNode
  className?: string
}

export const Card = ({ children, className }: CardProps) => (
  <div className={cn(
    "rounded-xl border border-gray-800 bg-gray-900 p-6",
    className,
  )}>
    {children}
  </div>
)

export const CardTitle = ({ children, className }: CardProps) => (
  <h3 className={cn("text-lg font-semibold text-white mb-2", className)}>
    {children}
  </h3>
)

// Usage:
<Card>
  <CardTitle>Settings</CardTitle>
  <p className="text-gray-400">Configure your account.</p>
</Card>

Badge

interface BadgeProps {
  children: React.ReactNode
  color?: "gray" | "green" | "red" | "blue" | "yellow"
  className?: string
}

const colorStyles: Record<string, string> = {
  gray: "bg-gray-800 text-gray-300",
  green: "bg-green-900/50 text-green-400",
  red: "bg-red-900/50 text-red-400",
  blue: "bg-blue-900/50 text-blue-400",
  yellow: "bg-yellow-900/50 text-yellow-400",
}

export const Badge = ({ children, color = "gray", className }: BadgeProps) => (
  <span className={cn(
    "inline-flex items-center rounded-full px-2.5 py-0.5 text-xs font-medium",
    colorStyles[color],
    className,
  )}>
    {children}
  </span>
)

// Usage:
<Badge color="green">Active</Badge>
<Badge color="red">Error</Badge>
<Badge>Draft</Badge>

Input

interface InputProps extends React.InputHTMLAttributes<HTMLInputElement> {
  label?: string
  error?: string
}

export const Input = ({ label, error, className, id, ...props }: InputProps) => (
  <div>
    {label && (
      <label
        htmlFor={id}
        className="block text-sm font-medium text-gray-300 mb-1.5"
      >
        {label}
      </label>
    )}
    <input
      id={id}
      className={cn(
        "block w-full rounded-lg border bg-gray-900 px-3 py-2 text-sm text-white",
        "placeholder:text-gray-500",
        "focus:outline-none focus:ring-2 focus:ring-offset-2 focus:ring-offset-gray-950",
        error
          ? "border-red-500 focus:ring-red-500"
          : "border-gray-700 focus:ring-blue-500",
        className,
      )}
      {...props}
    />
    {error && (
      <p className="mt-1.5 text-xs text-red-400">{error}</p>
    )}
  </div>
)

// Usage:
<Input label="Email" type="email" placeholder="you@example.com" />
<Input label="Name" error="Name is required" />

Anti-Patterns to Avoid

1. Don't use @apply
@apply defeats the purpose
/* Don't do this */
.btn-primary {
  @apply inline-flex items-center
    rounded-lg bg-blue-500 px-4 py-2
    text-sm font-medium text-white;
}
/* You just recreated traditional CSS
   with extra steps. The class string is
   still there, just hidden in CSS. */
Extract a component instead
// Do this instead
export const Button = ({ children }: Props) => (
  <button className="inline-flex items-center
    rounded-lg bg-blue-500 px-4 py-2
    text-sm font-medium text-white"
  >
    {children}
  </button>
)
// Encapsulated, reusable, typed props

The one acceptable use of @apply: styling third-party HTML you don't control (e.g., Markdown rendered by a CMS).

2. Don't mix CSS Modules and Tailwind

Pick one approach. Using both means two systems to maintain, conflicting specificity rules, and confusion about where styles live. Tailwind is designed to be your entire styling solution.

3. Don't use inconsistent spacing
Inconsistent
<div className="p-3">...</div>
<div className="p-5">...</div>
<div className="p-7">...</div>
{/* Slightly different padding everywhere */}
Consistent
<div className="p-4">...</div>
<div className="p-4">...</div>
<div className="p-6">...</div>
{/* Stick to 4, 6, 8 for most padding */}

Pick a small set of spacing values and stick with them. Most apps use 2, 4, 6, 8, 12 for padding/margin and 3, 4, 6, 8 for gap.

Organizing Tailwind Classes

When class strings are long, use a consistent order. The Tailwind Prettier plugin sorts them automatically, but here's the mental model:

OrderCategoryExamples
1Layout / Displayflex, grid, block, hidden
2Position / Overflowrelative, absolute, overflow-hidden
3Sizingw-full, h-10, max-w-lg
4Flex/Grid childrenitems-center, justify-between, gap-4
5Spacingp-4, px-6, mt-2, mb-4
6Border / Rounded / Shadowborder, rounded-lg, shadow-md
7Backgroundbg-gray-900
8Typographytext-sm, font-bold, text-gray-400
9States / Transitionshover:, focus:, transition
$ pnpm add -D prettier-plugin-tailwindcss

The Prettier plugin auto-sorts your classes on save. No need to think about ordering manually.

Recommended File Structure

src/
  components/
    ui/               // Primitive UI components
      Button.tsx
      Card.tsx
      Badge.tsx
      Input.tsx
      index.ts         // re-exports all UI components
    features/         // Feature-specific components
      UserCard.tsx
      TodoItem.tsx
  lib/
    utils.ts          // cn() helper lives here
  index.css           // @import "tailwindcss" + @theme
Pattern: ui/ folder contains generic, reusable components with no business logic (Button, Card, Badge). features/ contains app-specific components that compose ui/ components. This separation keeps your design system clean and your feature code focused.
