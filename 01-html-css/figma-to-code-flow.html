What You Extract from Figma

When you click on any element in Figma's inspect panel, you see:

Visual Properties
  • Exact colors (hex, RGB, or design tokens)
  • Font family, size, weight, line-height
  • Border radius, shadows, opacity
  • Spacing (padding and gaps between elements)
  • Widths and heights (fixed or relative)
Layout Information
  • Auto Layout = Flexbox (direction, gap, alignment)
  • Constraints (how elements resize)
  • Component hierarchy (what nests inside what)
  • States (hover, active, disabled variants)
  • Responsive behavior hints
Assets You Export
  • Icons (as SVG — vector, scales perfectly)
  • Images / illustrations
  • Logos in various sizes
  • Nothing else — you rebuild the rest in code
What You DON'T Get
  • Working code (Figma's code tab is mostly useless)
  • Responsive behavior (you decide how it adapts)
  • Animations / transitions (you implement these)
  • Real data / API integration
  • Interactive logic (what happens on click, etc.)

The Key Mapping: Figma → CSS

Figma's "Auto Layout" is literally Flexbox. When a designer sets up Auto Layout in Figma with direction, gap, padding, and alignment — they're describing a flex container. Your job is to recognize this and translate it.

// What you write based on that inspect

interface CardProps {
  title: string
  description: string
}

export const Card = ({ title, description }: CardProps) => {
  return (
    <div className="
      flex flex-col      {/* Auto Layout: vertical */}
      gap-3              {/* gap: 12px */}
      p-4                {/* padding: 16px */}
      w-80               {/* width: 320px */}
      rounded-xl         {/* border-radius: 12px */}
      bg-slate-800       {/* #1E293B */}
      shadow-md          {/* the shadow */}
    ">
      <h2 className="text-lg font-semibold">
        {title}
      </h2>
      <p className="text-sm text-slate-400">
        {description}
      </p>
    </div>
  )
}

The Reality: It's a Translation Job

You're not copying — you're translating. Figma gives you the visual spec. Your job is to decompose it into reusable components, pick the right layout approach (flex/grid), use your design system's tokens (not raw hex values), handle states the designer didn't show (loading, empty, error), and make it responsive across screen sizes.

Typical process for a new screen:

1 Study the design — understand the layout grid, repeating patterns, component hierarchy
2 Identify components — "this card repeats, that's a component. This header is shared across pages."
3 Build bottom-up — start with the smallest components (Button, Badge, Avatar), compose them into bigger ones (Card, Navbar, Sidebar)
4 Assemble the page — use Grid for page layout, Flexbox inside components
5 Compare & iterate — put Figma and browser side by side, tweak until they match

AI-assisted workflow (what's changing)

Tools like Figma MCP + Claude can now read designs directly and generate initial component code. This doesn't eliminate the developer — you still need to integrate with real data, handle edge cases, ensure accessibility, and make architectural decisions. But the "inspect values and type CSS" step is increasingly automated.
