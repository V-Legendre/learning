07 — Conditional Rendering

All the ways to show, hide, and switch UI based on state. In Python you have if/else and ternaries — React has the same tools, but inside JSX.

Early Return

Check conditions at the top of your component and return early. The most readable pattern when a component has preconditions (loading, error, empty state) before showing the main UI.

Python: Early return in a function
def get_user_display(user: User | None) -> str:
    if user is None:
        return "Loading..."
    if not user.is_active:
        return "Account disabled"
    return f"Welcome, {user.name}"
React: Early return in a component
const UserProfile = ({ user }: Props) => {
  if (!user) return <Spinner />
  if (!user.isActive) return <Disabled />

  return (
    <div>
      <h2>{user.name}</h2>
      <p>{user.email}</p>
    </div>
  )
}
Best for: Loading states, error states, auth guards, empty data checks. Keep the "happy path" at the bottom of the function, unindented.

&& Operator (Logical AND)

Render something or nothing. If the left side is truthy, React renders the right side. If falsy, React skips it.

Basic usage
const Notification = ({ count }: { count: number }) => (
  <div>
    <h1>Dashboard</h1>
    {count > 0 && (
      <span className="badge">{count} new</span>
    )}
  </div>
)
GOTCHA: The zero trap. JavaScript's && returns the first falsy value, not true/false. React renders 0 and NaN as text. This is the single most common conditional rendering bug.
Bug: Renders "0" on screen
// When count is 0:
// 0 && <Badge /> evaluates to 0
// React renders the number 0!

{count && <Badge count={count} />}

// User sees: "0" in the DOM
Fix: Use boolean expression
// count > 0 is always a boolean
// false && anything => nothing rendered

{count > 0 && <Badge count={count} />}

// Or use Boolean():
{Boolean(count) && <Badge count={count} />}

// Or double negation:
{!!count && <Badge count={count} />}
Falsy values in JavaScript — which ones React renders

Ternary Operator

Render one thing OR another. The either/or pattern. Works inline in JSX.

Python ternary
label = "Admin" if user.is_admin else "User"
JSX ternary
<span>{user.isAdmin ? "Admin" : "User"}</span>
Common pattern: Loading vs Content
const Dashboard = () => {
  const { data, isLoading } = useQuery({ queryKey: ["stats"], queryFn: fetchStats })

  return (
    <div>
      <h1>Dashboard</h1>
      {isLoading ? (
        <Skeleton lines={5} />
      ) : (
        <StatsTable data={data} />
      )}
    </div>
  )
}
Avoid nested ternaries. They're hard to read. If you have more than 2 branches, use early returns or the object lookup pattern (below).
Bad: Nested ternaries
{status === "loading"
  ? <Spinner />
  : status === "error"
    ? <Error />
    : status === "empty"
      ? <Empty />
      : <Content />}

// Unreadable. Don't do this.
Good: Use alternatives
// Option 1: Early returns (top of component)
if (status === "loading") return <Spinner />
if (status === "error") return <Error />
if (status === "empty") return <Empty />
return <Content />

// Option 2: Object lookup (next section)

Object Lookup / Switch Pattern

Map a value to a component using an object or switch. Much cleaner than nested ternaries when you have multiple branches. This is the React equivalent of Python's match/case.

Python match/case
match status:
    case "loading":
        return render_spinner()
    case "error":
        return render_error(msg)
    case "success":
        return render_data(data)
    case _:
        return render_empty()
React: Object lookup
const statusMap: Record<Status, React.ReactNode> = {
  loading: <Spinner />,
  error: <ErrorBanner message={msg} />,
  success: <DataTable data={data} />,
  empty: <EmptyState />,
}

return <div>{statusMap[status]}</div>
Real example: Status badge with object lookup
type OrderStatus = "pending" | "processing" | "shipped" | "delivered" | "cancelled"

const statusConfig: Record<OrderStatus, { label: string; color: string }> = {
  pending:    { label: "Pending",    color: "gray" },
  processing: { label: "Processing", color: "blue" },
  shipped:    { label: "Shipped",    color: "orange" },
  delivered:  { label: "Delivered",  color: "green" },
  cancelled:  { label: "Cancelled",  color: "red" },
}

const StatusBadge = ({ status }: { status: OrderStatus }) => {
  const { label, color } = statusConfig[status]
  return <span className={`badge badge--${color}`}>{label}</span>
}
TypeScript bonus: Using Record<Status, ...> ensures you handle every variant. If you add a new status to the union, TypeScript will error until you add it to the map. Exhaustiveness checking for free.

Loading / Error / Success Pattern

The most common real-world pattern. Almost every data-driven component goes through these three states. TanStack Query gives you this pattern built-in.

With TanStack Query (the standard approach)
const UserList = () => {
  const { data, isLoading, isError, error } = useQuery({
    queryKey: ["users"],
    queryFn: fetchUsers,
  })

  // Early returns for each state
  if (isLoading) return <UserListSkeleton />
  if (isError) return <ErrorMessage error={error} />
  if (data.length === 0) return <EmptyState message="No users found" />

  // Happy path: we have data
  return (
    <ul>
      {data.map(user => (
        <li key={user.id}>{user.name}</li>
      ))}
    </ul>
  )
}
Python analogy: Like a standard API handler pattern: try/except/else. First check for loading (pending), then errors (except), then handle the success case (else).

Returning null

A component can return null to render absolutely nothing. The component still exists in the tree and its hooks still run — it just produces no DOM output.

const AdminBanner = ({ user }: { user: User }) => {
  if (!user.isAdmin) return null   // renders nothing

  return (
    <div className="admin-banner">
      You have admin privileges
    </div>
  )
}
Important: Returning null doesn't unmount the component. Hooks still execute. If you want to completely remove a component (and its hooks), conditionally render it from the parent: {isAdmin && <AdminBanner />}.

Which Approach When?

PatternSyntaxBest ForWatch Out
Early return if (...) return <X /> Preconditions: loading, error, auth guards Only works at top level of component, not inside JSX
&& {flag && <X />} Show or hide a single element 0 and NaN render as text. Use boolean &&
Ternary {flag ? <A /> : <B />} Either/or within JSX Don't nest. Max one level deep
Object lookup {map[key]} Multiple branches based on a string/enum Define outside the component or memoize
null return null Component decides itself to render nothing Hooks still run. Not the same as unmounting

Putting It All Together

Real-world component using multiple patterns
type Props = {
  userId: string
  showActions?: boolean
}

const UserProfile = ({ userId, showActions = true }: Props) => {
  const { data: user, isLoading, isError, error } = useQuery({
    queryKey: ["user", userId],
    queryFn: () => fetchUser(userId),
  })

  // Early return: loading state
  if (isLoading) return <ProfileSkeleton />

  // Early return: error state
  if (isError) return <Alert variant="error">{error.message}</Alert>

  // Object lookup: role badge
  const roleBadge: Record<string, string> = {
    admin: "bg-red",
    editor: "bg-blue",
    viewer: "bg-gray",
  }

  return (
    <div className="profile">
      <h2>{user.name}</h2>

      {/* Ternary: verified vs unverified */}
      <span>{user.verified ? "Verified" : "Unverified"}</span>

      {/* Object lookup */}
      <span className={roleBadge[user.role]}>{user.role}</span>

      {/* && operator: optional section */}
      {user.bio && <p className="bio">{user.bio}</p>}

      {/* && with boolean prop */}
      {showActions && (
        <div className="actions">
          <Button>Message</Button>
          <Button>Follow</Button>
        </div>
      )}
    </div>
  )
}
Readability rule: If your conditional rendering logic makes the JSX hard to scan, extract it into a variable or a sub-component above the return statement. Keep the JSX tree clean and declarative.
