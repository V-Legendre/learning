07 — Lists & Keys

Rendering dynamic lists with .map() and understanding why keys are critical for React's reconciliation algorithm.

The .map() Pattern

In React, you render lists by mapping data to elements. No special template syntax, no v-for directive — just JavaScript's .map() returning JSX.

Python: List comprehension
# Transform data into output
users = [
    {"id": 1, "name": "Alice"},
    {"id": 2, "name": "Bob"},
]

html = "\n".join(
    f"<li>{u['name']}</li>"
    for u in users
)
React: .map() in JSX
const users = [
  { id: 1, name: "Alice" },
  { id: 2, name: "Bob" },
]

<ul>
  {users.map(user => (
    <li key={user.id}>
      {user.name}
    </li>
  ))}
</ul>
A realistic list component
type User = {
  id: string
  name: string
  email: string
  avatar: string
}

type UserListProps = {
  users: User[]
  onSelect: (userId: string) => void
}

const UserList = ({ users, onSelect }: UserListProps) => (
  <ul className="user-list">
    {users.map(user => (
      <li
        key={user.id}
        onClick={() => onSelect(user.id)}
        className="user-list__item"
      >
        <img src={user.avatar} alt={user.name} />
        <div>
          <strong>{user.name}</strong>
          <span>{user.email}</span>
        </div>
      </li>
    ))}
  </ul>
)

Why Keys Matter

Keys tell React which element is which across re-renders. Without them, React can only match elements by position — which breaks when items are added, removed, or reordered.

Visual: What happens without stable keys
// Initial render: ["Alice", "Bob", "Charlie"]
// React creates:
  <li>Alice</li>     // position 0
  <li>Bob</li>       // position 1
  <li>Charlie</li>   // position 2

// User deletes "Alice". New list: ["Bob", "Charlie"]
// WITHOUT keys, React matches by position:
  <li>Bob</li>       // position 0 — was "Alice", update text to "Bob"
  <li>Charlie</li>   // position 1 — was "Bob", update text to "Charlie"
                    // position 2 — remove last element

// React updated 2 elements and removed 1. Wasteful!
// If elements had state (inputs, focus), it'd be WRONG too.

// WITH keys (key="alice", key="bob", key="charlie"):
// React knows: "alice" was removed. Keep "bob" and "charlie" as-is.
// Just 1 DOM removal. Correct and efficient.
1
State changes
List data updates (add, remove, reorder)
->
2
New elements
.map() produces new JSX elements with keys
->
3
Reconciliation
React matches old and new elements by key
->
4
Minimal DOM update
Only moved, added, or removed elements are touched

Good Keys vs Bad Keys

A key must be unique among siblings and stable across re-renders.

Good keys
// Database ID — unique and stable
{users.map(user => (
  <UserCard key={user.id} user={user} />
))}

// UUID — unique and stable
{tasks.map(task => (
  <TaskRow key={task.uuid} task={task} />
))}

// Unique natural key (email, slug)
{posts.map(post => (
  <PostCard key={post.slug} post={post} />
))}
Bad keys
// Array index — breaks on reorder/delete
{users.map((user, index) => (
  <UserCard key={index} user={user} />
))}
// If Alice (0) is deleted, Bob shifts to 0.
// React thinks Bob IS Alice (same key=0).

// Random value — new key every render
{users.map(user => (
  <UserCard key={Math.random()} user={user} />
))}
// Every render: React unmounts ALL items
// and remounts new ones. Destroys state.

// Duplicate keys — undefined behavior
// React warns and may corrupt the DOM.
When is index OK? Only when ALL three conditions are true: (1) The list is static and never reordered, (2) Items have no state (no inputs, no animations), (3) Items are never added/removed. In practice, this is rare. Default to using a proper ID.

Keyed Fragments

When .map() returns multiple sibling elements per item, you need a Fragment with a key. The shorthand <>...</> doesn't support keys — you must use the explicit Fragment.

Won't work: Shorthand fragment
// <> doesn't accept key prop
{users.map(user => (
  <>
    <dt>{user.name}</dt>
    <dd>{user.email}</dd>
  </>  // Error: key not supported
))}
Correct: Explicit Fragment
import { Fragment } from "react"

{users.map(user => (
  <Fragment key={user.id}>
    <dt>{user.name}</dt>
    <dd>{user.email}</dd>
  </Fragment>
))}

Filtering and Sorting

Always filter and sort before .map(). Don't put logic inside JSX. Chain your array methods, then map to elements at the end.

Bad: Logic mixed into JSX
<ul>
  {users.map(user => (
    user.isActive ? (
      <li key={user.id}>
        {user.name}
      </li>
    ) : null
  ))}
</ul>

// Conditional returns inside map = messy.
// Creates null entries in the array.
Good: Filter, then map
const activeUsers = users
  .filter(user => user.isActive)
  .sort((a, b) => a.name.localeCompare(b.name))

<ul>
  {activeUsers.map(user => (
    <li key={user.id}>
      {user.name}
    </li>
  ))}
</ul>

// Clean separation of data prep and rendering.
Python analogy: You wouldn't write [f(x) for x in items if x.active] with the filtering logic buried deep inside a template. You'd prepare the data first, then render. Same principle here.

Empty State

Always handle the case where your list has zero items. An empty <ul> is bad UX. Tell the user what happened and what they can do.

Pattern: Empty check before map
const TaskList = ({ tasks }: { tasks: Task[] }) => {
  if (tasks.length === 0) {
    return (
      <div className="empty-state">
        <p>No tasks yet</p>
        <Button>Create your first task</Button>
      </div>
    )
  }

  return (
    <ul>
      {tasks.map(task => (
        <TaskRow key={task.id} task={task} />
      ))}
    </ul>
  )
}

Complete Example: Filterable, Sortable List

Real-world list component
type Product = {
  id: string
  name: string
  price: number
  inStock: boolean
}

type SortField = "name" | "price"

type ProductListProps = {
  products: Product[]
  showOutOfStock: boolean
  sortBy: SortField
}

const ProductList = ({ products, showOutOfStock, sortBy }: ProductListProps) => {
  // 1. Filter
  const filtered = showOutOfStock
    ? products
    : products.filter(p => p.inStock)

  // 2. Sort
  const sorted = [...filtered].sort((a, b) => {
    if (sortBy === "name") return a.name.localeCompare(b.name)
    return a.price - b.price
  })

  // 3. Handle empty
  if (sorted.length === 0) {
    return <p>No products match your filters.</p>
  }

  // 4. Render
  return (
    <div className="product-grid">
      {sorted.map(product => (
        <ProductCard
          key={product.id}  {/* stable, unique ID */}
          name={product.name}
          price={product.price}
          inStock={product.inStock}
        />
      ))}
    </div>
  )
}

Key Rules Summary

RuleWhy
Always provide a key when mapping React warns without one and falls back to index (bad default)
Keys must be unique among siblings React uses keys to identify elements. Duplicates cause bugs
Keys must be stable across renders Changing keys = unmount and remount. Destroys state and kills performance
Use data IDs, not array indexes Indexes shift when items are added, removed, or reordered
Keys are not passed as props key is consumed by React, not accessible via props.key. Pass as separate prop if needed
Keys only matter among siblings Two different lists can reuse the same key values. Keys are scoped to their parent element
Pro tip: If your data doesn't have IDs, generate them once when the data is created (e.g., with crypto.randomUUID()), not during each render. Keys must be stable.
