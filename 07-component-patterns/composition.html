07 — Component Composition

Building complex UIs by combining simple components. The React equivalent of Unix pipes — small, focused pieces that snap together.

The children Prop

The most fundamental composition pattern. Whatever you put between a component's opening and closing tags becomes its children prop. Type it as React.ReactNode.

Python analogy
# Context manager wraps arbitrary code
with open_db_connection() as conn:
    # anything goes here
    conn.execute("SELECT ...")
    conn.execute("INSERT ...")

# The `with` block wraps content
# without knowing what's inside.
React children
// Card wraps arbitrary content
<Card>
  <h2>User Profile</h2>
  <p>Any content goes here</p>
  <Button>Edit</Button>
</Card>

// Card doesn't know or care
// what's inside it.
Implementing the Card component
type CardProps = {
  children: React.ReactNode  // accepts anything renderable
  variant?: "default" | "outlined"
}

const Card = ({ children, variant = "default" }: CardProps) => (
  <div className={`card card--${variant}`}>
    {children}
  </div>
)
React.ReactNode is the broadest type for renderable content: strings, numbers, JSX elements, arrays of elements, null, undefined, booleans. Use this when you want to accept "anything React can render."
Bad: Hardcoded content
// Rigid. Can only show title + description.
type CardProps = {
  title: string
  description: string
  buttonText: string
}

const Card = ({ title, description, buttonText }: CardProps) => (
  <div className="card">
    <h2>{title}</h2>
    <p>{description}</p>
    <button>{buttonText}</button>
  </div>
)

// What if I need an image? A list?
// More props = more rigidity.
Good: Composition with children
// Flexible. Card handles layout,
// consumer decides content.
type CardProps = {
  children: React.ReactNode
}

const Card = ({ children }: CardProps) => (
  <div className="card">
    {children}
  </div>
)

// Now anything works:
<Card><img src="..." /></Card>
<Card><UserProfile /></Card>

Slots via Props

When you need multiple insertion points, pass JSX as named props. More explicit than children alone. Think of them as named slots in a template.

Page layout with named slots
type PageLayoutProps = {
  header: React.ReactNode
  sidebar: React.ReactNode
  children: React.ReactNode   // main content
  footer?: React.ReactNode   // optional slot
}

const PageLayout = ({ header, sidebar, children, footer }: PageLayoutProps) => (
  <div className="page">
    <header>{header}</header>
    <div className="page__body">
      <aside>{sidebar}</aside>
      <main>{children}</main>
    </div>
    {footer && <footer>{footer}</footer>}
  </div>
)

Usage

<PageLayout
  header={<NavBar user={currentUser} />}
  sidebar={<FilterPanel filters={filters} />}
  footer={<Copyright />}
>
  <DashboardContent data={data} />
</PageLayout>
Rule of thumb: Use children for the primary content. Use named props for secondary/auxiliary slots (header, footer, icon, actions).

Compound Components

A group of components that work together and share implicit state. The classic example: <select> + <option> in HTML. The parent manages state; children participate without prop drilling.

Bad: One monolithic component
// Rigid API with config objects
<Tabs
  tabs={[
    { label: "Profile", content: <Profile /> },
    { label: "Settings", content: <Settings /> },
    { label: "Billing", content: <Billing />,
      disabled: true },
  ]}
/>

// Hard to customize individual tabs.
// What if one tab needs an icon?
Good: Compound components
// Flexible, composable API
<Tabs defaultValue="profile">
  <Tabs.List>
    <Tabs.Trigger value="profile">
      Profile
    </Tabs.Trigger>
    <Tabs.Trigger value="settings">
      Settings
    </Tabs.Trigger>
  </Tabs.List>
  <Tabs.Content value="profile">
    <Profile />
  </Tabs.Content>
</Tabs>
Implementing compound components with Context
import { createContext, useContext, useState } from "react"

// 1. Internal context (not exported)
type TabsContextType = {
  activeTab: string
  setActiveTab: (value: string) => void
}

const TabsContext = createContext<TabsContextType | null>(null)

const useTabsContext = () => {
  const ctx = useContext(TabsContext)
  if (!ctx) throw new Error("Tab components must be used within <Tabs>")
  return ctx
}

// 2. Parent component provides context
const Tabs = ({ defaultValue, children }: {
  defaultValue: string
  children: React.ReactNode
}) => {
  const [activeTab, setActiveTab] = useState(defaultValue)
  return (
    <TabsContext.Provider value={{ activeTab, setActiveTab }}>
      {children}
    </TabsContext.Provider>
  )
}

// 3. Child components consume context
const Trigger = ({ value, children }: {
  value: string; children: React.ReactNode
}) => {
  const { activeTab, setActiveTab } = useTabsContext()
  return (
    <button
      className={activeTab === value ? "active" : ""}
      onClick={() => setActiveTab(value)}
    >
      {children}
    </button>
  )
}

const Content = ({ value, children }: {
  value: string; children: React.ReactNode
}) => {
  const { activeTab } = useTabsContext()
  return activeTab === value ? <>{children}</> : null
}

// 4. Attach sub-components as properties
Tabs.Trigger = Trigger
Tabs.Content = Content
Python analogy: Compound components are like Python's unittest.TestCase where setUp, tearDown, and test methods implicitly share the test instance state. Or like SQLAlchemy's Session where queries, adds, and commits all operate on shared session state without passing it explicitly.

Container / Presentational

Separate data logic from UI rendering. The container fetches data and manages state. The presentational component is a pure function of its props — easy to test, easy to reuse in Storybook.

Bad: Mixed concerns
const UserList = () => {
  const { data, isLoading } = useQuery({
    queryKey: ["users"],
    queryFn: fetchUsers,
  })

  if (isLoading) return <Spinner />

  return (
    <ul className="user-list">
      {data.map(user => (
        <li key={user.id}>
          <img src={user.avatar} />
          <span>{user.name}</span>
        </li>
      ))}
    </ul>
  )
}

// Data fetching + rendering = tangled.
// Hard to test UI without mocking API.
// Can't reuse UI with different data source.
Good: Separated
// Presentational — pure UI, easy to test
type UserListViewProps = {
  users: User[]
}

const UserListView = ({ users }: UserListViewProps) => (
  <ul className="user-list">
    {users.map(user => (
      <li key={user.id}>
        <img src={user.avatar} />
        <span>{user.name}</span>
      </li>
    ))}
  </ul>
)

// Container — handles data
const UserList = () => {
  const { data, isLoading } = useQuery({
    queryKey: ["users"],
    queryFn: fetchUsers,
  })
  if (isLoading) return <Spinner />
  return <UserListView users={data} />
}
Python analogy: Like separating your SQLAlchemy repository layer (data access) from your Jinja2 templates (presentation). The repository fetches data; the template renders it. Neither knows the other's implementation.

Render Props (Legacy-ish)

A component receives a function as a prop and calls it to render UI. This was the primary composition pattern before hooks. You'll still see it in some libraries (e.g., Formik, React Router's older API).

Mouse tracker with render prop
type MouseTrackerProps = {
  render: (position: { x: number; y: number }) => React.ReactNode
}

const MouseTracker = ({ render }: MouseTrackerProps) => {
  const [pos, setPos] = useState({ x: 0, y: 0 })

  return (
    <div onMouseMove={e => setPos({ x: e.clientX, y: e.clientY })}>
      {render(pos)}
    </div>
  )
}

// Usage
<MouseTracker
  render={({ x, y }) => <p>Mouse at {x}, {y}</p>}
/>
Before: Render props
<MouseTracker
  render={({ x, y }) => (
    <Tooltip x={x} y={y} />
  )}
/>

// Nesting = "callback hell"
After: Custom hook (preferred)
const MyComponent = () => {
  const { x, y } = useMousePosition()
  return <Tooltip x={x} y={y} />
}

// Flat, readable, composable
Modern advice: Prefer custom hooks over render props for new code. But know the pattern — you'll encounter it in existing codebases and library APIs.

Layout Components

Components whose sole job is arranging other components on screen. They handle spacing, alignment, and responsive behavior. Content components never worry about their own positioning.

Common layout components
// Stack: vertical or horizontal spacing
type StackProps = {
  direction?: "vertical" | "horizontal"
  gap?: number
  children: React.ReactNode
}

const Stack = ({ direction = "vertical", gap = 16, children }: StackProps) => (
  <div style={{
    display: "flex",
    flexDirection: direction === "vertical" ? "column" : "row",
    gap: `${gap}px`,
  }}>
    {children}
  </div>
)

// Center: horizontally and vertically center content
const Center = ({ children }: { children: React.ReactNode }) => (
  <div style={{
    display: "flex",
    alignItems: "center",
    justifyContent: "center",
  }}>
    {children}
  </div>
)

// Container: max-width wrapper with auto margins
const Container = ({ children, maxWidth = 1200 }: {
  children: React.ReactNode; maxWidth?: number
}) => (
  <div style={{ maxWidth, margin: "0 auto", padding: "0 16px" }}>
    {children}
  </div>
)

Usage: Layout separates from content

const Dashboard = () => (
  <Container>
    <Stack gap={24}>
      <Header />                     {/* doesn't know about spacing */}
      <Stack direction="horizontal" gap={16}>
        <Sidebar />                  {/* doesn't know it's beside Main */}
        <Main />
      </Stack>
      <Footer />
    </Stack>
  </Container>
)
Bad: Content handles own spacing
const UserCard = () => (
  <div style={{
    marginBottom: 16,
    marginLeft: 24,
    // This card "knows" about the
    // layout it lives in. Fragile.
  }}>
    ...
  </div>
)
Good: Layout handles spacing
// UserCard has zero layout knowledge
const UserCard = () => (
  <div>...</div>
)

// Parent decides spacing
<Stack gap={16}>
  <UserCard />
  <UserCard />
  <UserCard />
</Stack>
Python analogy: Like separating orchestration from business logic. Your Airflow DAG defines task order and dependencies (layout); each task does its own work without knowing where it sits in the pipeline.

Pattern Summary

PatternWhen to UseKey Benefit
children Wrapping arbitrary content Maximum flexibility
Named slots Multiple insertion points (header, footer, icon) Explicit, self-documenting API
Compound Components that share implicit state (Tabs, Accordion, Menu) Clean API, no prop drilling between siblings
Container / Presentational Separating data from UI Testability, reusability
Render props Legacy code, some library APIs Dynamic rendering (prefer hooks)
Layout components Spacing, alignment, responsive structure Content components stay layout-agnostic
The golden rule of composition: A component should do one thing well. If it's handling data, layout, AND rendering — break it apart. Composition beats configuration every time.
