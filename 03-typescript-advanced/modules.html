03 — ES Modules

How TypeScript/JavaScript organizes code into files. Every file is a module.

Module System Overview

What's a module?

In modern JS/TS, every file is a module. Variables, functions, and types are private to the file unless explicitly exported. This is the same as Python: code in utils.py isn't accessible elsewhere unless you import it.

TypeScript / ES Modules
// math.ts
export const add = (a: number, b: number) => a + b
export const PI = 3.14159

// app.ts
import { add, PI } from "./math"
Python
# math_utils.py
def add(a: int, b: int) -> int:
    return a + b
PI = 3.14159

# app.py
from math_utils import add, PI
Historical context: JS originally had no module system. Scripts shared a global scope (chaos). CommonJS (require()) came with Node.js. ES Modules (import/export) is the modern standard. Always use ES Modules in new code.

Named Exports (Preferred)

Export specific items by name. Consumers import them by the exact same name.

Two equivalent syntaxes

// Style 1: inline export (preferred — export as you define)
export const formatDate = (d: Date): string => d.toISOString()
export type User = { name: string; age: number }
export const MAX_RETRIES = 3

// Style 2: export list at bottom (less common)
const formatDate = (d: Date): string => d.toISOString()
type User = { name: string; age: number }
const MAX_RETRIES = 3

export { formatDate, User, MAX_RETRIES }

Importing Named Exports

// Import specific items
import { formatDate, type User } from "./utils"

// Rename on import (like Python's "as")
import { formatDate as fmt } from "./utils"

// Import everything as a namespace
import * as utils from "./utils"
utils.formatDate(new Date())
// Like Python: import utils
TypeScript imports
import { add } from "./math"
import { add as sum } from "./math"
import * as math from "./math"
Python imports
from math import add
from math import add as sum_fn
import math

Default Exports (Avoid)

Each module can have one "default" export. The consumer picks any name for it.

// Button.tsx — default export
const Button = () => <button>Click</button>
export default Button

// App.tsx — import with ANY name (no braces)
import Button from "./Button"      // works
import MyButton from "./Button"    // also works — same thing!
import Banana from "./Button"      // also works — confusing!
Why avoid default exports?
  • No autocomplete — editors can't suggest the name since you can use anything
  • Inconsistent naming — different files import the same thing with different names
  • Harder to refactor — rename-symbol doesn't catch all usages
  • Can't tree-shake as effectively — bundlers optimize named exports better
Default export (avoid)
// UserCard.tsx
export default const UserCard = ...

// Various importers use different names:
import UserCard from "./UserCard"
import Card from "./UserCard"
import UC from "./UserCard"
Named export (preferred)
// UserCard.tsx
export const UserCard = ...

// Everyone uses the same name:
import { UserCard } from "./UserCard"
import { UserCard } from "./UserCard"
import { UserCard } from "./UserCard"
Exception: Some frameworks (Next.js pages, React.lazy) require default exports. Use them there, but prefer named exports everywhere else.

Type-Only Imports

TypeScript adds import type to explicitly mark imports that are types only — they're erased at compile time and produce no JS code.

// types.ts
export type User = { id: number; name: string }
export interface ApiResponse<T> { data: T; status: number }

// consumer.ts — three ways to import types:

// 1. Type-only import (whole statement)
import type { User, ApiResponse } from "./types"

// 2. Inline type modifier (mix values and types)
import { fetchUsers, type User } from "./api"

// 3. Regular import (also works, but less explicit)
import { User } from "./types"
// TS knows User is a type and will erase it
Why bother with import type?
  • Clarity — makes it obvious what's a type vs a runtime value
  • Prevents bugs — can't accidentally use a type as a value at runtime
  • Better bundling — bundlers can more easily drop type-only imports
  • Required sometimes — some TS configs (verbatimModuleSyntax) enforce this

Re-exports (Barrel Files)

Re-export items from other modules. Used to create "barrel files" — a single entry point for a folder.

// components/Button.tsx
export const Button = () => { ... }

// components/Card.tsx
export const Card = () => { ... }

// components/index.ts — barrel file
export { Button } from "./Button"
export { Card } from "./Card"
export type { ButtonProps } from "./Button"

// Now consumers import from the folder:
import { Button, Card } from "./components"
// instead of:
// import { Button } from "./components/Button"
// import { Card } from "./components/Card"
TypeScript barrel
// index.ts
export { Button } from "./Button"
export { Card } from "./Card"

// Re-export all:
export * from "./utils"

// Re-export with rename:
export { Button as Btn } from "./Button"
Python __init__.py
# __init__.py
from .button import Button
from .card import Card

# Re-export all:
from .utils import *

# Same concept — index.ts = __init__.py
Gotcha: Barrel files can hurt performance. If you export * from "./utils" and utils has 100 exports, importing one thing pulls in everything. Keep barrels small and specific. Some teams avoid them entirely.

Dynamic Imports

Load modules at runtime instead of at build time. Returns a Promise. Used for code splitting.

// Static import — always loaded
import { heavyFunction } from "./heavy-module"

// Dynamic import — loaded on demand
const loadHeavy = async () => {
  const { heavyFunction } = await import("./heavy-module")
  heavyFunction()
}

// React lazy loading (code splitting)
const AdminPanel = lazy(() => import("./AdminPanel"))
// AdminPanel code is only downloaded when the component renders
TypeScript
const mod = await import("./module")
mod.doStuff()
Python
import importlib
mod = importlib.import_module("module")
mod.do_stuff()

Side-Effect Imports

Import a module purely for its side effects (no values imported). The module's top-level code runs.

// Import CSS — the bundler processes it, no export needed
import "./styles.css"

// Import a polyfill — modifies global scope
import "core-js/stable"

// Import just to run initialization code
import "./init-analytics"
CSS imports are a bundler feature (Vite, webpack), not native JS. The bundler sees the import and injects the CSS into the page. This is how component-scoped styles work in React projects.

Module Resolution

How TypeScript/bundlers find the file when you write import ... from "...".

Import pathWhat it resolves toType
"./utils"./utils.ts or ./utils/index.tsRelative
"../hooks/useAuth"../hooks/useAuth.tsRelative

File extensions
  • TypeScript: Omit .ts / .tsx extensions in imports. TS resolves them automatically.
  • CSS / JSON / assets: Include the extension: import "./styles.css", import data from "./data.json"

This differs from Python where you never include .py in imports.

Path Aliases

// tsconfig.json — configure path aliases
{
  "compilerOptions": {
    "paths": {
      "@/*": ["./src/*"]
    }
  }
}

// Now instead of this nightmare:
import { Button } from "../../../components/Button"

// You write:
import { Button } from "@/components/Button"

Common Mistakes & Gotchas

Circular imports
// a.ts
import { B } from "./b"
export const A = "A uses " + B

// b.ts
import { A } from "./a"  // circular!
export const B = "B uses " + A

// A is undefined at import time
// Fix: restructure or use a shared module
Importing types as values
import type { User } from "./types"

// ERROR: can't use type as a value
const u = new User()         // types don't exist at runtime
console.log(User)             // types are erased

// OK: use type as a type annotation
const u: User = { ... }       // type annotation is fine
const fn = (u: User) => ... // parameter type is fine
Import order convention
// 1. External packages (node_modules)
import { useState } from "react"
import { useQuery } from "@tanstack/react-query"

// 2. Internal aliases
import { Button } from "@/components/Button"
import { useAuth } from "@/hooks/useAuth"

// 3. Relative imports
import { formatName } from "./utils"

// 4. Type imports
import type { User } from "@/types/user"

// 5. Side-effect imports
import "./styles.css"

// ESLint plugins can auto-sort imports for you
