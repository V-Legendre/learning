03 — Destructuring

Extract values from objects and arrays into variables. Every React component destructures its props.

Object Destructuring

Pull named properties out of an object into local variables. The names must match the object's keys.

TypeScript
const user = { name: "Alice", age: 30, role: "admin" }

// Without destructuring
const name = user.name
const age = user.age

// With destructuring
const { name, age, role } = user
// name = "Alice", age = 30, role = "admin"
Python equivalent
user = {"name": "Alice", "age": 30, "role": "admin"}

# No direct equivalent — you do:
name = user["name"]
age = user["age"]

# Or with dataclass/NamedTuple:
# user.name, user.age — but no unpacking

# Python dicts don't have JS-style
# destructuring syntax

Renaming Variables

When the property name clashes with a local variable, or you want a more descriptive name:

const response = { data: [1, 2, 3], status: 200 }

// Rename: { originalKey: newName }
const { data: items, status: statusCode } = response
// items = [1, 2, 3], statusCode = 200
// "data" and "status" are NOT created as variables
Gotcha: The colon in { data: items } means "take property data, assign it to variable items". It does NOT mean type annotation here. The syntax looks identical but the context is different.

Default Values

Provide a fallback when the property is undefined:

const config = { host: "localhost" }

const { host, port = 3000, debug = false } = config
// host = "localhost", port = 3000, debug = false

// Combine rename + default:
const { host: hostname = "0.0.0.0" } = config
// hostname = "localhost" (value exists, so default not used)
Note: Defaults only apply when the value is undefined, NOT null. If port: null, the default is not used.

Array Destructuring

Like object destructuring, but positional instead of by name. You can name the variables anything you want.

TypeScript
const colors = ["red", "green", "blue"]

// Positional extraction
const [first, second, third] = colors
// first = "red", second = "green"

// Skip elements with commas
const [, , blue] = colors
// blue = "blue"

// Swap variables (no temp needed!)
let a = 1, b = 2;
[a, b] = [b, a]
// a = 2, b = 1
Python equivalent
colors = ["red", "green", "blue"]

# Tuple unpacking — same idea!
first, second, third = colors
# first = "red", second = "green"

# Skip with _
_, _, blue = colors
# blue = "blue"

# Swap — Python has this too
a, b = 1, 2
a, b = b, a
# a = 2, b = 1

Rest Element in Arrays

const nums = [1, 2, 3, 4, 5]

const [head, ...tail] = nums
// head = 1, tail = [2, 3, 4, 5]

const [first, second, ...rest] = nums
// first = 1, second = 2, rest = [3, 4, 5]
TypeScript — rest element
const [head, ...tail] = [1, 2, 3]
Python — starred assignment
head, *tail = [1, 2, 3]

Nested Destructuring

Destructure nested objects/arrays in a single expression. Useful for API responses.

const apiResponse = {
  user: {
    name: "Alice",
    address: {
      city: "Paris",
      zip: "75001"
    }
  },
  meta: { page: 1, total: 50 }
}

// Nested destructuring
const {
  user: {
    name,
    address: { city }
  },
  meta: { total }
} = apiResponse

// name = "Alice", city = "Paris", total = 50
// NOTE: "user", "address", "meta" are NOT created as variables
// Only the leaf values are extracted
Gotcha: In nested destructuring, intermediate levels (user, address, meta) are NOT assigned to variables. Only the final targets (name, city, total) become local variables. If you need both, destructure separately.

Mixed array + object destructuring:

const data = {
  users: [
    { name: "Alice", scores: [90, 85, 95] },
    { name: "Bob", scores: [70, 80] }
  ]
}

// Extract first user's name and first score
const { users: [{ name, scores: [topScore] }] } = data
// name = "Alice", topScore = 90
Tip: Deep nesting gets hard to read. If you go deeper than 2 levels, consider extracting in multiple steps instead.

Destructuring in Function Parameters

This is the most important pattern for React. Every component receives a props object, and you destructure it in the parameter list.

The React Pattern

Instead of writing props.name, props.age everywhere in the component, destructure the props right in the function signature. This is idiomatic React.

Without destructuring
type Props = {
  name: string
  age: number
  email?: string
}

const UserCard = (props: Props) => {
  return (
    <div>
      <h2>{props.name}</h2>
      <p>Age: {props.age}</p>
      <p>{props.email}</p>
    </div>
  )
}
With destructuring (idiomatic)
type Props = {
  name: string
  age: number
  email?: string
}

const UserCard = ({ name, age, email }: Props) => {
  return (
    <div>
      <h2>{name}</h2>
      <p>Age: {age}</p>
      <p>{email}</p>
    </div>
  )
}

Defaults in Function Parameters

type ButtonProps = {
  label: string
  variant?: "primary" | "secondary"
  size?: "sm" | "md" | "lg"
  disabled?: boolean
}

const Button = ({
  label,
  variant = "primary",
  size = "md",
  disabled = false
}: ButtonProps) => {
  // variant, size, disabled all have defaults
  // No need for ?? or || inside the function body
  return <button className={`btn-${variant} btn-${size}`}>{label}</button>
}

Rest in Function Parameters

type InputProps = {
  label: string
  error?: string
} & React.InputHTMLAttributes<HTMLInputElement>

// Extract your custom props, pass the rest to <input>
const Input = ({ label, error, ...inputProps }: InputProps) => {
  return (
    <div>
      <label>{label}</label>
      <input {...inputProps} />
      {error && <span>{error}</span>}
    </div>
  )
}

// Usage: <Input label="Email" error="Required" type="email" placeholder="..." />
// "label" and "error" are consumed; type, placeholder go to <input>

Common Patterns

Extract from API Responses
// fetch returns a Response object
const { data, error, isLoading } =
  useSWR("/api/users")

// Extract from response JSON
const { results, pagination: { page, total } } =
  await response.json()
React useState (array destructuring)
// useState returns [value, setter]
const [count, setCount] = useState(0)
const [name, setName] = useState("")
const [items, setItems] = useState<string[]>([])

// This is why array destructuring matters
// — you pick the variable names yourself
Destructuring in Loops
const users = [
  { id: 1, name: "Alice", role: "admin" },
  { id: 2, name: "Bob", role: "user" }
]

// Destructure in for...of
for (const { name, role } of users) {
  console.log(`${name} is ${role}`)
}

// Destructure in .map() — you'll do this ALL the time in React
users.map(({ id, name }) => (
  <li key={id}>{name}</li>
))

Destructuring with Type Annotations

Annotating destructured variables has a specific syntax. The type comes after the whole pattern, not inline.

// Type annotation for destructured object
const { name, age }: { name: string; age: number } = getUser()

// In practice, you use a named type (much cleaner):
type User = { name: string; age: number }
const { name, age }: User = getUser()

// Array destructuring with types
const [x, y]: [number, number] = getCoords()

// Usually TS infers this — annotate only when needed
Common Mistake: You might be tempted to write const { name: string } = obj thinking it's a type annotation. It's NOT. That renames name to a variable called string. The type goes after the closing brace: const { name }: { name: string } = obj.

Common Mistakes & Gotchas

Destructuring undefined/null
// This CRASHES at runtime
const { name } = undefined  // TypeError!
const { name } = null       // TypeError!

// Safe: provide fallback
const { name } = data ?? {}
// Or use optional chaining first:
const name = data?.name
Unused variables from destructuring
// ESLint will warn about unused vars
const { name, age, role } = user
// If you only need "name", don't extract the rest

// Better: only extract what you use
const { name } = user

// If you need to skip in arrays, use _
const [, second] = arr
Destructuring does NOT deep clone
const original = { user: { name: "Alice" } }
const { user } = original

user.name = "Bob"
console.log(original.user.name) // "Bob" — same reference!

// Destructuring copies the REFERENCE for objects/arrays, not the value.
// Same as Python: extracting a dict value doesn't deep-copy it.
