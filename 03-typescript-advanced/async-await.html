03 — Async/Await & Promises

How JavaScript handles concurrency: single-threaded, non-blocking, event-loop-driven.

The JavaScript Event Loop

Mental Model: JS vs Python Concurrency

Both JS and Python are single-threaded for your code. The key difference is the default mode:

  • Python = synchronous by default. You opt into async with asyncio.
  • JavaScript = async by default for I/O. All network, timers, and file operations are non-blocking.

In Python, requests.get(url) blocks. In JS, fetch(url) returns immediately with a Promise.

// This demonstrates the event loop order
console.log("1 - synchronous")

setTimeout(() => {
  console.log("2 - timeout callback")
}, 0)  // Even 0ms delay goes through the queue!

Promise.resolve().then(() => {
  console.log("3 - promise (microtask)")
})

console.log("4 - synchronous")

// Output order: 1, 4, 3, 2
// Synchronous runs first, then microtasks (Promises),
// then macrotasks (setTimeout)
Note: Promises use a microtask queue which has higher priority than the regular task queue (setTimeout, setInterval). This is why Promise callbacks run before setTimeout callbacks even with setTimeout(..., 0).

Promises

A Promise represents a value that will be available in the future. It's like Python's asyncio.Future.

Creating Promises

// Creating a Promise manually (rare — mostly you consume them)
const delay = (ms: number): Promise<void> =>
  new Promise(resolve => setTimeout(resolve, ms))

// Wrapping a callback API in a Promise
const readFile = (path: string): Promise<string> =>
  new Promise((resolve, reject) => {
    fs.readFile(path, "utf-8", (err, data) => {
      if (err) reject(err)
      else resolve(data)
    })
  })

.then() / .catch() / .finally()

TypeScript — Promise chain
fetch("/api/users")
  .then(response => response.json())
  .then(data => {
    console.log(data)
  })
  .catch(error => {
    console.error("Failed:", error)
  })
  .finally(() => {
    console.log("Done")
  })
Python equivalent
# Python doesn't chain like this.
# The equivalent is just sequential:
try:
    response = requests.get("/api/users")
    data = response.json()
    print(data)
except Exception as e:
    print(f"Failed: {e}")
finally:
    print("Done")
Key insight: .then() returns a new Promise, enabling chaining. Each .then() receives the return value of the previous one. A single .catch() at the end handles errors from any step in the chain.

async/await

Syntactic sugar over Promises. Makes async code look synchronous. Almost identical to Python's async/await.

TypeScript
const fetchUsers = async (): Promise<User[]> => {
  const response = await fetch("/api/users")

  if (!response.ok) {
    throw new Error(
      `HTTP ${response.status}`
    )
  }

  const data = await response.json()
  return data
}
Python equivalent
async def fetch_users() -> list[User]:
    async with aiohttp.ClientSession() as s:
        response = await s.get("/api/users")

        if response.status != 200:
            raise Exception(
                f"HTTP {response.status}"
            )

        data = await response.json()
        return data
Rules of async/await
  • async functions always return a Promise (even if you return a plain value)
  • await can only be used inside an async function (or at the top level of a module)
  • await pauses execution of that function until the Promise settles
  • Other code (event handlers, other async functions) continues running while awaiting
Key difference from Python: In Python, you need an event loop (asyncio.run()) to run async code. In JS, the event loop is always running. You can call an async function from anywhere — it just runs.

Error Handling

try/catch with async/await

const loadData = async () => {
  try {
    const response = await fetch("/api/data")

    // fetch does NOT throw on HTTP errors (404, 500)
    // It only throws on network failures
    if (!response.ok) {
      throw new Error(`HTTP error: ${response.status}`)
    }

    const data = await response.json()
    return data
  } catch (error) {
    // Catches: network errors, HTTP errors we threw, JSON parse errors
    console.error("Failed to load:", error)
    return null
  } finally {
    // Runs regardless of success/failure
    setLoading(false)
  }
}
Gotcha: fetch doesn't throw on HTTP errors! Unlike Python's requests.raise_for_status() or httpx, the fetch API resolves successfully for 404, 500, etc. You must check response.ok (true for 200-299) yourself.

Typed Error Handling

// TypeScript catch blocks receive 'unknown' (not 'Error')
try {
  await riskyOperation()
} catch (error) {
  // error is 'unknown' — you must narrow the type

  if (error instanceof Error) {
    console.log(error.message) // now TS knows it's Error
  }

  // Or assert it (less safe but common)
  const msg = error instanceof Error ? error.message : String(error)
}
Python comparison: Python's except Exception as e: gives you a typed exception. TS's catch (error) gives you unknown because JS can throw anything — strings, numbers, objects, not just Error instances.

Promise.all — Parallel Requests

// Fire multiple requests in parallel, wait for all
const loadDashboard = async () => {
  const [users, posts, comments] = await Promise.all([
    fetch("/api/users").then(r => r.json()),
    fetch("/api/posts").then(r => r.json()),
    fetch("/api/comments").then(r => r.json()),
  ])

  // All three resolved — use the data
  return { users, posts, comments }
}

// WARNING: If ANY request fails, ALL results are lost
// Use allSettled if you want partial results
TypeScript — Promise.all
const [a, b] = await Promise.all([
  fetchA(),
  fetchB()
])
Python equivalent
a, b = await asyncio.gather(
    fetch_a(),
    fetch_b()
)

Promise.allSettled — Partial Success

const results = await Promise.allSettled([
  fetch("/api/users"),
  fetch("/api/posts"),    // might fail
  fetch("/api/comments"),
])

// Each result has { status, value? | reason? }
results.forEach(result => {
  if (result.status === "fulfilled") {
    console.log("Success:", result.value)
  } else {
    console.log("Failed:", result.reason)
  }
})

// Extract only successful results
const successes = results
  .filter((r): r is PromiseFulfilledResult<Response> =>
    r.status === "fulfilled")
  .map(r => r.value)

Promise.race — Timeout Pattern

// Create a timeout that rejects after N ms
const timeout = (ms: number): Promise<never> =>
  new Promise((_, reject) =>
    setTimeout(() => reject(new Error("Timeout")), ms)
  )

// Race the actual request against the timeout
const fetchWithTimeout = async (url: string, ms: number) => {
  const response = await Promise.race([
    fetch(url),
    timeout(ms)
  ])
  return response.json()
}

// Usage: will throw "Timeout" if request takes > 5s
const data = await fetchWithTimeout("/api/slow", 5000)

Common Patterns

Fetch + Parse (the standard pattern)
type User = {
  id: number
  name: string
  email: string
}

const getUsers = async (): Promise<User[]> => {
  const response = await fetch("https://api.example.com/users")

  if (!response.ok) {
    throw new Error(`Failed to fetch users: ${response.status}`)
  }

  // response.json() returns Promise<any> — cast to your type
  const users: User[] = await response.json()
  return users
}

// In practice, use Zod to validate the response shape at runtime
Sequential vs Parallel
// SEQUENTIAL — each waits for the previous (slow!)
const users = await fetchUsers()      // 500ms
const posts = await fetchPosts()      // 300ms
const comments = await fetchComments() // 200ms
// Total: ~1000ms

// PARALLEL — all fire at once (fast!)
const [users, posts, comments] = await Promise.all([
  fetchUsers(),     // 500ms
  fetchPosts(),     // 300ms
  fetchComments()  // 200ms
])
// Total: ~500ms (limited by slowest)

// Use SEQUENTIAL when: B depends on A's result
// Use PARALLEL when: requests are independent
Retry Pattern
const fetchWithRetry = async <T>(
  fn: () => Promise<T>,
  retries = 3,
  delayMs = 1000
): Promise<T> => {
  for (let i = 0; i < retries; i++) {
    try {
      return await fn()
    } catch (error) {
      if (i === retries - 1) throw error
      await new Promise(r => setTimeout(r, delayMs * (i + 1)))
    }
  }
  throw new Error("Unreachable")
}

// Usage
const data = await fetchWithRetry(() => getUsers(), 3, 1000)

Async in React (Preview)

useEffect + async (the most common pattern)
// You can't make the useEffect callback itself async.
// Instead, define an async function inside and call it.

const UserList = () => {
  const [users, setUsers] = useState<User[]>([])
  const [loading, setLoading] = useState(true)
  const [error, setError] = useState<string | null>(null)

  useEffect(() => {
    const load = async () => {
      try {
        const data = await getUsers()
        setUsers(data)
      } catch (e) {
        setError(e instanceof Error ? e.message : "Unknown error")
      } finally {
        setLoading(false)
      }
    }

    load()
  }, []) // empty deps = run once on mount

  if (loading) return <p>Loading...</p>
  if (error) return <p>Error: {error}</p>
  return <ul>{users.map(u => <li key={u.id}>{u.name}</li>)}</ul>
}
Tip: In real apps, you'll use a data-fetching library (TanStack Query, SWR) instead of manual fetch + useEffect. They handle caching, loading states, error states, deduplication, and more. We'll cover this in the React hooks sections.

Common Mistakes & Gotchas

Forgetting await
// WRONG — comparing Promise, not value
const data = fetchData()
if (data.length > 0) { ... }
// data is Promise, not the resolved value!

// RIGHT
const data = await fetchData()
if (data.length > 0) { ... }
Unhandled rejections
// WRONG — no error handling
fetchData() // if this rejects, crash!

// RIGHT — handle errors
fetchData().catch(console.error)
// or
try {
  await fetchData()
} catch (e) { ... }
Sequential when parallel is better
// SLOW — unnecessary sequential
const a = await fetchA()
const b = await fetchB()

// FAST — parallel when independent
const [a, b] = await Promise.all([
  fetchA(), fetchB()
])
async in .forEach (doesn't work)
// WRONG — forEach ignores returned promises
urls.forEach(async url => {
  await fetch(url) // fires and forgets!
})

// RIGHT — use for...of or Promise.all
for (const url of urls) {
  await fetch(url) // sequential
}
// or parallel:
await Promise.all(urls.map(fetch))
