03 — Utility Types

Built-in type transformations that derive new types from existing ones. TypeScript's superpower over Python's type system.

Why Utility Types?

The DRY principle for types

Instead of duplicating type definitions with small variations, utility types let you derive new types from existing ones. Define a base type once, then create variants programmatically.

// Instead of defining these separately:
type User = { id: number; name: string; email: string; role: string }
type UserUpdate = { id?: number; name?: string; email?: string; role?: string }  // tedious duplication

// Derive it:
type UserUpdate = Partial<User>  // same thing, zero duplication

Partial<T>

Makes all properties optional. Returns a type where every key from T has ? added.

type User = {
  id: number
  name: string
  email: string
}

type UserUpdate = Partial<User>
// Equivalent to:
// { id?: number; name?: string; email?: string }

// Perfect for update functions — pass only fields that changed
const updateUser = (id: number, changes: Partial<User>): User => {
  const existing = getUser(id)
  return { ...existing, ...changes }
}

updateUser(1, { name: "Bob" })            // only update name
updateUser(1, { email: "bob@example.com" }) // only update email
TypeScript — Partial
type Config = {
  host: string
  port: number
  debug: boolean
}

const configure = (
  overrides: Partial<Config>
): Config => ({
  host: "localhost",
  port: 3000,
  debug: false,
  ...overrides
})
Python equivalent
# Python has no Partial — you'd use
# TypedDict with total=False or Optional

class Config(TypedDict):
    host: str
    port: int
    debug: bool

# No easy way to make "all optional"
# variant of a TypedDict
class ConfigUpdate(TypedDict, total=False):
    host: str
    port: int
    debug: bool

Required<T>

Opposite of Partial. Makes all properties required (removes ?).

type FormData = {
  name?: string
  email?: string
  age?: number
}

type CompleteForm = Required<FormData>
// { name: string; email: string; age: number }
// All fields are now required

// Useful for validation: "after validation, all fields are guaranteed"
const validateForm = (data: FormData): CompleteForm | null => {
  if (data.name && data.email && data.age) {
    return data as CompleteForm
  }
  return null
}

Pick<T, K>

Create a type with only the specified properties from T.

type User = {
  id: number
  name: string
  email: string
  role: string
  createdAt: Date
}

// Only pick what a profile card needs
type ProfileCardProps = Pick<User, "name" | "email">
// { name: string; email: string }

// Only pick what the login form returns
type LoginCredentials = Pick<User, "email"> & { password: string }
// { email: string; password: string }
React use case: When a component only needs a few props from a larger type, use Pick to create a focused prop type. This makes components more reusable and their dependencies explicit.

Omit<T, K>

Create a type with all properties from T except the specified ones. Inverse of Pick.

type User = {
  id: number
  name: string
  email: string
  role: string
  createdAt: Date
}

// For creating a new user — no id or createdAt yet
type CreateUserInput = Omit<User, "id" | "createdAt">
// { name: string; email: string; role: string }

// For a public API — hide internal fields
type PublicUser = Omit<User, "role">
// { id: number; name: string; email: string; createdAt: Date }
Pick vs Omit — which to use?
  • Pick when you need a few properties from a large type
  • Omit when you need most properties but want to exclude a few

Rule of thumb: use whichever results in a shorter list of keys.

Record<K, V>

Create an object type with keys of type K and values of type V. This is the TS version of Python's Dict[KeyType, ValueType].

// Dictionary/map with string keys and number values
type ScoreBoard = Record<string, number>
// Python: Dict[str, int]

const scores: ScoreBoard = {
  alice: 95,
  bob: 87,
  charlie: 92
}

// Use a union type for keys — restricts allowed keys
type Theme = "light" | "dark" | "system"
type ThemeColors = Record<Theme, { bg: string; text: string }>

const themes: ThemeColors = {
  light: { bg: "#fff", text: "#000" },
  dark: { bg: "#000", text: "#fff" },
  system: { bg: "auto", text: "auto" },
  // missing any key or adding extra = compile error!
}
TypeScript
type Cache = Record<string, User>

// Equivalent to:
type Cache = {
  [key: string]: User
}
Python
Cache = Dict[str, User]

# Or with newer syntax:
cache: dict[str, User] = {}

Readonly<T>

Makes all properties readonly (can't be reassigned). Like frozen dataclasses in Python.

type User = {
  id: number
  name: string
}

type FrozenUser = Readonly<User>
// { readonly id: number; readonly name: string }

const user: FrozenUser = { id: 1, name: "Alice" }
user.name = "Bob"  // ERROR: Cannot assign to 'name' because it is a read-only property
TypeScript
type Config = Readonly<{
  host: string
  port: number
}>

// Also works with arrays:
type IDs = ReadonlyArray<number>
// or: readonly number[]
Python
from dataclasses import dataclass

@dataclass(frozen=True)
class Config:
    host: str
    port: int

# Or: tuple for immutable sequence
ids: tuple[int, ...] = (1, 2, 3)
Gotcha: Readonly is shallow — it only freezes the top level. Nested objects can still be mutated. There's no built-in DeepReadonly; libraries like ts-essentials provide one.

keyof T

Extracts all property names of a type as a union. Essential for building generic utilities.

type User = {
  id: number
  name: string
  email: string
}

type UserKeys = keyof User
// "id" | "name" | "email"

// Use it to make type-safe property access functions
const getValue = <T, K extends keyof T>(obj: T, key: K): T[K] => {
  return obj[key]
}

const user: User = { id: 1, name: "Alice", email: "a@b.com" }

const name = getValue(user, "name")    // type: string
const id = getValue(user, "id")        // type: number
getValue(user, "phone")                 // ERROR: "phone" is not in keyof User
Python comparison: Python has no keyof. The closest is typing.Literal["id", "name", "email"], but you'd maintain it manually. TypeScript derives it automatically from the type.

Mapped Types

Create new types by transforming each property of an existing type. This is how Partial, Required, and Readonly are actually implemented.

// This IS the implementation of Partial:
type MyPartial<T> = {
  [K in keyof T]?: T[K]
}

// This IS the implementation of Readonly:
type MyReadonly<T> = {
  readonly [K in keyof T]: T[K]
}

// Make all values nullable
type Nullable<T> = {
  [K in keyof T]: T[K] | null
}

type User = { name: string; age: number }
type NullableUser = Nullable<User>
// { name: string | null; age: number | null }
How to read mapped types

{ [K in keyof T]: ... } means "for each key K in type T, create a property with this value type." Think of it as a type-level for loop:

// Pseudo-code mental model:
// for K in keys_of(T):
//     new_type[K] = transform(T[K])

Practical React Examples

Form state management
// Base user type from your API
type User = {
  id: number
  name: string
  email: string
  role: "admin" | "user"
  createdAt: Date
}

// Form only needs editable fields
type EditableFields = Omit<User, "id" | "createdAt">
// { name: string; email: string; role: "admin" | "user" }

// Form state — all optional during editing
type FormState = Partial<EditableFields>
// { name?: string; email?: string; role?: "admin" | "user" }

// Error state — each field might have an error message
type FormErrors = Partial<Record<keyof EditableFields, string>>
// { name?: string; email?: string; role?: string }
Component prop subsets
// Full button config
type ButtonConfig = {
  label: string
  variant: "primary" | "secondary" | "danger"
  size: "sm" | "md" | "lg"
  disabled: boolean
  onClick: () => void
}

// Icon button only needs a subset
type IconButtonProps = Pick<ButtonConfig, "variant" | "size" | "onClick"> & {
  icon: string
}

// Toolbar button — ButtonConfig but with optional size
type ToolbarButtonProps = Omit<ButtonConfig, "size"> & {
  size?: ButtonConfig["size"]
}
API response transformations
// API returns dates as strings
type ApiUser = {
  id: number
  name: string
  createdAt: string  // ISO date string from JSON
  updatedAt: string
}

// App uses proper Date objects
type AppUser = Omit<ApiUser, "createdAt" | "updatedAt"> & {
  createdAt: Date
  updatedAt: Date
}

// Transform function with proper types
const toAppUser = (api: ApiUser): AppUser => ({
  ...api,
  createdAt: new Date(api.createdAt),
  updatedAt: new Date(api.updatedAt),
})

Other Useful Utility Types

TypeWhat it doesExample
Extract<T, U> Keep members of union T that are assignable to U Extract<"a" | "b" | "c", "a" | "c"> = "a" | "c"
Exclude<T, U> Remove members of union T that are assignable to U Exclude<"a" | "b" | "c", "a"> = "b" | "c"
NonNullable<T> Remove null and undefined from T NonNullable<string | null> = string
ReturnType<F> Extract return type of a function type ReturnType<typeof fetchUser>
Parameters<F> Extract parameter types as a tuple Parameters<typeof fn> = [string, number]
Awaited<T> Unwrap Promise type Awaited<Promise<User>> = User 
// ReturnType — extract type from function without duplicating
const createUser = (name: string, age: number) => ({
  id: crypto.randomUUID(),
  name,
  age,
  createdAt: new Date()
})

type User = ReturnType<typeof createUser>
// { id: string; name: string; age: number; createdAt: Date }
// No need to define the type separately!

// Awaited — unwrap async return types
const fetchUsers = async (): Promise<User[]> => { ... }
type Users = Awaited<ReturnType<typeof fetchUsers>>
// User[]  (unwraps the Promise)

Combining Utility Types

The real power comes from composing multiple utilities together:

type User = {
  id: number
  name: string
  email: string
  role: "admin" | "user"
  preferences: {
    theme: string
    notifications: boolean
  }
}

// Create a user (no id, all other fields required)
type CreateUserDTO = Omit<User, "id">

// Update a user (id required, everything else optional)
type UpdateUserDTO = Pick<User, "id"> & Partial<Omit<User, "id">>

// Public-facing user (readonly, no role)
type PublicUser = Readonly<Omit<User, "role">>

// Settings page only needs preferences
type SettingsProps = Pick<User, "preferences">

// Status map for a list of users
type UserStatusMap = Record<User["id"], "online" | "offline">
// { [key: number]: "online" | "offline" }

Common Mistakes & Gotchas

Omit doesn't catch typos
type User = { name: string; age: number }

// No error! "nme" doesn't exist but Omit
// doesn't validate keys
type Bad = Omit<User, "nme">
// Still { name: string; age: number }

// Pick DOES catch typos:
type Also = Pick<User, "nme"> // ERROR!
Readonly is shallow
type State = Readonly<{
  user: { name: string }
}>

const s: State = { user: { name: "A" } }
s.user = { name: "B" }      // ERROR (good)
s.user.name = "B"           // OK! (bad)
// Nested objects are still mutable
Partial makes EVERYTHING optional
// If you only want SOME fields optional, don't use Partial on the whole type
type User = { id: number; name: string; email: string }

// WRONG if you still need id to be required:
type Update = Partial<User>  // id is now optional too!

// RIGHT — combine Pick + Partial:
type Update = Pick<User, "id"> & Partial<Omit<User, "id">>
// { id: number; name?: string; email?: string }
