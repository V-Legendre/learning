09 — Router Setup

Client-side routing with TanStack Router: type-safe, file-based, with built-in data loading.

What Is Client-Side Routing?

In a traditional server-rendered app (Flask, Django), every URL change triggers a full HTTP request. The server responds with an entirely new HTML page. In a Single Page Application (SPA), the browser loads your React app once, and then URL changes are handled entirely in JavaScript — no page reload, no server round-trip.

Server-side routing (Flask)
# Each URL = full page reload
@app.route("/users")
def users():
    return render_template("users.html")

@app.route("/users/<id>")
def user_detail(id):
    return render_template("user.html")

Browser sends GET to server, receives full HTML. White flash between pages.

Client-side routing (React)
// URL changes, but the page doesn't reload
// React swaps out the component tree
"/users"    // renders <UserList />
"/users/42" // renders <UserDetail />

// Under the hood: Browser History API
// window.history.pushState()

No server request. Instant transitions. Shared state persists across "pages."

Key concept: The browser's History API (pushState, popstate event) is what makes SPA routing possible. The router library wraps this API so you never touch it directly.

Why TanStack Router Over React Router?

React Router is the classic choice (been around since 2014), but TanStack Router is the modern alternative built for TypeScript-first development. Here's why we're using it:

FeatureReact RouterTanStack Router
Type safetyPartial (string-based paths)Full (all paths, params, search params typed)
Search paramsManual parsing, untypedBuilt-in Zod validation, fully typed
Route loadersBasic (v6.4+)Deep integration with TanStack Query
File-based routingRequires Remix or pluginBuilt-in via Vite plugin
PreloadingManualBuilt-in (preload on hover)
MaturityVery mature, huge ecosystemNewer, smaller ecosystem
Bundle size~14 KB~12 KB
Python analogy: Choosing TanStack Router is like choosing FastAPI over Flask. Both work, but one is built from the ground up with type safety and modern patterns (like FastAPI using Pydantic for validation). TanStack Router uses Zod for search params the same way FastAPI uses Pydantic for query params.

Installation

$ pnpm add @tanstack/react-router
$ pnpm add -D @tanstack/router-plugin

Vite Configuration

The Vite plugin watches your src/routes/ folder and auto-generates a route tree. You never manually maintain route config.

// vite.config.ts
import { defineConfig } from 'vite'
import react from '@vitejs/plugin-react'
import { TanStackRouterVite } from '@tanstack/router-plugin/vite'

export default defineConfig({
  plugins: [
    TanStackRouterVite(),  // Must be before react()
    react(),
  ],
})
Order matters: TanStackRouterVite() must come before react() in the plugins array. It needs to process routes before React's compilation step.

File-Based Routing

Instead of writing route config manually, you create files in src/routes/ and the plugin generates the route tree for you. The file name determines the URL path.

File naming conventions
File nameURL pathPurpose
index.tsx/Default route for that directory
about.tsx/aboutStatic route
users.tsx/usersStatic route (also a layout if it has children)
users/$userId.tsx/users/:userIdDynamic segment (like Flask's <userId>)
_layout.tsx(none)Layout wrapper (no URL segment)
_authenticated.tsx(none)Pathless layout for route groups
Flask (decorators)
# Routes defined in Python code
@app.route("/")
@app.route("/about")
@app.route("/users")
@app.route("/users/<user_id>")

Routes live alongside handler code. You wire URL to function via decorator.

TanStack Router (files)
// Routes defined by file structure
src/routes/
  index.tsx          // /
  about.tsx          // /about
  users/
    index.tsx        // /users
    $userId.tsx      // /users/:userId

File path IS the URL. Each file exports a route component. No manual wiring.

Project Structure with Routing

src/
  routes/
    __root.tsx ← Root layout (wraps everything: nav, footer)
    index.tsx ← Home page (/)
    about.tsx ← About page (/about)
    users/
      index.tsx ← User list (/users)
      $userId.tsx ← User detail (/users/42)
    _authenticated/ ← Pathless layout (no URL segment)
      dashboard.tsx ← /dashboard (protected)
      settings.tsx ← /settings (protected)
  routeTree.gen.ts ← Auto-generated! Never edit this.
  main.tsx ← App entry: creates router, renders <RouterProvider>
  components/ ← Shared components (not routes)
  lib/ ← Utilities, API clients, etc.
Never edit routeTree.gen.ts: This file is auto-generated by the Vite plugin every time you add, remove, or rename a route file. It will be overwritten on the next build.

Router Configuration

The root route (__root.tsx)

This is the top-level layout that wraps every page. Think of it like your base template in Jinja2.

// src/routes/__root.tsx
import { createRootRoute, Outlet } from '@tanstack/react-router'

export const Route = createRootRoute({
  component: () => (
    <div>
      <nav>
        <Link to="/">Home</Link>
        <Link to="/about">About</Link>
        <Link to="/users">Users</Link>
      </nav>
      <Outlet />  {/* Child route renders here */}
    </div>
  ),
})
<Outlet /> is the slot where child routes render. Same concept as Jinja2's {% block content %} or Django's {{ block.super }}. The nav stays; the content swaps.

A basic route file

// src/routes/about.tsx
import { createFileRoute } from '@tanstack/react-router'

export const Route = createFileRoute('/about')({
  component: AboutPage,
})

function AboutPage() {
  return <h1>About us</h1>
}

Creating the router instance (main.tsx)

// src/main.tsx
import { StrictMode } from 'react'
import { createRoot } from 'react-dom/client'
import { RouterProvider, createRouter } from '@tanstack/react-router'
import { routeTree } from './routeTree.gen'

// Create a router instance
const router = createRouter({ routeTree })

// Register the router for type safety
declare module '@tanstack/react-router' {
  interface Register {
    router: typeof router
  }
}

createRoot(document.getElementById('root')!).render(
  <StrictMode>
    <RouterProvider router={router} />
  </StrictMode>
)
The declare module trick

The declare module block tells TypeScript about your specific router instance. This is how TanStack Router achieves full type safety: every <Link to="...">, every useParams(), every useSearch() in your entire app knows the exact set of valid routes, params, and search params. You get autocomplete and compile-time errors for invalid paths.

Python analogy: The declare module is conceptually like Python's TYPE_CHECKING pattern. It's purely for the type system, has zero runtime cost, but gives you IDE support everywhere.
