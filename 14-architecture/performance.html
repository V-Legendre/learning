14 — React Performance

Profiling, code splitting, memoization, and optimization strategies for React apps.

React Is Fast by Default

The golden rule of React performance

Don't optimize prematurely. React's virtual DOM diffing is efficient. Most apps never need performance work. The workflow is always:

1
Notice slowness
User-visible lag
→
2
Measure
React DevTools Profiler
→
3
Identify cause
Find the slow component
→
4
Optimize
Apply targeted fix
→
5
Measure again
Verify improvement
Anti-pattern: Wrapping everything in React.memo and useCallback "just in case." This adds complexity and can actually make things slower (memo comparison has a cost). Only optimize what you've measured to be slow.
Python analogy: Same as profiling Python with cProfile or py-spy. You wouldn't rewrite a Python function in C without first proving it's the bottleneck. Same principle here.

React DevTools Profiler

The Profiler is built into the React DevTools browser extension. It records renders and shows you exactly what re-rendered and why.

How to use it

1
Install React DevTools browser extension (Chrome or Firefox)
2
Open DevTools → Profiler tab (next to Components tab)
3
Click Record, interact with your app, click Stop
4
Read the flamegraph: wide bars = slow renders, gray bars = didn't re-render

What to look for

Signal Meaning Fix
Component renders on every keystroke State is too high / lifting too much Move state down, or use uncontrolled inputs
Large list re-renders completely Parent re-renders, all children follow React.memo on list items, or virtualize
Expensive computation every render Calculation not cached useMemo for expensive derivations
Callback causes child to re-render New function reference each render useCallback to stabilize the reference
Context change re-renders everything Too much in one context Split contexts, or use Zustand instead

Code Splitting with React.lazy

By default, Vite bundles your entire app into a few JavaScript files. Code splitting lets you load parts of your app on demand, reducing the initial download.

React.lazy + Suspense

import { lazy, Suspense } from 'react'

// Instead of: import { SettingsPage } from './features/settings/SettingsPage'
// Lazy load — creates a separate chunk
const SettingsPage = lazy(() => import('./features/settings/SettingsPage'))

// Must wrap in Suspense with a fallback
const App = () => (
  <Suspense fallback={<div>Loading...</div>}>
    <SettingsPage />
  </Suspense>
)
How it works: React.lazy tells the bundler (Vite/Rollup) to split this import into a separate file. When the component is first needed, React downloads the chunk, shows the fallback while loading, then renders the component.
Python analogy: Like lazy imports in Python (importlib.import_module() at call time instead of top-level). The module isn't loaded until you actually need it.

Route-based splitting (TanStack Router)

The most impactful place to split is at the route level. Each page becomes its own chunk.

// routes/_authenticated/settings.lazy.tsx
// The .lazy.tsx suffix tells TanStack Router to code-split this route

import { createLazyFileRoute } from '@tanstack/react-router'
import { SettingsPage } from '@features/settings'

export const Route = createLazyFileRoute('/_authenticated/settings')({
  component: SettingsPage,
})
Result: When a user visits /dashboard, they don't download the settings page code. It loads only when they navigate to /settings. This can cut initial bundle size by 50%+ for large apps.

Bundle Analysis

Before optimizing, see what's in your bundle. Large dependencies you didn't expect are often the biggest win.

$ pnpm add -D rollup-plugin-visualizer

import { visualizer } from 'rollup-plugin-visualizer'

export default defineConfig({

    }),

})

Common findings and fixes

Finding Fix
moment.js (300KB+) Replace with date-fns or dayjs (tree-shakeable)
lodash (full library imported) Use lodash-es or import individual functions
Icon library ships all icons Use tree-shakeable imports: import { Plus } from 'lucide-react'
Syntax highlighting library (200KB+) Lazy-load it — only needed on specific pages
Entire UI library in bundle shadcn/ui copies code, so you only have what you use

When to Memoize

React provides three memoization tools. Use them only when the Profiler shows a problem.

Tool What It Does Use When
React.memo(Component) Skips re-render if props haven't changed Component renders often with same props (list items)
useMemo(() => value, [deps]) Caches a computed value between renders Expensive calculation (sorting, filtering large lists)
useCallback((args) => fn, [deps]) Caches a function reference between renders Function passed as prop to a memoized child

Example: Memoizing an expensive list

Before (re-renders all items)
const UserList = ({ users, filter }: Props) => {
  // Runs on EVERY render
  const filtered = users
    .filter((u) => u.name.includes(filter))
    .sort((a, b) =>
      a.name.localeCompare(b.name))

  return (
    <ul>
      {filtered.map((u) =>
        <UserCard key={u.id} user={u} />
      )}
    </ul>
  )
}
After (memoized)
const UserList = ({ users, filter }: Props) => {
  // Only recalculates when users or
  // filter actually change
  const filtered = useMemo(() =>
    users
      .filter((u) => u.name.includes(filter))
      .sort((a, b) =>
        a.name.localeCompare(b.name)),
    [users, filter]
  )

  return (
    <ul>
      {filtered.map((u) =>
        <UserCard key={u.id} user={u} />
      )}
    </ul>
  )
}

React.memo for list items

// Wrap the child component so it skips re-render when props are the same
import { memo } from 'react'

interface UserCardProps {
  user: User
  onSelect: (id: string) => void
}

export const UserCard = memo(({ user, onSelect }: UserCardProps) => {
  return (
    <div onClick={() => onSelect(user.id)}>
      {user.name}
    </div>
  )
})

// In the parent, stabilize the callback with useCallback
const handleSelect = useCallback((id: string) => {
  setSelectedId(id)
}, []) // empty deps — function never changes

Virtualization

When rendering very long lists (1,000+ items), the DOM itself becomes the bottleneck. Virtualization renders only the visible items plus a small buffer.

Without virtualization

10,000 items = 10,000 DOM nodes.

Browser struggles to layout, paint, and handle events for all of them. Scroll becomes janky.

With virtualization

10,000 items but only ~30 DOM nodes.

As you scroll, items are recycled. Only visible items exist in the DOM.

// Using @tanstack/react-virtual
import { useVirtualizer } from '@tanstack/react-virtual'
import { useRef } from 'react'

const VirtualList = ({ items }: { items: string[] }) => {
  const parentRef = useRef<HTMLDivElement>(null)

  const virtualizer = useVirtualizer({
    count: items.length,
    getScrollElement: () => parentRef.current,
    estimateSize: () => 50, // estimated row height in px
  })

  return (
    <div ref={parentRef} style={{ height: '400px', overflow: 'auto' }}>
      <div style={{ height: virtualizer.getTotalSize() }}>
        {virtualizer.getVirtualItems().map((virtualRow) => (
          <div
            key={virtualRow.key}
            style={{
              position: 'absolute',
              top: virtualRow.start,
              height: virtualRow.size,
            }}
          >
            {items[virtualRow.index]}
          </div>
        ))}
      </div>
    </div>
  )
}
Python analogy: Virtualization is like pagination, but for the DOM. Instead of sending 10,000 rows from a database (you'd paginate with LIMIT/OFFSET), you're "paginating" DOM nodes to only render what's visible on screen.

Performance Optimization Cheat Sheet

Problem Tool / Technique When to Apply
Large initial bundle React.lazy / route splitting Always for routes, lazy-load heavy features
Unknown bundle contents rollup-plugin-visualizer Before any optimization — see what you're working with
Unnecessary re-renders React.memo + useCallback Only when Profiler shows wasted renders
Expensive calculations useMemo Sorting/filtering large datasets on every render
Huge lists (1000+ items) @tanstack/react-virtual When scrolling becomes janky
Slow image loading loading="lazy", WebP, responsive Always for below-the-fold images
Identifying render issues React DevTools Profiler Step 1 of any performance investigation
