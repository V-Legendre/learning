14 — Project Structure

How to organize a production React project: feature folders, barrel exports, and the API layer.

Two Approaches to Organization

As a project grows past a handful of components, you need a consistent structure. There are two dominant patterns in the React ecosystem, and they map nicely to patterns you already know from Python.

Feature Folders (Recommended)

Group files by domain/feature. Everything related to "auth" lives together. This is the pattern used by most production React codebases.

Python analogy: Django apps — each app (users, orders, payments) has its own models, views, urls, tests.

Layer Folders (Simpler)

Group files by type — all components together, all hooks together. Works for small apps but becomes a mess at scale.

Python analogy: Flask app organized by type (all routes in routes/, all models in models/).

Feature Folder Structure

This is what a mature React project looks like. Each feature is a self-contained unit with its own components, hooks, API calls, and types.

Layer Folder Structure

For small apps (under 10 components), layer folders are fine. Everything is flat and easy to find.

src/
  components/
    LoginForm.tsx
    SignupForm.tsx
    DashboardPage.tsx
    Button.tsx
    Modal.tsx
  hooks/
    useAuth.ts
    useDebounce.ts
  api/
    auth.ts
    dashboard.ts
  types/
  utils/
  routes/
  main.tsx
When to switch: Start with layer folders. When you find yourself scrolling through a components/ directory with 15+ files, it's time to refactor into feature folders.

Rules of Thumb

Rule Why
Feature folders when 10+ components Below that, the overhead of folders isn't worth it
One component per file Easy to find, easy to test, clear ownership
Co-locate tests next to source LoginForm.tsx + LoginForm.test.tsx in the same folder
Keep shared/ truly shared If only one feature uses it, keep it in that feature
Flat over nested Don't nest more than 3 levels deep inside a feature
Name files after the component LoginForm.tsx exports LoginForm — never index.tsx for components

Barrel Exports

A barrel export is an index.ts file that re-exports the public API of a feature. It lets consumers import from the feature folder directly instead of reaching into its internals.

Without barrel exports
// Deep imports — coupled to internal structure
import { LoginForm } from '../features/auth/components/LoginForm'
import { useAuth } from '../features/auth/hooks/useAuth'
import type { User } from '../features/auth/types'
With barrel exports
// Clean imports — feature controls its API
import { LoginForm, useAuth } from '../features/auth'
import type { User } from '../features/auth'

The barrel file

// features/auth/index.ts

// Re-export public components
export { LoginForm } from './components/LoginForm'
export { SignupForm } from './components/SignupForm'

// Re-export hooks
export { useAuth } from './hooks/useAuth'

// Re-export types
export type { User, LoginData, AuthState } from './types'

// NOT exported: internal components, helpers, etc.
// This gives the feature control over its public surface

Pros and Cons

Pros Cons
Clean, short imports Can mask where things actually live
Feature controls its public API Risk of circular dependencies between features
Refactoring internals doesn't break consumers Can hurt tree-shaking if barrel re-exports everything
Enforces encapsulation IDE auto-imports might bypass barrels
Tip: Only barrel-export what's truly public. Internal helpers and sub-components should not be re-exported. Think of it like Python's __init__.py with explicit __all__.

The API Layer

Centralize all API calls in each feature's api.ts file. This keeps fetch logic out of components and makes it easy to swap backends, add error handling, or mock in tests.

Feature API file

// features/dashboard/api.ts

import { api } from '../../shared/utils/api'
import type { DashboardStats, Activity } from './types'

// Pure fetch functions — no React, no hooks
export const fetchStats = (): Promise<DashboardStats> =>
  api.get('/dashboard/stats')

export const fetchActivity = (page: number): Promise<Activity[]> =>
  api.get(`/dashboard/activity?page=${page}`)

TanStack Query hooks that use the API

// features/dashboard/hooks/useDashboardStats.ts

import { useQuery } from '@tanstack/react-query'
import { fetchStats } from '../api'

export const useDashboardStats = () => {
  return useQuery({
    queryKey: ['dashboard', 'stats'],
    queryFn: fetchStats,
    staleTime: 5 * 60 * 1000, // 5 minutes
  })
}

Shared API utility

// shared/utils/api.ts
// Thin wrapper around fetch with auth, error handling, JSON parsing

const BASE_URL = import.meta.env.VITE_API_URL

const request = async <T>(
  path: string,
  options: RequestInit = {}
): Promise<T> => {
  const res = await fetch(`${BASE_URL}${path}`, {
    headers: {
      'Content-Type': 'application/json',
      ...options.headers,
    },
    ...options,
  })

  if (!res.ok) {
    throw new Error(`API error: ${res.status}`)
  }

  return res.json()
}

export const api = {
  get:    <T>(path: string) => request<T>(path),
  post:   <T>(path: string, data: unknown) =>
    request<T>(path, { method: 'POST', body: JSON.stringify(data) }),
  put:    <T>(path: string, data: unknown) =>
    request<T>(path, { method: 'PUT', body: JSON.stringify(data) }),
  delete: <T>(path: string) =>
    request<T>(path, { method: 'DELETE' }),
}
Python analogy: This is exactly like having a services/ or clients/ layer in a Python project — a class DashboardClient with typed methods that wrap HTTP calls. The TanStack Query hook is like adding caching and retry logic on top.

Dependency Rules

Keep your imports sane with these directional rules:

Import direction rules
  • Features can import from shared/ — always OK
  • Features should NOT import from other features — if two features need the same thing, lift it to shared/
  • shared/ never imports from features — it has no knowledge of specific domains
  • Routes import from features — routes wire features to URLs

Wire URL → Feature
→
Features
Self-contained domains
→
Shared
Reusable utilities
Circular dependency trap: Feature A imports from Feature B, which imports from Feature A. This is the #1 sign you need to extract shared code. ESLint plugins like eslint-plugin-import can enforce these boundaries automatically.

Path Aliases

Instead of counting ../../../, set up path aliases in your tsconfig.json:

// tsconfig.json
{
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@features/*": ["src/features/*"],
      "@shared/*":   ["src/shared/*"],
      "@routes/*":   ["src/routes/*"]
    }
  }
}

Then imports become:

// Clean, absolute-style imports
import { LoginForm, useAuth } from '@features/auth'
import { Button } from '@shared/components/Button'
Vite support: Also add the aliases to vite.config.ts via resolve.alias or use the vite-tsconfig-paths plugin which reads them from tsconfig.json automatically.
