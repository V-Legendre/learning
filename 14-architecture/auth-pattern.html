14 — Authentication Pattern

JWT-based auth for React SPAs: token management, protected routes, and managed auth options.

The JWT Authentication Flow

JWT (JSON Web Token) auth is the standard pattern for SPAs. It's the same flow you'd use from any HTTP client — the frontend just adds UI state management on top.

1
Login
POST /auth/login with credentials
→
2
Receive Tokens
Access token + refresh token
→
3
Store Tokens
Access: memory. Refresh: httpOnly cookie
→
4
Attach to Requests
Authorization: Bearer <token>
→
5
Refresh on Expiry
Auto-refresh when 401 received
Python analogy: This is the exact same JWT flow you'd implement with Flask-JWT-Extended or FastAPI's OAuth2 password bearer. The only difference: on the frontend, you manage where to store tokens and how to attach them to requests.

Where to Store Tokens

Storage Access Token Refresh Token Notes
localStorage Never Never Accessible to any JS on the page (XSS attack vector)
JavaScript variable (memory) Yes No Lost on page refresh, but that's OK — refresh token restores it
httpOnly cookie No Yes Not accessible to JS, sent automatically by browser
Security rule: NEVER store tokens in localStorage or sessionStorage. Any JavaScript on the page (including third-party scripts and XSS payloads) can read them. The access token lives in a JavaScript variable (Zustand store), and the refresh token goes in an httpOnly cookie set by your backend.

Auth Store with Zustand

The auth store holds the current user, access token, and provides login/logout actions.

// features/auth/types.ts

export interface User {
  id: string
  email: string
  name: string
  role: 'admin' | 'user'
}

export interface LoginData {
  email: string
  password: string
}
// features/auth/store.ts

import { create } from 'zustand'
import type { User, LoginData } from './types'
import { loginApi, logoutApi, refreshApi } from './api'

interface AuthState {
  user: User | null
  token: string | null
  isAuthenticated: boolean
  login: (credentials: LoginData) => Promise<void>
  logout: () => void
  refresh: () => Promise<void>
  setAuth: (user: User, token: string) => void
  clearAuth: () => void
}

export const useAuthStore = create<AuthState>((set) => ({
  user: null,
  token: null,
  isAuthenticated: false,

  login: async (credentials) => {
    const { user, accessToken } = await loginApi(credentials)
    // Refresh token is set as httpOnly cookie by the server
    set({ user, token: accessToken, isAuthenticated: true })
  },

  logout: () => {
    logoutApi() // clears httpOnly cookie on server
    set({ user: null, token: null, isAuthenticated: false })
  },

  refresh: async () => {
    const { user, accessToken } = await refreshApi()
    set({ user, token: accessToken, isAuthenticated: true })
  },

  setAuth: (user, token) =>
    set({ user, token, isAuthenticated: true }),

  clearAuth: () =>
    set({ user: null, token: null, isAuthenticated: false }),
}))

Auth API Layer

// features/auth/api.ts

import type { User, LoginData } from './types'

const API_URL = import.meta.env.VITE_API_URL

interface AuthResponse {
  user: User
  accessToken: string
}

export const loginApi = async (data: LoginData): Promise<AuthResponse> => {
  const res = await fetch(`${API_URL}/auth/login`, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    credentials: 'include', // send/receive cookies
    body: JSON.stringify(data),
  })
  if (!res.ok) throw new Error('Login failed')
  return res.json()
}

export const refreshApi = async (): Promise<AuthResponse> => {
  const res = await fetch(`${API_URL}/auth/refresh`, {
    method: 'POST',
    credentials: 'include', // sends httpOnly cookie
  })
  if (!res.ok) throw new Error('Refresh failed')
  return res.json()
}

export const logoutApi = () =>
  fetch(`${API_URL}/auth/logout`, {
    method: 'POST',
    credentials: 'include',
  })

Protected Routes

With TanStack Router, you protect routes using beforeLoad — a function that runs before the route renders. If the user isn't authenticated, redirect to login.

// routes/_authenticated.tsx
// Layout route — all child routes require auth

import { createFileRoute, redirect } from '@tanstack/react-router'
import { useAuthStore } from '@features/auth'

export const Route = createFileRoute('/_authenticated')({
  beforeLoad: () => {
    const { isAuthenticated } = useAuthStore.getState()
    if (!isAuthenticated) {
      throw redirect({
        to: '/login',
        search: { redirect: location.pathname },
      })
    }
  },
})

Login / Logout Flow

Login component

// features/auth/components/LoginForm.tsx

import { useAuthStore } from '../store'
import { useNavigate, useSearch } from '@tanstack/react-router'

export const LoginForm = () => {
  const login = useAuthStore((s) => s.login)
  const navigate = useNavigate()
  const { redirect } = useSearch({ from: '/login' })

  const handleSubmit = async (e: React.FormEvent<HTMLFormElement>) => {
    e.preventDefault()
    const formData = new FormData(e.currentTarget)

    await login({
      email: formData.get('email') as string,
      password: formData.get('password') as string,
    })

    // Redirect to intended page or dashboard
    navigate({ to: redirect ?? '/dashboard' })
  }

  return (
    <form onSubmit={handleSubmit}>
      <input name="email" type="email" required />
      <input name="password" type="password" required />
      <button type="submit">Log in</button>
    </form>
  )
}

Logout action

// In a header or settings component

import { useAuthStore } from '@features/auth'
import { useQueryClient } from '@tanstack/react-query'
import { useNavigate } from '@tanstack/react-router'

const handleLogout = () => {
  useAuthStore.getState().logout()

  // Clear ALL cached data — it belongs to the old user
  queryClient.clear()

  // Redirect to login
  navigate({ to: '/login' })
}

Token Refresh Interceptor

When the access token expires, API requests return a 401. Instead of logging the user out, you silently refresh the token and retry the request.

// shared/utils/api.ts (enhanced version)

import { useAuthStore } from '@features/auth'

let isRefreshing = false
let refreshPromise: Promise<void> | null = null

const fetchWithAuth = async (
  url: string,
  options: RequestInit = {}
): Promise<Response> => {
  const { token } = useAuthStore.getState()

  const res = await fetch(url, {
    ...options,
    headers: {
      ...options.headers,
      Authorization: `Bearer ${token}`,
    },
    credentials: 'include',
  })

  if (res.status === 401) {
    // Prevent multiple simultaneous refresh requests
    if (!isRefreshing) {
      isRefreshing = true
      refreshPromise = useAuthStore.getState().refresh()
        .finally(() => { isRefreshing = false })
    }
    await refreshPromise

    // Retry original request with new token
    const { token: newToken } = useAuthStore.getState()
    return fetch(url, {
      ...options,
      headers: { ...options.headers, Authorization: `Bearer ${newToken}` },
      credentials: 'include',
    })
  }

  return res
}
Key detail: The isRefreshing flag ensures that if 5 API calls fail simultaneously with 401, only ONE refresh request is made. All others wait for the same promise.

Managed Auth Options

Building auth from scratch is complex and security-sensitive. For many projects, a managed auth provider handles the hard parts for you.

Provider Best For Pricing Notes
Supabase Auth Full-stack with Supabase DB Free tier generous Built-in to Supabase. Email, OAuth, magic link. Row-level security.
Clerk Best DX, pre-built UI Free up to 10K MAU Drop-in components. Session management. Multi-factor. Great React SDK.
Auth0 Enterprise, complex requirements Free tier, then expensive Very configurable. RBAC, SSO, compliance features.
Firebase Auth Google ecosystem Free (generous) Phone auth, email, OAuth. Tightly coupled to Firebase.
Use managed auth when...
  • Building an MVP / shipping fast
  • You need OAuth (Google, GitHub login)
  • You don't want to handle password hashing, email verification, rate limiting
  • You need compliance features (SOC2, GDPR)
Roll your own when...
  • You need full control over the auth flow
  • Cost at scale is a concern (managed auth charges per user)
  • Your backend already has auth (Django/FastAPI)
  • Regulatory requirements demand self-hosted auth

Security Checklist

Frontend auth security essentials
Rule Why
Never store tokens in localStorage XSS can read it. Any malicious script gets your token.
Use httpOnly cookies for refresh tokens JavaScript cannot access httpOnly cookies.
Short-lived access tokens (15 min) Limits damage window if token is leaked.
CSRF protection on cookie endpoints Cookies are sent automatically — verify origin header or use CSRF tokens.
Always validate on the server Frontend auth is UX. Backend auth is security. Never trust the client.
Clear query cache on logout Cached data belongs to the previous user.
Use credentials: 'include' in fetch Required for cookies to be sent cross-origin.
Python analogy: This is the same as Flask-Login's session management, just split differently. In Flask, the session cookie handles everything. In a SPA, you split into access token (for API calls) + refresh token (for session continuity). The security principles are identical: httpOnly, secure, short-lived, server-validated.
