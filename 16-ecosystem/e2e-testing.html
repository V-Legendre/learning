16 — E2E Testing (Playwright)

Test the full application from a real user's perspective — real browser, real clicks, real navigation.

What E2E Tests Cover

The testing pyramid

E2E tests sit at the top of the pyramid — they're the most realistic but also the slowest and most expensive to maintain.

LayerScopeSpeedQuantityTool
UnitFunctions, utilities, storesFastestManyVitest
ComponentIndividual components + mocked APIsFastSomeVitest + Testing Library
E2EFull app, real browser, real APISlowestFewPlaywright

E2E tests catch problems that unit/component tests can't:

  • Routing and navigation between pages
  • Authentication flows end-to-end
  • Real API integration issues
  • Browser-specific bugs (CSS, JavaScript compatibility)
  • Multi-step user journeys (signup → onboard → dashboard)
Python analogy: Like Selenium tests but modern. Or like integration tests that spin up the full app and hit real endpoints. The frontend equivalent of pytest with a live server + real database.

Writing E2E Tests

Basic Navigation Test

// e2e/navigation.spec.ts
import { test, expect } from "@playwright/test"

test("homepage has title and navigation", async ({ page }) => {
  await page.goto("/")

  // Check page title
  await expect(page).toHaveTitle(/My App/)

  // Check heading
  await expect(page.getByRole("heading", { name: "Welcome" })).toBeVisible()

  // Click a navigation link
  await page.getByRole("link", { name: "About" }).click()

  // Verify URL changed
  await expect(page).toHaveURL(/\/about/)
})

Login Flow

test("user can log in and see dashboard", async ({ page }) => {
  await page.goto("/login")

  // Fill in the form
  await page.getByLabel("Email").fill("test@example.com")
  await page.getByLabel("Password").fill("password123")

  // Submit
  await page.getByRole("button", { name: "Sign in" }).click()

  // Verify redirect to dashboard
  await expect(page).toHaveURL("/dashboard")
  await expect(page.getByText("Welcome, Test User")).toBeVisible()
})

Form Submission with Validation

test("shows validation errors and submits successfully", async ({ page }) => {
  await page.goto("/contact")

  // Submit empty form
  await page.getByRole("button", { name: "Send" }).click()

  // Check validation errors appear
  await expect(page.getByText("Name is required")).toBeVisible()
  await expect(page.getByText("Email is required")).toBeVisible()

  // Fill in valid data
  await page.getByLabel("Name").fill("Alice")
  await page.getByLabel("Email").fill("alice@example.com")
  await page.getByLabel("Message").fill("Hello!")

  // Submit
  await page.getByRole("button", { name: "Send" }).click()

  // Check success
  await expect(page.getByText("Message sent!")).toBeVisible()
})

Finding Elements (Locators)

Playwright locators auto-wait for elements to be visible and actionable — no manual sleep or waitFor needed.

LocatorExampleWhen to use
getByRolepage.getByRole("button", { name: "Save" })Best — matches accessibility tree
getByLabelpage.getByLabel("Email")Form inputs with labels
getByTextpage.getByText("Welcome")Static text content
getByPlaceholderpage.getByPlaceholder("Search...")Inputs without labels
getByTestIdpage.getByTestId("sidebar")Last resort
locatorpage.locator(".card >> text=Alice")Complex CSS/text combos
Same priority as Testing Library: Prefer getByRole → getByLabel → getByText → getByTestId. Accessible locators make your app more accessible too.

Running Tests

$ pnpm playwright test # Run all tests
$ pnpm playwright test --ui # Interactive UI mode
$ pnpm playwright test login.spec.ts # Run specific file
$ pnpm playwright test --project=chromium # Specific browser
$ pnpm playwright show-report # Open HTML report
$ pnpm playwright codegen # Record interactions → generate code
playwright codegen is incredible for beginners — it opens a browser, records your interactions, and generates the test code for you. Great for bootstrapping tests quickly.

When to Add E2E Tests

Good candidates for E2E
  • Critical user flows: signup, login, checkout, payment
  • Multi-page journeys that span several components and routes
  • Flows that involve real API interactions and data persistence
  • Smoke tests for core functionality (does the app load? can you navigate?)
Don't E2E test these
  • Individual component behavior (use component tests)
  • Edge cases and validation rules (use unit tests)
  • Every permutation of a feature (test the happy path + one error path)
  • Styling/visual details (use Storybook + visual regression)
E2E tests are expensive. They're slow to run, flaky if not careful, and expensive to maintain. Aim for 10-20 E2E tests covering critical paths, not 200 E2E tests covering everything. The bulk of your test coverage should come from unit and component tests.
