Server-side rendering, React Server Components, and when you actually need Next.js vs a Vite SPA.

The Problem with SPAs

Everything we've built so far is a Single Page Application (SPA). The server sends an empty HTML shell, and JavaScript builds the entire page in the browser.

Why this matters

Problem Impact
SEO Search engine crawlers see an empty <div>. Google can execute JS, but other crawlers (social media previews, etc.) often can't.
First load Users on slow connections (mobile, developing countries) stare at a blank screen while JS downloads and executes.
Time to interactive Even after HTML appears, the user has to wait for all JS to download before anything is clickable.
Python analogy: In Flask/Django, the server sends a fully rendered HTML page. The browser displays it immediately. That's what we want to get back to — but with React's interactivity.

Server-Side Rendering (SSR)

SSR renders your React components to HTML on the server, sends that HTML to the browser, then React "hydrates" it (attaches event listeners to make it interactive).

1
Request
Browser asks for /products
→
2

React runs on Node.js, generates HTML
→
3
Full HTML
Complete page with content
→
4
Visible instantly
User sees content
→
5
Hydration
JS loads, makes it interactive
SPA (Client-Side Rendering)
  • Server sends empty HTML
  • Browser downloads JS (150KB+)
  • JS renders the page
  • Then JS fetches data
  • Then page is visible

Time to content: 2-5 seconds on slow connections

SSR (Server-Side Rendering)
  • Server fetches data + renders HTML
  • Browser receives complete page
  • Content is visible immediately
  • JS downloads in background
  • Hydration makes it interactive

Time to content: <1 second (server is fast)

React Server Components (RSC)

RSC is React's newest paradigm (and its future direction). Components that run only on the server — their JavaScript is never sent to the browser.

Server Components vs Client Components
Feature Server Component Client Component
Where it runs Server only Server (SSR) + Browser (hydration)
JS sent to browser No (zero JS) Yes (component code shipped)
Can use hooks No (useState, useEffect, etc.) Yes
Can use event handlers No (onClick, onChange, etc.) Yes
Can access database Yes (directly, like Python) No (must go through API)
Can use env secrets Yes (server-only) No (would expose to browser)
Async component Yes (async function Component()) No 
// Server Component (default in Next.js App Router)
// This JavaScript NEVER reaches the browser

import { db } from './database' // Direct database access!

export default async function ProductList() {
  // This runs on the server — like a Django view
  const products = await db.product.findMany()

  return (
    <ul>
      {products.map((p) => (
        <li key={p.id}>{p.name} — ${p.price}</li>
      ))}
    </ul>
  )
}
// Client Component — add 'use client' directive
'use client'

import { useState } from 'react'

export const AddToCart = ({ productId }: { productId: string }) => {
  const [count, setCount] = useState(0)

  return (
    <button onClick={() => setCount((c) => c + 1)}>
      Add to cart ({count})
    </button>
  )
}
Mental model: In Next.js App Router, every component is a Server Component by default. Only add 'use client' when you need interactivity (state, effects, event handlers). Push 'use client' as far down the tree as possible.

Next.js App Router

Next.js is the most popular React framework. Its App Router (introduced in Next.js 13) uses Server Components by default with file-based routing.

app/
  layout.tsx ← root layout (wraps all pages)
  page.tsx ← / (home page)
  products/
    page.tsx ← /products
    [id]/
      page.tsx ← /products/123
  about/
    page.tsx ← /about

Rendering strategies in Next.js

Strategy When HTML is Generated Use Case
SSG (Static Site Generation) At build time Blog posts, docs, marketing pages
SSR (Server-Side Rendering) On each request Pages with user-specific or frequently changing data
ISR (Incremental Static Regeneration) At build time, then revalidates on a schedule Product pages (rebuild every 60s)
Client-side In the browser (like our Vite SPA) Interactive dashboards, auth-only content

When You Need Next.js vs Vite SPA

Next.js makes sense for...
  • SEO-critical pages — blogs, e-commerce, marketing sites, documentation
  • First load performance — content must be visible before JS loads
  • Public content — pages that Google must index
  • Hybrid apps — some pages public (SSR), some behind login (client)
  • Full-stack React — API routes, database access, all in one project
Vite SPA is fine for...
  • Dashboards — admin panels, analytics, internal tools
  • Apps behind login — no SEO needed for authenticated pages
  • Internal tools — company tools that aren't public
  • Simpler deployment — just static files, no Node.js server needed
  • Separate frontend/backend — frontend team deploys independently

This is what we're building.

Vite SPA vs Next.js

Feature Vite SPA Next.js
Rendering Client only Server + Client (hybrid)
SEO Poor (empty HTML initially) Excellent (full HTML from server)
First load speed Slower (wait for JS) Faster (HTML arrives ready)
Deployment Static files (CDN) Node.js server or Vercel
Hosting cost Nearly free (static hosting) Needs compute (server)
Complexity Simpler — just React More concepts (Server/Client components, caching, streaming)
Routing TanStack Router (type-safe) File-based (convention)
Data fetching TanStack Query (client-side) Server Components + fetch (server-side)
State management Zustand (client-side) Less needed (data fetched on server)
Learning curve Moderate Steep (many rendering modes, caching, Server vs Client components)
The industry trend: React is moving toward server-first with RSC. Next.js is the reference implementation. But SPAs aren't going away — they're the right choice for applications (as opposed to websites). Know both patterns exist, start with SPA, adopt Next.js when you need server rendering.

Other SSR Frameworks

Framework Approach Notes
Remix Web standards-first SSR Now merged into React Router v7. Less magic than Next.js, uses native form handling.
Astro Static-first, partial hydration Sends zero JS by default. Perfect for content sites. Can use React, Vue, Svelte components.
TanStack Start Full-stack React with TanStack Router New, by the TanStack team. Type-safe SSR with the router we already know.
Future path: Once you're comfortable with React SPAs (our current focus), the natural next step is to try Next.js for a project that needs SEO. Everything you've learned (components, hooks, TypeScript, Tailwind) carries over directly.
