16 — CSS Alternatives

Beyond Tailwind: CSS Modules, styled-components, vanilla-extract, and modern CSS features.

The CSS Scaling Problem

CSS has no built-in scoping. Every class name is global. In a large React app, this causes:

  • Naming collisions — two developers create .card with different styles
  • Dead code — CSS grows but nobody knows which rules are still used
  • Specificity wars — overriding styles requires increasingly specific selectors or !important

Every CSS approach below solves these problems differently. Tailwind is our primary choice, but knowing the alternatives helps you work in any codebase.

CSS Modules

CSS Modules automatically scope class names to the component that imports them. You write normal CSS, but class names are made unique at build time.

Button.module.css
/* Normal CSS — but class names are scoped */
.primary {
  background: #3b82f6;
  color: white;
  padding: 8px 16px;
  border-radius: 6px;
  border: none;
}

.primary:hover {
  background: #2563eb;
}

.danger {
  background: #ef4444;
  color: white;
}
Button.tsx
import styles from './Button.module.css'

interface ButtonProps {
  variant: 'primary' | 'danger'
  children: React.ReactNode
}

export const Button = ({
  variant,
  children,
}: ButtonProps) => (
  <button className={styles[variant]}>
    {children}
  </button>
)

// Rendered HTML:
// <button class="Button_primary_x7ht9">
// Unique hash = no conflicts
CSS Modules pros and cons

Pros

  • No naming conflicts (scoped by default)
  • Standard CSS syntax (no learning curve)
  • Zero runtime cost (resolved at build time)
  • Works with Vite out of the box
  • Easy to migrate existing CSS

Cons

  • No dynamic styles based on props (use inline styles or multiple classes)
  • Class name composition is awkward (clsx needed for conditionals)
  • Separate file per component
  • TypeScript doesn't know available class names by default

styled-components / Emotion (CSS-in-JS)

CSS-in-JS writes styles as JavaScript template literals. The styles live in the same file as the component and can be dynamic based on props.

// CSS-in-JS with styled-components
import styled from 'styled-components'

interface ButtonProps {
  variant: 'primary' | 'danger'
  size?: 'sm' | 'md' | 'lg'
}

const StyledButton = styled.button<ButtonProps>`
  background: ${(props) => props.variant === 'primary' ? '#3b82f6' : '#ef4444'};
  color: white;
  padding: ${(props) => props.size === 'lg' ? '12px 24px' : '8px 16px'};
  border-radius: 6px;
  border: none;
  cursor: pointer;

  &:hover {
    opacity: 0.9;
  }
`

// Usage — props drive the styles
<StyledButton variant="primary" size="lg">Save</StyledButton>
CSS-in-JS pros and cons

Pros

  • Dynamic styles based on props/state
  • Co-located with component (no separate file)
  • Full power of JavaScript in styles
  • Automatic scoping
  • Theme support built in

Cons

  • Runtime cost — styles generated in the browser
  • Larger bundle — library (12KB+) + all style strings
  • RSC incompatible — doesn't work with React Server Components
  • Performance issues with many dynamic styles
  • Community moving away from this pattern
Industry trend: CSS-in-JS (styled-components, Emotion) had their peak around 2019-2021. The React community is moving away from runtime CSS-in-JS because of performance costs and Server Component incompatibility. New projects should prefer Tailwind, CSS Modules, or vanilla-extract.

vanilla-extract

The best of both worlds: write styles in TypeScript, but they compile to static CSS at build time. Zero runtime cost.

// Button.css.ts (note: .css.ts extension)
import { style, styleVariants } from '@vanilla-extract/css'

export const base = style({
  color: 'white',
  padding: '8px 16px',
  borderRadius: '6px',
  border: 'none',
  cursor: 'pointer',
  ':hover': {
    opacity: 0.9,
  },
})

export const variant = styleVariants({
  primary: { background: '#3b82f6' },
  danger:  { background: '#ef4444' },
})
// Button.tsx
import { base, variant } from './Button.css'

export const Button = ({ v, children }: Props) => (
  <button className={`${base} ${variant[v]}`}>
    {children}
  </button>
)

// TypeScript knows variant keys: 'primary' | 'danger'
// Typos caught at compile time!
vanilla-extract pros and cons

Pros

  • Zero runtime (CSS extracted at build)
  • Full TypeScript type safety
  • Works with Server Components
  • Atomic CSS recipes (like Tailwind)
  • Theme tokens with type safety

Cons

  • More setup required
  • Learning curve (new API)
  • Smaller community than Tailwind
  • Verbose for simple styles

Modern CSS Features

Modern CSS itself is solving many problems these tools were created for. These features are available in all modern browsers.

Feature What It Does Browser Support
CSS Nesting Native nesting like Sass: .card { & .title { } } All modern browsers (2024+)
CSS Layers (@layer) Control specificity order. Base, components, utilities layers. All modern browsers
@scope Scope styles to a subtree of the DOM Chrome, Edge (2024)
:where() / :is() Group selectors. :where() has zero specificity. All modern browsers
Container queries Responsive based on parent size, not viewport All modern browsers
CSS Custom Properties --color-primary: #3b82f6 — theme tokens in CSS All browsers (long supported) 
/* Modern CSS with nesting, layers, and custom properties */
@layer base, components, utilities;

@layer components {
  .card {
    background: var(--color-surface);
    border-radius: 8px;
    padding: 16px;

    /* Native nesting! */
    & .title {
      font-weight: bold;
      color: var(--color-text);
    }

    &:hover {
      box-shadow: 0 2px 8px rgba(0,0,0,0.1);
    }
  }
}

Why We Picked Tailwind

The case for Tailwind in 2026
  • Consistent design system — spacing, colors, typography all come from a predefined scale. No more "should this be 14px or 15px?"
  • No naming — never spend time naming CSS classes again. className="flex items-center gap-4 p-6" is self-documenting.
  • Fast iteration — change styles directly in JSX, no context switching between files
  • Tiny production CSS — only the utilities you actually use are included (typically 5-15KB gzipped)
  • Ecosystem — shadcn/ui gives you beautiful, accessible, customizable components built on Tailwind
  • Server Component safe — no runtime, no JavaScript, just class names
  • Community momentum — most popular approach for new React projects
When to pick something else: If you join a team using CSS Modules or styled-components, work within their system. All of these approaches are valid. The important thing is consistency within a project.
