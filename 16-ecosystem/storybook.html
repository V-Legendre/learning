16 — Storybook

Building UI components in isolation: stories, component-driven development, and visual regression testing.

What Is Storybook?

Storybook is a tool for developing UI components in isolation. Instead of running your full app and navigating to a page to see a component, you build it in a separate environment where each component gets its own "story" — a rendered example with specific props.

The problem Storybook solves
Without Storybook
  • Start the full app
  • Log in (if behind auth)
  • Navigate to the page
  • Set up the right data state
  • Finally see your component
  • Change code, repeat...
With Storybook
  • Open Storybook (separate dev server)
  • Pick your component from the sidebar
  • See it instantly with various prop combinations
  • Change code, see updates immediately
  • No app, no auth, no data setup
Python analogy: Think of Storybook like Swagger UI for your components. Swagger lets you interact with API endpoints in isolation. Storybook lets you interact with UI components in isolation.

Writing Stories

A story file exports multiple examples ("stories") of a component, each showing different props/states.

The component

// shared/components/Button.tsx

interface ButtonProps {
  variant: 'primary' | 'secondary' | 'danger'
  size: 'sm' | 'md' | 'lg'
  disabled?: boolean
  loading?: boolean
  children: React.ReactNode
  onClick?: () => void
}

export const Button = ({
  variant,
  size,
  disabled,
  loading,
  children,
  onClick,
}: ButtonProps) => {
  return (
    <button
      className={cn('btn', `btn-${variant}`, `btn-${size}`)}
      disabled={disabled || loading}
      onClick={onClick}
    >
      {loading ? 'Loading...' : children}
    </button>
  )
}

The stories file

// shared/components/Button.stories.tsx

import type { Meta, StoryObj } from '@storybook/react'
import { Button } from './Button'

// Meta: describes the component
const meta: Meta<typeof Button> = {
  title: 'Shared/Button',     // sidebar path
  component: Button,
  tags: ['autodocs'],         // auto-generate docs page
  argTypes: {
    onClick: { action: 'clicked' }, // logs clicks in Storybook panel
  },
}

export default meta
type Story = StoryObj<typeof Button>

// Each export is a story (a rendered example)

export const Primary: Story = {
  args: {
    variant: 'primary',
    size: 'md',
    children: 'Click me',
  },
}

export const Secondary: Story = {
  args: {
    variant: 'secondary',
    size: 'md',
    children: 'Cancel',
  },
}

export const Danger: Story = {
  args: {
    variant: 'danger',
    size: 'md',
    children: 'Delete',
  },
}

export const Loading: Story = {
  args: {
    variant: 'primary',
    size: 'md',
    loading: true,
    children: 'Save',
  },
}

export const Disabled: Story = {
  args: {
    variant: 'primary',
    size: 'md',
    disabled: true,
    children: 'Disabled',
  },
}

export const AllSizes: Story = {
  render: () => (
    <div style={{ display: 'flex', gap: '12px', alignItems: 'center' }}>
      <Button variant="primary" size="sm">Small</Button>
      <Button variant="primary" size="md">Medium</Button>
      <Button variant="primary" size="lg">Large</Button>
    </div>
  ),
}
Key pattern: Each story is a named export with an args object. Storybook uses args to generate interactive controls — you can tweak props in the UI panel. The render function gives you custom layout for more complex examples.

Component-Driven Development

Storybook encourages a bottom-up development workflow: build small components, then compose them into larger ones.

1
Atoms
Button, Input, Badge, Avatar
→
2
Molecules
SearchBar, UserCard, FormField
→
3
Organisms
Header, Sidebar, DataTable
→
4
Pages
DashboardPage, SettingsPage

At each level, the component has stories showing all its states. When you build pages, you're just wiring together already-tested pieces.

Interaction stories

Storybook can also simulate user interactions in stories, useful for testing flows:

// LoginForm.stories.tsx
import { within, userEvent } from '@storybook/test'

export const FilledForm: Story = {
  play: async ({ canvasElement }) => {
    const canvas = within(canvasElement)

    // Simulate typing
    await userEvent.type(
      canvas.getByLabelText('Email'),
      'user@example.com'
    )
    await userEvent.type(
      canvas.getByLabelText('Password'),
      'password123'
    )

    // Click submit
    await userEvent.click(
      canvas.getByRole('button', { name: 'Log in' })
    )
  },
}

When Storybook Is Worth It

Worth it when...
  • Design system / component library — shared components used across apps need documentation
  • Team with designers — Storybook serves as a visual contract between design and dev
  • Complex component states — loading, error, empty, disabled, responsive — hard to test all in the app
  • Visual regression testing — Chromatic/Percy can screenshot every story on every PR
  • Onboarding — new developers can browse all components and their props
Skip it when...
  • Small solo project — the overhead of maintaining stories isn't worth it
  • Rapid prototyping — stories slow down iteration when the API is changing daily
  • Few shared components — if most components are page-specific, there's less value
  • No visual QA process — Storybook shines when someone reviews visuals

Visual Regression Testing

The killer feature of Storybook at scale: automatically detect visual changes on every PR.

1
PR opened
Developer changes a component
→
2
CI runs
Builds Storybook, publishes to Chromatic
→
3
Screenshots
Every story captured
→
4
Diff
Compared to baseline (main branch)
→
5
Review
Approve or reject visual changes
Tool Type Notes
Chromatic Paid (free tier: 5,000 snapshots/month) By the Storybook team. Tightest integration. Catches UI changes, shows pixel diffs.
Percy Paid By BrowserStack. Cross-browser visual testing. Works with or without Storybook.
Playwright screenshots Free (DIY) Use Playwright to screenshot pages/components, compare in CI. More manual setup.
