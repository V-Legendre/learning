16 — GraphQL

A query language for APIs — the client asks for exactly what it needs, nothing more.

What Is GraphQL?

Core idea

GraphQL is a query language and runtime for APIs. Instead of multiple REST endpoints returning fixed data shapes, you have one endpoint where the client specifies the exact fields it needs. The server returns only those fields.

Python analogy: GraphQL is like writing SQL queries from the frontend. REST is like calling stored procedures or pre-defined views — the backend decides what data comes back. GraphQL lets the frontend decide.

GraphQL vs REST

Imagine fetching a user's profile page that shows their name, their 5 most recent posts, and each post's comments.

REST — 3 requests
// Request 1: Get user
GET /api/users/42
// Returns: id, name, email, avatar,
// bio, createdAt, ... (20 fields)
// You only need name + avatar

// Request 2: Get posts
GET /api/users/42/posts?limit=5
// Returns: id, title, body, tags,
// likes, shares, ... (15 fields per post)
// You only need title + createdAt

// Request 3: Get comments for each post
GET /api/posts/101/comments
GET /api/posts/102/comments
// ... N+1 problem

Over-fetching (too many fields) + under-fetching (too many requests)

GraphQL — 1 request
# Single query, exact data needed
query {
  user(id: 42) {
    name
    avatar
    posts(limit: 5) {
      title
      createdAt
      comments {
        text
        author {
          name
        }
      }
    }
  }
}

Key Concepts

Schema — The Type System

The server defines a schema that describes all available data and operations. This is the contract between frontend and backend.

# Schema definition (server side)
type User {
  id: ID!
  name: String!
  email: String!
  posts: [Post!]!
}

type Post {
  id: ID!
  title: String!
  body: String!
  author: User!
  comments: [Comment!]!
}

type Query {
  user(id: ID!): User
  posts(limit: Int): [Post!]!
}

type Mutation {
  createPost(title: String!, body: String!): Post!
  deletePost(id: ID!): Boolean!
}
! means non-nullable. String! = always a string, never null. [Post!]! = always an array (never null), containing non-null Posts. Like TypeScript's strict null checks but in the schema.

Queries — Read Data

# Client sends this query
query GetUser {
  user(id: "42") {
    name
    email
    posts {
      title
    }
  }
}

# Server responds with exactly this shape
{
  "data": {
    "user": {
      "name": "Alice",
      "email": "alice@example.com",
      "posts": [
        { "title": "My First Post" },
        { "title": "GraphQL Is Neat" }
      ]
    }
  }
}

Mutations — Write Data

mutation CreatePost {
  createPost(title: "Hello World", body: "My first post") {
    id
    title
    author {
      name
    }
  }
}

Subscriptions — Real-Time Data

# Subscribe to new messages (WebSocket)
subscription OnNewMessage {
  messageAdded(channelId: "general") {
    text
    sender {
      name
    }
    createdAt
  }
}

Basic Apollo Client Usage

import { useQuery, gql } from "@apollo/client"

const GET_USERS = gql`
  query GetUsers {
    users {
      id
      name
      email
    }
  }
`

const UserList = () => {
  const { loading, error, data } = useQuery(GET_USERS)

  if (loading) return <p>Loading...</p>
  if (error) return <p>Error: {error.message}</p>

  return (
    <ul>
      {data.users.map((user) => (
        <li key={user.id}>{user.name}</li>
      ))}
    </ul>
  )
}
Notice the similarity to TanStack Query. The hooks pattern (useQuery returning { loading, error, data }) is the same concept. If you know TanStack Query, GraphQL clients feel familiar — the difference is the query language, not the React integration.

TypeScript + GraphQL Codegen

One of GraphQL's best features: auto-generate TypeScript types from your schema. No manual type definitions.

// Without codegen: manual types that might drift
interface User {
  id: string
  name: string
  email: string  // did the server add a field? remove one? you don't know
}

// With codegen: types are auto-generated from the schema
// Run: pnpm graphql-codegen
// Generates typed hooks automatically:
const { data } = useGetUsersQuery()
// data.users[0].name — fully typed, autocomplete works
// data.users[0].nonExistent — TypeScript error!
GraphQL Codegen (@graphql-codegen/cli) reads your schema + queries and generates TypeScript types and typed React hooks. It's the main reason GraphQL has excellent TypeScript DX.

When GraphQL Is Worth It

GraphQL shines when...
  • Multiple clients need different data shapes (web, mobile, third-party)
  • Complex, deeply nested data relationships
  • Backend team is separate from frontend team (schema = contract)
  • You want auto-generated TypeScript types from the schema
  • You need real-time subscriptions
  • Large, evolving APIs where versioning is painful
REST is simpler when...
  • Simple CRUD with one client (you control both ends)
  • Small API surface area
  • You want simple HTTP caching (each URL = cacheable resource)
  • File uploads are a core use case
  • Team is unfamiliar with GraphQL and there's no time to learn
  • Most projects should start with REST
Our roadmap uses REST + TanStack Query. For most apps (especially those behind a login where SEO doesn't matter), REST with a good caching layer (TanStack Query) is simpler and sufficient. GraphQL adds value when the data graph is complex or when you have multiple consumers. Don't adopt it just because it's trendy.

Common Concerns

ConcernReality
"Isn't it slow?" Not inherently. Performance depends on your resolvers and query complexity, just like REST depends on your endpoints. Use dataloader to batch database queries.
"Can clients query anything?" You control the schema. Use depth limiting, query complexity analysis, and persisted queries to prevent abuse.
"How do I cache?" Apollo and urql have normalized caches. Not as simple as HTTP caching, but more powerful for complex data relationships.
"Do I need to rewrite my backend?" No. A GraphQL server can sit in front of existing REST APIs, databases, or microservices. It's a gateway layer.
