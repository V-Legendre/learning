13 — Testing Setup

Setting up Vitest and Testing Library for a Vite + React + TypeScript project.

Why Vitest Over Jest?

Python Analogy

Vitest is to React what pytest is to Python:

  • Fast, modern, opinionated
  • Compatible with the old standard (Jest) but better DX
  • Auto-discovery of test files
  • Watch mode by default
  • Rich assertion library

Testing Library is like testing the behavior users see, rather than internal implementation. Think of it as testing your web app the way a QA person would.

Vitest advantages
  • Same config as Vite — no separate build pipeline for tests
  • Native ESM — no transpilation issues, import/export just works
  • Faster — reuses Vite's module resolution and transform pipeline
  • Compatible API — drop-in replacement for Jest (describe, it, expect)
  • TypeScript out of the box — no ts-jest needed
  • Watch mode — only re-runs tests for changed files

Configuration

vite.config.ts

import { defineConfig } from 'vite'
import react from '@vitejs/plugin-react'

export default defineConfig({
  plugins: [react()],
  test: {
    // DOM environment for component tests
    environment: 'happy-dom',

    // Make describe, it, expect globally available
    // (no need to import from 'vitest' in every test file)
    globals: true,

    // Run this file before all tests
    setupFiles: './src/test/setup.ts',

    // CSS handling (don't process CSS in tests)
    css: false,
  },
})
Note: The test field is part of Vitest's config, which extends Vite's config. This means one config file for both building and testing.

tsconfig.json — Add Vitest types

{
  "compilerOptions": {
    // ... existing options
    "types": ["vitest/globals"]
  }
}

This gives you TypeScript autocompletion for describe, it, expect, vi, etc. without imports.

Test File Conventions

Convention Example Notes
File pattern *.test.ts, *.test.tsx Vitest auto-discovers these
Location Next to the file being tested Co-locate: Button.tsx + Button.test.tsx
Helpers/fixtures src/test/ folder Shared test utilities, mocks, factories
Extension .test.tsx (not .spec.tsx) Convention in Vite ecosystem
Python (pytest)
# test_calculator.py
def test_add():
    assert add(1, 2) == 3

def test_subtract():
    assert subtract(5, 3) == 2

class TestCalculator:
    def test_multiply(self):
        assert multiply(3, 4) == 12
Vitest equivalent
// calculator.test.ts
describe('Calculator', () => {
  it('adds numbers', () => {
    expect(add(1, 2)).toBe(3)
  })

  it('subtracts numbers', () => {
    expect(subtract(5, 3)).toBe(2)
  })

  it('multiplies numbers', () => {
    expect(multiply(3, 4)).toBe(12)
  })
})

happy-dom vs jsdom

Feature happy-dom jsdom
Speed 2-3x faster Baseline
API completeness Most common APIs More complete

Recommendation: Start with happy-dom. It's faster and works for 95% of tests. Switch to jsdom only if you encounter a missing DOM API.

Your First Test

Let's verify the setup works with a minimal test.

src/components/Greeting.tsx
interface GreetingProps {
  name: string
}

export const Greeting = ({ name }: GreetingProps) => {
  return <h1>Hello, {name}!</h1>
}
src/components/Greeting.test.tsx
import { render, screen } from '@testing-library/react'
import { Greeting } from './Greeting'

describe('Greeting', () => {
  it('renders a greeting with the given name', () => {
    render(<Greeting name="Alice" />)

    expect(
      screen.getByText('Hello, Alice!')
    ).toBeInTheDocument()
  })
})

Common Assertion Matchers

Vitest built-in (from expect)

Matcher Python equivalent Example
.toBe(value) assert x == y expect(1 + 1).toBe(2)
.toEqual(obj) assert x == y (deep) expect({a: 1}).toEqual({a: 1})
.toBeTruthy() assert x expect(result).toBeTruthy()
.toHaveLength(n) assert len(x) == n expect([1,2]).toHaveLength(2)
.toThrow() with pytest.raises() expect(() => fn()).toThrow('msg')
.toContain(item) assert x in y expect([1,2,3]).toContain(2)

DOM matchers (from @testing-library/jest-dom)

Matcher What it checks
.toBeInTheDocument() Element exists in the DOM
.toHaveTextContent('text') Element contains text
.toBeDisabled() Input/button is disabled
.toBeVisible() Element is visible (not hidden)
.toHaveClass('name') Element has CSS class
.toHaveValue('val') Input has specific value
.toHaveAttribute('attr', 'val') Element has HTML attribute with value
.toBeChecked() Checkbox/radio is checked
