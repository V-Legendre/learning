13 — Component Testing

Test behavior, not implementation. What does the user see and do?

Testing Philosophy

The golden rule

Test your components the way a user interacts with them. Don't test internal state, hook calls, or implementation details. If you refactor a component's internals and the behavior doesn't change, no tests should break.

Implementation testing
// Brittle — breaks on refactor
expect(component.state.count).toBe(1)
expect(useStateSpy).toHaveBeenCalledWith(0)
expect(wrapper.find(".counter-value"))
  .text().toBe("1")
Behavior testing
// Resilient — tests what users see
const user = userEvent.setup()
render(<Counter />)

await user.click(
  screen.getByRole("button", { name: /increment/i })
)
expect(screen.getByText("Count: 1"))
  .toBeInTheDocument()
Python analogy: Like testing a Flask endpoint by sending HTTP requests and checking responses, not by inspecting internal function calls. Black-box testing at the right boundary.

Core API: render, screen, queries

render

import { render, screen } from "@testing-library/react"

// Render a component into the virtual DOM
render(<Button label="Save" onClick={mockFn} />)

// With providers (theme, query client, router, etc.)
render(<Button />, {
  wrapper: ({ children }) => (
    <QueryClientProvider client={queryClient}>
      {children}
    </QueryClientProvider>
  ),
})

Query Priority (most to least preferred)

PriorityQueryExampleWhy
1 getByRole getByRole("button", { name: /submit/i }) Accessibility tree — how screen readers see it
2 getByLabelText getByLabelText("Email") Form inputs — what the label says
3 getByPlaceholderText getByPlaceholderText("Search...") When no label exists
4 getByText getByText("Welcome back") Non-interactive content
5 getByTestId getByTestId("nav-menu") Last resort — no semantic way to find it
Avoid getByTestId as your first choice. It tests implementation (you added a data-testid), not behavior. If you need it, that's a hint your component may have accessibility gaps.

User Events

@testing-library/user-event simulates real user interactions (click, type, tab, etc.) more accurately than fireEvent.

import userEvent from "@testing-library/user-event"

// Always call setup() first — creates an instance with proper state tracking
const user = userEvent.setup()

// Click
await user.click(screen.getByRole("button", { name: /save/i }))

// Type text
await user.type(screen.getByRole("textbox"), "hello@example.com")

// Clear then type (replace existing value)
await user.clear(screen.getByRole("textbox"))
await user.type(screen.getByRole("textbox"), "new value")

// Select dropdown option
await user.selectOptions(
  screen.getByRole("combobox"),
  screen.getByRole("option", { name: "France" })
)

// Check/uncheck checkbox
await user.click(screen.getByRole("checkbox"))

// Tab through elements
await user.tab()

// Keyboard shortcuts
await user.keyboard("{Enter}")
await user.keyboard("{Escape}")
Why userEvent over fireEvent? userEvent simulates the full browser event chain: focus → keydown → keypress → input → keyup. fireEvent only dispatches a single event. Use userEvent for realistic interaction testing.

Common Assertions

Extended matchers from @testing-library/jest-dom:

// Presence
expect(screen.getByText("Welcome")).toBeInTheDocument()
expect(screen.queryByText("Error")).not.toBeInTheDocument()

// Content
expect(screen.getByRole("heading")).toHaveTextContent("Dashboard")

// Form state
expect(screen.getByRole("button")).toBeDisabled()
expect(screen.getByRole("textbox")).toHaveValue("alice@example.com")
expect(screen.getByRole("checkbox")).toBeChecked()

// Visibility
expect(element).toBeVisible()

// CSS classes
expect(element).toHaveClass("active")

// Attributes
expect(link).toHaveAttribute("href", "/dashboard")

Mocking APIs with MSW

Mock Service Worker intercepts actual network requests at the service worker level. Your component code doesn't know it's being mocked — it makes real fetch calls that MSW intercepts.

Setup

// Install
// pnpm add -D msw

// src/test/handlers.ts — define API mocks
import { http, HttpResponse } from "msw"

export const handlers = [
  http.get("/api/users", () => {
    return HttpResponse.json([
      { id: 1, name: "Alice", email: "alice@example.com" },
      { id: 2, name: "Bob", email: "bob@example.com" },
    ])
  }),

  http.post("/api/users", async ({ request }) => {
    const body = await request.json()
    return HttpResponse.json(
      { id: 3, ...body },
      { status: 201 }
    )
  }),

  http.delete("/api/users/:id", ({ params }) => {
    return new HttpResponse(null, { status: 204 })
  }),
]
// src/test/server.ts — create test server
import { setupServer } from "msw/node"
import { handlers } from "./handlers"

export const server = setupServer(...handlers)
// src/test/setup.ts — wire into Vitest
import { server } from "./server"

beforeAll(() => server.listen())
afterEach(() => server.resetHandlers())  // reset per-test overrides
afterAll(() => server.close())

Override Handlers in Tests

it("shows error when API fails", async () => {
  // Override the default handler for this test only
  server.use(
    http.get("/api/users", () => {
      return HttpResponse.json(
        { message: "Internal Server Error" },
        { status: 500 }
      )
    })
  )

  render(<UserList />)

  await screen.findByText(/something went wrong/i)
})
Python analogy: MSW is like responses or httpretty in Python — intercepts HTTP calls at the transport level. But it works in the browser too (not just tests), so you can use the same mocks during development.

Complete Example: Testing a User Form

A realistic test covering form rendering, user interaction, API submission, and success feedback.

// CreateUserForm.test.tsx
import { render, screen, waitFor } from "@testing-library/react"
import userEvent from "@testing-library/user-event"
import { QueryClient, QueryClientProvider } from "@tanstack/react-query"
import { CreateUserForm } from "./CreateUserForm"

// Helper to wrap component with providers
const renderWithProviders = (ui: React.ReactElement) => {
  const queryClient = new QueryClient({
    defaultOptions: { queries: { retry: false } },
  })
  return render(ui, {
    wrapper: ({ children }) => (
      <QueryClientProvider client={queryClient}>
        {children}
      </QueryClientProvider>
    ),
  })
}

describe("CreateUserForm", () => {
  it("renders the form fields", () => {
    renderWithProviders(<CreateUserForm />)

    expect(screen.getByLabelText(/name/i)).toBeInTheDocument()
    expect(screen.getByLabelText(/email/i)).toBeInTheDocument()
    expect(screen.getByRole("button", { name: /create/i })).toBeInTheDocument()
  })

  it("shows validation errors for empty fields", async () => {
    const user = userEvent.setup()
    renderWithProviders(<CreateUserForm />)

    // Submit without filling fields
    await user.click(screen.getByRole("button", { name: /create/i }))

    expect(await screen.findByText(/name is required/i)).toBeInTheDocument()
    expect(screen.getByText(/email is required/i)).toBeInTheDocument()
  })

  it("submits the form and shows success", async () => {
    const user = userEvent.setup()
    renderWithProviders(<CreateUserForm />)

    // Fill in the form
    await user.type(screen.getByLabelText(/name/i), "Alice")
    await user.type(screen.getByLabelText(/email/i), "alice@example.com")

    // Submit
    await user.click(screen.getByRole("button", { name: /create/i }))

    // Button should show loading state
    expect(screen.getByRole("button")).toBeDisabled()

    // Wait for success message (API call via MSW)
    await screen.findByText(/user created successfully/i)

    // Form should be cleared
    expect(screen.getByLabelText(/name/i)).toHaveValue("")
  })

  it("shows error message on API failure", async () => {
    // Override handler for this test
    server.use(
      http.post("/api/users", () => {
        return HttpResponse.json(
          { message: "Email already exists" },
          { status: 409 }
        )
      })
    )

    const user = userEvent.setup()
    renderWithProviders(<CreateUserForm />)

    await user.type(screen.getByLabelText(/name/i), "Alice")
    await user.type(screen.getByLabelText(/email/i), "alice@example.com")
    await user.click(screen.getByRole("button", { name: /create/i }))

    await screen.findByText(/email already exists/i)
  })
})

Async Testing Patterns

findBy — wait for elements to appear

// Component fetches data on mount
render(<UserList />)

// findBy waits up to 1000ms (configurable) for the element
const users = await screen.findAllByRole("listitem")
expect(users).toHaveLength(2)

waitFor — wait for assertions to pass

// Wait for multiple things to be true
await waitFor(() => {
  expect(screen.getByText("Alice")).toBeInTheDocument()
  expect(screen.getByText("Bob")).toBeInTheDocument()
})

waitForElementToBeRemoved — wait for loading states to clear

render(<UserList />)

// Wait for loading spinner to disappear
await waitForElementToBeRemoved(
  () => screen.queryByText(/loading/i)
)

// Now assert on the loaded content
expect(screen.getByText("Alice")).toBeInTheDocument()

Snapshot Testing

Mostly avoid snapshots. They're brittle — any markup change (even whitespace) breaks them. Teams end up blindly updating snapshots. Test specific behavior instead. The one exception: small, stable components where you want to catch unintended markup changes (like an icon library). 
// If you must use them:
it("matches snapshot", () => {
  const { container } = render(<Badge variant="success">Active</Badge>)
  expect(container.firstChild).toMatchSnapshot()
})

// Inline snapshots are slightly better (visible in test file):
expect(container.firstChild).toMatchInlineSnapshot(`
  <span class="badge badge-success">Active</span>
`)
