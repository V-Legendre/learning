04 — Components & Props

Components are functions. Props are their typed arguments. Composition replaces inheritance.

Components Are Functions

A React component is a regular TypeScript function that returns JSX. That's it. No magic, no special base class.

// Simplest possible component
const Greeting = () => {
  return <h1>Hello, world!</h1>
}

// Use it like an HTML tag
<Greeting />
Component rules
RuleExampleWhy
Name must be uppercase UserCard, not userCard Lowercase = HTML element (<div>). Uppercase = React component.
Must return JSX or null return <div>...</div> Returning null renders nothing (useful for conditional rendering)
Pure during render No side effects in the return path Given same props/state, same output. Side effects go in hooks.
One component per file UserCard.tsx exports UserCard Convention for organization (small helpers can coexist)
Python analogy
# A function that produces output
def greeting() -> str:
    return "<h1>Hello, world!</h1>"

# Calling it
html = greeting()
React
// A function that produces UI
const Greeting = () => {
  return <h1>Hello, world!</h1>
}

// "Calling" it (React does this for you)
<Greeting />

Props = Function Arguments

Props (short for "properties") are how you pass data from a parent component to a child. They're the component's function arguments, typed with a TypeScript interface.

Python
from dataclasses import dataclass

@dataclass
class UserCardProps:
    name: str
    age: int
    email: str | None = None

def user_card(props: UserCardProps) -> str:
    return f"<div>{props.name}</div>"
TypeScript / React
interface UserCardProps {
  name: string
  age: number
  email?: string  // optional
}

const UserCard = ({ name, age, email }: UserCardProps) => {
  return <div>{name}</div>
}

Passing props

// Parent passes data down via attributes
const App = () => {
  return (
    <div>
      <UserCard name="Alice" age={30} email="alice@example.com" />
      <UserCard name="Bob" age={25} />   {/* email is optional, omitted */}
    </div>
  )
}
Props type patterns
interface ButtonProps {
  label: string                          // required string
  size?: "sm" | "md" | "lg"               // optional union type
  disabled?: boolean                     // optional boolean
  onClick: () => void                    // required callback function
  icon?: React.ReactNode                 // optional JSX element
  style?: React.CSSProperties             // optional CSS object
  items: string[]                         // required array
  config: { theme: string; debug: boolean }  // required object
}

Default prop values

interface AlertProps {
  message: string
  severity?: "info" | "warn" | "error"
}

// Use destructuring defaults — same as Python default args
const Alert = ({ message, severity = "info" }: AlertProps) => {
  return <div className={`alert-${severity}`}>{message}</div>
}

// Usage:
<Alert message="Saved" />               {/* severity defaults to "info" */}
<Alert message="Uh oh" severity="error" />

The Children Prop

The children prop represents whatever you put between a component's opening and closing tags. It's how you build wrapper/layout components.

Python analogy
# Like a decorator/wrapper function
def with_border(content: str) -> str:
    return f"""
    <div class="border">
      {content}
    </div>
    """

# Content passed as argument
with_border("<p>Hello</p>")
React children
// Wrapper component
interface CardProps {
  children: React.ReactNode
}

const Card = ({ children }: CardProps) => (
  <div className="card">
    {children}
  </div>
)

// Content passed between tags
<Card>
  <p>Hello</p>
</Card>
React.ReactNode — what can children be?

React.ReactNode is the type for anything renderable:

  • JSX elements: <p>text</p>
  • Strings and numbers: "hello", 42
  • Arrays of elements: [<p>a</p>, <p>b</p>]
  • Fragments: <>...</>
  • null, undefined, false (render nothing)

Combining children with other props

interface PanelProps {
  title: string
  variant?: "default" | "highlighted"
  children: React.ReactNode
}

const Panel = ({ title, variant = "default", children }: PanelProps) => (
  <section className={`panel panel-${variant}`}>
    <h2>{title}</h2>
    <div className="panel-body">
      {children}
    </div>
  </section>
)

// Usage: title is a prop, everything between tags is children
<Panel title="User Details" variant="highlighted">
  <p>Name: Alice</p>
  <p>Email: alice@example.com</p>
</Panel>

Composition Over Inheritance

React has no inheritance. You never extend a base component. Instead, you compose components together — small pieces combined into bigger pieces.

Inheritance (NOT how React works)
// DON'T do this. This is not React.
class Button extends BaseComponent { }
class DangerButton extends Button { }
class IconDangerButton extends DangerButton { }

// Deep inheritance chains = fragile,
// hard to change, tightly coupled.
Composition (the React way)
// Compose small components into bigger ones
const Button = ({ children, ...props }: Props) =>
  <button {...props}>{children}</button>

const DangerButton = ({ children, ...props }: Props) =>
  <Button className="danger" {...props}>
    {children}
  </Button>

const IconButton = ({ icon, children }: Props) =>
  <Button>{icon} {children}</Button>
Python analogy: This is like preferring composition over inheritance in Python. Instead of deep class hierarchies, you inject dependencies and compose behaviors. A DangerButton is just a Button with specific props, not a subclass.

One-Way Data Flow

Props are read-only. A child component cannot modify its props. To communicate back to the parent, the parent passes a callback function as a prop.

1
Parent
Owns state and passes data + callbacks down
->
2
Props (down)
name="Alice", onDelete={fn}
->
3
Child
Renders data, calls callbacks on user action
// Parent owns the state, passes data and callbacks down
const TodoApp = () => {
  const [todos, setTodos] = useState([
    { id: 1, text: "Learn React" },
    { id: 2, text: "Build something" },
  ])

  const handleDelete = (id: number) => {
    setTodos(prev => prev.filter(t => t.id !== id))
  }

  return (
    <ul>
      {todos.map(todo => (
        <TodoItem
          key={todo.id}
          text={todo.text}
          onDelete={() => handleDelete(todo.id)}
        />
      ))}
    </ul>
  )
}

// Child receives data and callback via props
interface TodoItemProps {
  text: string
  onDelete: () => void
}

const TodoItem = ({ text, onDelete }: TodoItemProps) => (
  <li>
    {text}
    <button onClick={onDelete}>Delete</button>
  </li>
)
Never mutate props: props.name = "Bob" is wrong and will cause bugs. Props are immutable from the child's perspective. If a child needs to change something, it calls a callback that lets the parent update its own state.

Component Organization

File structure conventions
src/
  components/
    UserCard.tsx        # One component per file
    UserList.tsx        # Named export: export { UserList }
    TodoItem.tsx
    TodoApp.tsx
    Button.tsx          # Reusable UI components
    Panel.tsx
  App.tsx               # Root component
  main.tsx              # Entry point
Conventions
  • File name = component name — UserCard.tsx contains UserCard
  • Named exports — export { UserCard }, not export default UserCard. Named exports are easier to refactor and search for.
  • Props interface defined in the same file — interface UserCardProps lives next to UserCard
  • Co-locate related files — put UserCard.tsx, UserCard.css, UserCard.test.tsx together

Realistic Example: UserList with UserCard

Let's build a complete, typed example with multiple components working together.

Types (types.ts)

// Shared type definitions
interface User {
  id: number
  name: string
  email: string
  role: "admin" | "user" | "viewer"
  active: boolean
}

export type { User }

UserCard.tsx

import type { User } from './types'

interface UserCardProps {
  user: User
  onDelete: (id: number) => void
}

const UserCard = ({ user, onDelete }: UserCardProps) => {
  const roleColors: Record<User["role"], string> = {
    admin: "#e94560",
    user: "#82aaff",
    viewer: "#888",
  }

  return (
    <div className="user-card">
      <div className="user-card-header">
        <h3>{user.name}</h3>
        <span style={{ color: roleColors[user.role] }}>
          {user.role}
        </span>
      </div>
      <p>{user.email}</p>
      {!user.active && <p className="inactive">Inactive</p>}
      <button onClick={() => onDelete(user.id)}>
        Remove
      </button>
    </div>
  )
}

export { UserCard }

UserList.tsx

import type { User } from './types'
import { UserCard } from './UserCard'

interface UserListProps {
  users: User[]
  onDeleteUser: (id: number) => void
}

const UserList = ({ users, onDeleteUser }: UserListProps) => {
  if (users.length === 0) {
    return <p>No users found.</p>
  }

  return (
    <div className="user-list">
      <h2>Users ({users.length})</h2>
      {users.map(user => (
        <UserCard
          key={user.id}
          user={user}
          onDelete={onDeleteUser}
        />
      ))}
    </div>
  )
}

export { UserList }

App.tsx (putting it together)

import { useState } from 'react'
import type { User } from './types'
import { UserList } from './UserList'

const App = () => {
  const [users, setUsers] = useState<User[]>([
    { id: 1, name: "Alice", email: "alice@co.com", role: "admin", active: true },
    { id: 2, name: "Bob", email: "bob@co.com", role: "user", active: true },
    { id: 3, name: "Charlie", email: "charlie@co.com", role: "viewer", active: false },
  ])

  const handleDeleteUser = (id: number) => {
    setUsers(prev => prev.filter(user => user.id !== id))
  }

  return (
    <main>
      <h1>My App</h1>
      <UserList users={users} onDeleteUser={handleDeleteUser} />
    </main>
  )
}

export { App }
Data flow in this example
1
App
Owns users state and handleDeleteUser
->
2
UserList
Receives users[] and onDeleteUser, maps over users
->
3
UserCard
Receives single user and onDelete, renders card + button

When the "Remove" button is clicked: UserCard calls onDelete(id) -> UserList passes it up -> App's handleDeleteUser updates state -> React re-renders the tree with the new user list.
