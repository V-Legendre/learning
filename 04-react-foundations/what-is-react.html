04 — What Is React

A JavaScript library for building user interfaces. The core mental model: UI = f(state).

UI = f(state)

The single most important idea in React: your UI is a pure function of your data. Given the same state, you always get the same output. No hidden mutations, no manual DOM poking.

The mental model
// This is basically all of React:
const UI = (state: AppState) => <SomeHTML />

// State changes? React calls your function again.
// New output differs from old output? React updates the DOM.
Python analogy: Jinja2
# Server-side: data -> HTML string
# Sent to browser once, done.
render_template("users.html",
  users=user_list
)

Data flows into a template. Output is HTML. But it's a one-shot render on the server.

React
// Client-side: data -> UI, continuously
// State changes -> re-render -> DOM update
const UserList = () => {
  const [users, setUsers] = useState([])
  return <ul>{users.map(u =>
    <li>{u.name}</li>
  )}</ul>
}

Same idea, but runs in the browser and reacts to state changes automatically.

Key insight: You never write code like "find the third <li> and change its text." You describe the final state of the UI. React figures out what to change in the DOM.

Declarative vs Imperative

Declarative = describe what you want. Imperative = describe how to do it step by step.

Imperative (jQuery-style)
// Step-by-step DOM manipulation
const list = document.getElementById("myList")
const li = document.createElement("li")
li.textContent = "New item"
li.className = "active"
list.appendChild(li)

// You manage every mutation yourself.
// Gets messy fast with complex UIs.
Declarative (React)
// Describe the desired output
const MyList = ({ items }: Props) => (
  <ul>
    {items.map(item =>
      <li className="active">
        {item.text}
      </li>
    )}
  </ul>
)

// React handles the DOM changes.
Python analogy
StyleExample
Declarative SELECT name FROM users WHERE active = true — you say what you want, the DB engine decides how
Imperative Loop through a file line by line, parse each field, filter manually, collect results — you control every step

React is SQL for UIs. You describe the result, not the procedure.

Virtual DOM & Reconciliation

React doesn't modify the real DOM directly. It maintains a lightweight in-memory copy (the Virtual DOM), diffs it against the previous version, and applies only the minimal changes to the real DOM.

1
State Changes
User clicks a button, data arrives from API, timer fires
->
2
Re-render
React calls your component function again with new state
->
3
New Virtual DOM
Your function returns a new tree of elements (plain JS objects)
->
4
Diff (Reconciliation)
React compares new tree vs old tree, finds what changed
->
5
Patch Real DOM
Only the changed elements get updated. Minimal, fast mutations.
Why not just update the real DOM directly?
  • DOM operations are slow — each mutation triggers layout recalculation, style recomputation, and repainting. Browsers are optimized for reading the DOM, not writing to it thousands of times.
  • Batching — React batches multiple state changes into a single DOM update. Change 10 things? One DOM write.
  • Predictability — You always describe the complete UI. No partial updates that leave the DOM in an inconsistent state.
Python analogy: Think of it like SQLAlchemy's unit of work pattern. You make changes to your ORM objects in memory, then session.commit() flushes only the diffs to the database in a single transaction. React's reconciliation is the commit() step.

Components = Functions

In React, every piece of UI is a function that returns JSX (HTML-like syntax). No classes needed.

Python
# A function that returns formatted data
def user_card(name: str, role: str) -> str:
    return f"""
    <div class="card">
      <h2>{name}</h2>
      <p>{role}</p>
    </div>
    """
React component
// A function that returns UI
const UserCard = ({ name, role }: Props) => {
  return (
    <div className="card">
      <h2>{name}</h2>
      <p>{role}</p>
    </div>
  )
}
Key rules for components
  • Name must start with uppercase — UserCard not userCard. Lowercase = HTML element.
  • Must return JSX (or null to render nothing).
  • Pure by convention — given the same props and state, always return the same output. No side effects during rendering.
  • Composable — components can use other components. <App> renders <Header> which renders <NavLink>.

One-Way Data Flow

Data flows down (parent to child via props). Events flow up (child to parent via callbacks). Never sideways, never backwards.

1
Parent owns state
const [users, setUsers] = useState([])
->
2
Passes data down
<UserList users={users} />
->
3
Child renders it
Child uses props, never modifies them
3
Child event fires
User clicks "Delete" button
->
2
Calls parent callback
onDelete(userId) — function passed as prop
->
1
Parent updates state
setUsers(prev => prev.filter(...))
Python analogy
# Functions take args (data down)
# and return values (results up)
def process(data: list) -> Result:
    filtered = filter_items(data)
    return format_output(filtered)

# You'd never have filter_items()
# reach back and modify process()'s
# local variables. Same idea in React.
React equivalent
// Props = arguments (data down)
// Callbacks = return channel (events up)
const Parent = () => {
  const [items, setItems] = useState([])
  const handleDelete = (id: number) => {
    setItems(prev => prev.filter(
      i => i.id !== id
    ))
  }
  return <List
    items={items}
    onDelete={handleDelete}
  />
}
Why one-way? Two-way data binding (like Angular v1) sounds convenient but creates feedback loops and makes it impossible to trace where a state change came from. One-way flow is predictable and debuggable, like a well-structured Python call stack.

When NOT to Use React

React is a tool, not a religion. It adds complexity. Use it when you need it.

ScenarioUse React?Better alternative
Blog, documentation, marketing page No Static HTML, Astro, Hugo
Simple form + server submission Probably not Plain HTML + a sprinkle of JS
Dashboard with live updates, filters, interactive charts Yes -
Multi-step wizard, complex form logic Yes -
SPA (single-page application) with client-side routing Yes -
Email templates No MJML, plain HTML
Rule of thumb: If your page has lots of interactive state that changes over time, React shines. If it's mostly static content, you're adding overhead for nothing.
