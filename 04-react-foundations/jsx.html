04 — JSX

The syntax React uses to describe UI. Looks like HTML, but it's JavaScript expressions that compile to function calls.

JSX Is NOT HTML

JSX stands for JavaScript XML. It looks like HTML embedded in your TypeScript, but the compiler transforms it into plain function calls.

What the compiler does
What you write (JSX)
const element = (
  <h1 className="title">
    Hello, world!
  </h1>
)
What the compiler produces
const element = _jsx("h1", {
  className: "title",
  children: "Hello, world!"
})

JSX is syntactic sugar. It gets compiled away before it reaches the browser. The .tsx extension tells TypeScript to enable JSX parsing.

Python analogy: Think of JSX like f-strings. f"Hello {name}" looks like a plain string but actually compiles to string concatenation. JSX looks like HTML but compiles to function calls.

Style gotcha: The double curly braces in style={{ color: 'red' }} are not special syntax. The outer {} is a JSX expression. The inner {} is a JS object literal. CSS properties use camelCase: font-size becomes fontSize, background-color becomes backgroundColor.

Expressions in {}

Inside JSX, curly braces {} let you embed any JavaScript expression. An expression is anything that produces a value.

What you CAN put in {}
const name = "Alice"
const count = 42
const items = ["a", "b", "c"]

return (
  <div>
    <p>{name}</p>                          {/* variable */}
    <p>{count * 2}</p>                     {/* math */}
    <p>{name.toUpperCase()}</p>             {/* method call */}
    <p>{items.length} items</p>             {/* property access */}
    <p>{count > 10 ? "many" : "few"}</p>  {/* ternary */}
    <p>{formatDate(new Date())}</p>       {/* function call */}
  </div>
)
What you CANNOT put in {}
// These are STATEMENTS, not expressions. They don't produce a value.

<p>{if (x) "yes"}</p>           {/* ERROR: if is a statement */}
<p>{for (let i...) ...}</p>      {/* ERROR: for is a statement */}
<p>{const x = 5}</p>              {/* ERROR: declarations are statements */}
Python Jinja2
{# Jinja has its own syntax #}
<p>{{ name }}</p>
<p>{{ name | upper }}</p>
{% if count > 10 %}
  <p>many</p>
{% endif %}
{% for item in items %}
  <li>{{ item }}</li>
{% endfor %}

Jinja has special template syntax for control flow ({% %}).

JSX
{/* JSX uses plain JS expressions */}
<p>{name}</p>
<p>{name.toUpperCase()}</p>
{count > 10 && <p>many</p>}

{items.map(item =>
  <li>{item}</li>
)}

No special template language. Just JavaScript with {}.

Conditional Rendering

Since you can't use if statements inside JSX, there are three patterns for conditional rendering.

Pattern 1: Logical AND (&&)

Render something or nothing.

// If isAdmin is true, render the button. Otherwise render nothing.
<div>
  <h1>Dashboard</h1>
  {isAdmin && <button>Delete All</button>}
</div>
Gotcha with &&: If the left side is 0 (not false), React will render the number 0. This is a common bug: {count && <p>Items</p>} renders "0" when count is 0. Fix: use {count > 0 && <p>Items</p>} to ensure a boolean left side.

Pattern 2: Ternary

Render one thing or another.

// Show different content based on a condition
<div>
  {isLoggedIn
    ? <p>Welcome back, {name}!</p>
    : <p>Please log in.</p>
  }
</div>

Pattern 3: Early return

For bigger conditional blocks, use if before the return statement.

const UserProfile = ({ user }: Props) => {
  if (!user) {
    return <p>Loading...</p>
  }

  return (
    <div>
      <h1>{user.name}</h1>
      <p>{user.email}</p>
    </div>
  )
}
When to use which: && for show/hide. Ternary for toggle between two things. Early return for guard clauses (loading, error, empty states).

Rendering Lists

Use .map() to transform an array of data into an array of JSX elements. Every item needs a unique key prop.

Python Jinja2
<ul>
{% for user in users %}
  <li>{{ user.name }}</li>
{% endfor %}
</ul>
JSX
<ul>
  {users.map(user => (
    <li key={user.id}>
      {user.name}
    </li>
  ))}
</ul>
The key prop

React uses key to identify which items changed, were added, or removed during reconciliation. Without keys, React re-renders the entire list on every change.

Good keysBad keys
Database IDs: key={user.id}Array index: key={index} (breaks on reorder/delete)
Unique slugs: key={post.slug}Random values: key={Math.random()} (new key every render)
Composite: key={`${type}-${id}`}Non-unique values: key={user.name} (names can repeat)
Why index as key is bad: If you delete item at index 2, items at index 3, 4, 5 all shift down. React thinks index 2 changed content (it now has index 3's data), index 3 changed, etc. Instead of removing one DOM node, it updates all of them. With stable IDs, React knows exactly which node to remove.

Full list example with types

interface User {
  id: number
  name: string
  email: string
  active: boolean
}

const UserList = ({ users }: { users: User[] }) => {
  if (users.length === 0) {
    return <p>No users found.</p>
  }

  return (
    <ul>
      {users
        .filter(u => u.active)
        .map(user => (
          <li key={user.id}>
            <strong>{user.name}</strong> — {user.email}
          </li>
        ))}
    </ul>
  )
}

Fragments

A component must return a single element. When you need to return multiple elements without adding a wrapper <div>, use a Fragment.

Unnecessary wrapper div
// Adds an extra <div> to the DOM
const UserInfo = () => (
  <div>   {/* useless wrapper */}
    <h2>Alice</h2>
    <p>alice@example.com</p>
  </div>
)
Fragment (no extra DOM node)
// No extra element in the DOM
const UserInfo = () => (
  <>      {/* Fragment shorthand */}
    <h2>Alice</h2>
    <p>alice@example.com</p>
  </>
)
When to use which: Use <>...</> (shorthand) for most cases. Use <Fragment key={id}> when you need a key (e.g., inside .map()). Import with import { Fragment } from 'react'.

TypeScript + JSX

Use the .tsx file extension for any file that contains JSX. TypeScript will type-check your JSX expressions and attributes.

What TypeScript checks in JSX
  • Attribute types — onClick must be a function, className must be a string
  • Required attributes — <img> requires src and alt
  • Expression types — {count.toUpperCase()} errors if count is a number
  • Component props — if UserCard requires name: string, you can't skip it
// TypeScript catches these at compile time:

<button onClick="handleClick">   {/* ERROR: string is not a function */}
<button onClick={handleClick}>  {/* OK */}

<img src="photo.jpg" />           {/* ERROR: missing required alt attribute */}
<img src="photo.jpg" alt="Photo" /> {/* OK */}

<div class="box">              {/* ERROR: use className, not class */}
<div className="box">          {/* OK */}
Practical benefit: TypeScript + JSX catches HTML attribute typos at compile time. No more deploying a typo and finding it in production. Think of it like a linter that actually understands your UI structure.

Comments in JSX

HTML comments (<!-- -->) don't work in JSX. Use JS comments inside curly braces:

return (
  <div>
    {/* This is a JSX comment */}
    <p>Hello</p>
    {/*
      Multi-line comments
      work like this
    */}
  </div>
)
