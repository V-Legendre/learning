12 — Async Error Handling

Handling errors in API calls, mutations, event handlers, and network failures.

TanStack Query Error Handling

TanStack Query gives you per-query error state out of the box. Every useQuery call returns isError and error alongside the data.

Python Analogy

Like structured exception handling at different levels:

# Per-endpoint error handling
@app.get("/users")
def get_users():
    try:
        return db.query("SELECT * FROM users")
    except DatabaseError as e:
        raise HTTPException(500, str(e))

# Global error handler (middleware)
@app.exception_handler(Exception)
def handle_error(req, exc):
    logger.error(exc)
    return JSONResponse(500, ...)
React equivalent
// Per-query error handling
const { data, isError, error } = useQuery({
  queryKey: ['users'],
  queryFn: fetchUsers,
})

if (isError) {
  return <p>Error: {error.message}</p>
}

// Global error handler
const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      onError: (err) => toast(err.message),
    },
  },
})

Per-Query Error Display

const UserList = () => {
  const { data, isLoading, isError, error, refetch } = useQuery({
    queryKey: ['users'],
    queryFn: fetchUsers,
    retry: 3,              // retry failed requests 3 times
    retryDelay: (attempt) => // exponential backoff
      Math.min(1000 * 2 ** attempt, 30000),
  })

  if (isLoading) return <Spinner />

  if (isError) {
    return (
      <div role="alert">
        <p>Failed to load users: {error.message}</p>
        <button onClick={() => refetch()}>
          Try again
        </button>
      </div>
    )
  }

  return (
    <ul>
      {data.map((user) => (
        <li key={user.id}>{user.name}</li>
      ))}
    </ul>
  )
}

Retry Configuration

Option Default Description
retry 3 Number of retries. Set false to disable. Can be a function: (count, error) => boolean
retryDelay Exponential Delay between retries. Default doubles each time, capped at 30s.
retryOnMount true Retry failed queries when component mounts
Smart retry: Don't retry 4xx errors (client mistakes). Only retry 5xx (server errors) and network failures. 
const { data } = useQuery({
  queryKey: ['users'],
  queryFn: fetchUsers,
  // Only retry on server errors, not client errors
  retry: (failureCount, error) => {
    if (error instanceof Response && error.status < 500) {
      return false // 4xx: don't retry
    }
    return failureCount < 3 // 5xx / network: retry up to 3 times
  },
})

Global Error Handler

Set up a global error handler on the QueryClient to show a toast notification whenever any query fails. This is your safety net — individual components can still handle errors locally if needed.

import { toast } from 'sonner'  // or react-hot-toast
import { QueryClient, QueryCache, MutationCache } from '@tanstack/react-query'

const queryClient = new QueryClient({
  queryCache: new QueryCache({
    onError: (error, query) => {
      // Only show toast if no component-level error handling
      if (query.meta?.showGlobalError !== false) {
        toast.error(`Failed to load data: ${error.message}`)
      }
    },
  }),
  mutationCache: new MutationCache({
    onError: (error) => {
      toast.error(`Operation failed: ${error.message}`)
    },
  }),
})
Note: The meta field on queries lets you pass arbitrary data. Here we use it to let individual queries opt out of the global error toast when they handle errors themselves. 
// Opt out of global error toast for this query
const { data, isError, error } = useQuery({
  queryKey: ['users'],
  queryFn: fetchUsers,
  meta: { showGlobalError: false },  // handle errors locally
})

// This component renders its own error UI
if (isError) return <InlineError error={error} />

Mutation Error Handling

Mutations (POST, PUT, DELETE) need special attention because they modify data. Show errors near the action that caused them.

const CreateUserForm = () => {
  const mutation = useMutation({
    mutationFn: (data: CreateUserInput) =>
      fetch('/api/users', {
        method: 'POST',
        body: JSON.stringify(data),
      }).then((r) => {
        if (!r.ok) throw new Error(`HTTP ${r.status}`)
        return r.json()
      }),
    onSuccess: () => {
      toast.success('User created!')
      queryClient.invalidateQueries({ queryKey: ['users'] })
    },
    // onError from global mutationCache handles the toast
  })

  const handleSubmit = (e: React.FormEvent) => {
    e.preventDefault()
    mutation.mutate({ name, email })
  }

  return (
    <form onSubmit={handleSubmit}>
      {/* form fields */}

      {/* Show error next to submit button */}
      {mutation.isError && (
        <p style={{ color: 'red' }}>
          {mutation.error.message}
        </p>
      )}

      <button
        type="submit"
        disabled={mutation.isPending}
      >
        {mutation.isPending ? 'Creating...' : 'Create User'}
      </button>
    </form>
  )
}
Note: mutation.reset() clears the error state. Call it when the user starts editing again, or when you want to dismiss the error message.

try/catch in Event Handlers

Error boundaries don't catch errors in event handlers (onClick, onChange, onSubmit). Use try/catch with local error state.

Pattern: event handler with error state
const FileUploader = () => {
  const [error, setError] = useState<string | null>(null)
  const [uploading, setUploading] = useState(false)

  const handleUpload = async (e: React.ChangeEvent<HTMLInputElement>) => {
    const file = e.target.files?.[0]
    if (!file) return

    setError(null)  // Clear previous error
    setUploading(true)

    try {
      // Validate
      if (file.size > 5_000_000) {
        throw new Error('File must be under 5MB')
      }

      // Upload
      const formData = new FormData()
      formData.append('file', file)

      const res = await fetch('/api/upload', {
        method: 'POST',
        body: formData,
      })

      if (!res.ok) throw new Error(`Upload failed: ${res.statusText}`)

      toast.success('File uploaded!')
    } catch (err) {
      const message = err instanceof Error
        ? err.message
        : 'Something went wrong'
      setError(message)
    } finally {
      setUploading(false)
    }
  }

  return (
    <div>
      <input type="file" onChange={handleUpload} disabled={uploading} />
      {uploading && <p>Uploading...</p>}
      {error && <p style={{ color: 'red' }}>{error}</p>}
    </div>
  )
}

Differentiating Error Types

Different HTTP status codes need different UI treatment. A 404 is not the same as a 500.

Status Category UI Treatment Retry?
400 Bad request Show validation errors No (user must fix input)
401 Unauthorized Redirect to login No (re-auth needed)
403 Forbidden "You don't have access" No (permissions issue)
404 Not found "Item not found" page No (resource doesn't exist)
429 Rate limited "Too many requests, slow down" Yes (after delay)
500 Server error "Something went wrong, try again" Yes (automatic)
Network No connection "Check your internet connection" Yes (when online)
Custom error class for API errors
// lib/api.ts — typed API error
export class ApiError extends Error {
  constructor(
    public status: number,
    message: string,
    public code?: string,
  ) {
    super(message)
    this.name = 'ApiError'
  }

  get isClientError() { return this.status >= 400 && this.status < 500 }
  get isServerError() { return this.status >= 500 }
  get isNotFound()    { return this.status === 404 }
  get isUnauthorized() { return this.status === 401 }
}

// Wrapper for fetch
export const apiFetch = async <T>(url: string, init?: RequestInit): Promise<T> => {
  const res = await fetch(url, init)

  if (!res.ok) {
    const body = await res.json().catch(() => null)
    throw new ApiError(
      res.status,
      body?.message ?? res.statusText,
      body?.code,
    )
  }

  return res.json()
}

// Smart retry — only for server errors
const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      retry: (count, error) => {
        if (error instanceof ApiError && error.isClientError) {
          return false // never retry 4xx
        }
        return count < 3
      },
    },
  },
})

Network Error Patterns

Offline Detection

// Hook to detect online/offline status
const useOnlineStatus = () => {
  const [isOnline, setIsOnline] = useState(navigator.onLine)

  useEffect(() => {
    const handleOnline = () => setIsOnline(true)
    const handleOffline = () => setIsOnline(false)

    window.addEventListener('online', handleOnline)
    window.addEventListener('offline', handleOffline)

    return () => {
      window.removeEventListener('online', handleOnline)
      window.removeEventListener('offline', handleOffline)
    }
  }, [])

  return isOnline
}

// Usage: show a banner
const OfflineBanner = () => {
  const isOnline = useOnlineStatus()

  if (isOnline) return null
  return (
    <div style={{ background: '#f59e0b', padding: '8px', textAlign: 'center' }}>
      You're offline. Changes will sync when you reconnect.
    </div>
  )
}
Tip: TanStack Query automatically pauses queries when offline and retries when the connection comes back. You get this for free with the onlineManager.

Error Logging Strategy

Development
  • Log everything to console
  • Show detailed error messages
  • Show component stack traces
  • Use React DevTools error overlay
Production
  • Send errors to monitoring service (Sentry, LogRocket)
  • Show user-friendly messages only
  • Never expose raw error messages, stack traces, or API details
  • Include context: user ID, URL, action attempted
Security rule: Never show raw error messages to users in production. A message like ECONNREFUSED postgres://admin:pass@db:5432 leaks your infrastructure. Always map errors to user-friendly messages. 
// Map API errors to user-friendly messages
const getUserMessage = (error: unknown): string => {
  if (error instanceof ApiError) {
    if (error.isNotFound) return 'The item you requested was not found.'
    if (error.isUnauthorized) return 'Please log in to continue.'
    if (error.isServerError) return 'Something went wrong. Please try again later.'
    return error.message // 4xx with specific message from server
  }

  if (error instanceof TypeError && error.message === 'Failed to fetch') {
    return 'Network error. Check your internet connection.'
  }

  return 'An unexpected error occurred.'
}

Error handling checklist
  • Error boundaries around independent sections (not just the root)
  • TanStack Query for all API error states (loading, error, data)
  • Global query/mutation error handler with toast notifications
  • try/catch in event handlers with local error state
  • Custom ApiError class to differentiate 4xx vs 5xx
  • Smart retry: only retry server errors and network failures
  • User-friendly error messages in production, detailed in development
  • Error monitoring service (Sentry) for production
