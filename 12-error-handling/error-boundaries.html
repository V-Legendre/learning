12 — Error Boundaries

Catching render errors gracefully instead of crashing the entire app.

The Problem

In React, an unhandled error during rendering crashes the entire component tree. The user sees a white screen with nothing. No error message, no way to recover, no navigation.

What happens without error boundaries
// One buggy component...
const UserProfile = ({ user }: { user: User }) => {
  // Oops: user.address might be null
  return <p>{user.address.street}</p>
  //          ^^^ TypeError: Cannot read property 'street' of null
}

// ...crashes EVERYTHING
const App = () => (
  <div>
    <Navbar />        {/* gone */}
    <Sidebar />       {/* gone */}
    <UserProfile />   {/* the offender */}
    <Footer />        {/* gone */}
  </div>
)
// Result: white screen of death
Python Analogy

Imagine if an exception in one endpoint handler killed your entire Flask/FastAPI server:

# If Flask worked like React without
# error boundaries:

@app.get("/profile")
def profile():
    raise ValueError("oops")
    # Entire server goes down
    # All routes stop working

Error boundaries are like try/except but for the React render tree. They isolate failures.

With error boundaries
{/* Navbar and Sidebar keep working */}
<Navbar />
<Sidebar />

{/* Only this section shows the error */}
<ErrorBoundary fallback={<p>Error</p>}>
  <UserProfile />
</ErrorBoundary>

<Footer />

The error is contained. The user can still navigate, use the sidebar, etc.

react-error-boundary Library

React's built-in error boundary API requires class components (the only remaining use case for them). The react-error-boundary library wraps this in a clean functional API.

$ pnpm add react-error-boundary

Basic Usage

import { ErrorBoundary } from 'react-error-boundary'

const App = () => (
  <ErrorBoundary
    fallbackRender={({ error, resetErrorBoundary }) => (
      <div role="alert">
        <h2>Something went wrong</h2>
        <p style={{ color: 'red' }}>
          {error.message}
        </p>
        <button onClick={resetErrorBoundary}>
          Try again
        </button>
      </div>
    )}
  >
    <MyComponent />
  </ErrorBoundary>
)

API Options

Prop Type Purpose
fallbackRender ({ error, resetErrorBoundary }) => ReactNode Render function for error UI (most flexible)
fallback ReactNode Static fallback JSX (simplest)
FallbackComponent ComponentType Component that receives error and resetErrorBoundary as props
onError (error, info) => void Called when error is caught — use for logging
onReset () => void Called when resetErrorBoundary is invoked — clear state
resetKeys any[] Auto-reset when these values change (like a dependency array)

Granular Boundaries

Don't wrap the entire app in one boundary. Place boundaries around independent sections so a failure in one area doesn't destroy the rest.

One boundary (everything fails together)
<ErrorBoundary>
  <App>
    <Navbar />
    <Sidebar />
    <MainContent />
    <ChatWidget />
  </App>
</ErrorBoundary>
{/* Any error = entire app replaced
    with error fallback */}
Granular boundaries
<ErrorBoundary fallback={<FullPageError />}>
  <Navbar />
  <div className="layout">
    <ErrorBoundary fallback={<SidebarError />}>
      <Sidebar />
    </ErrorBoundary>
    <ErrorBoundary fallback={<ContentError />}>
      <MainContent />
    </ErrorBoundary>
  </div>
  <ErrorBoundary fallback={null}>
    <ChatWidget />
  </ErrorBoundary>
</ErrorBoundary>
Strategy: Think of error boundaries like try/except blocks in Python. Catch at the right level:
  • Outermost: Full-page error with "reload" button (safety net)
  • Section level: Replace broken section with error message
  • Card/widget level: Replace individual card, others still work
  • Non-critical: Silently hide broken widget (fallback={null})

Reset Patterns

After an error, you often want to let the user try again. The resetErrorBoundary function re-mounts the children.

Manual Reset (button click)

<ErrorBoundary
  fallbackRender={({ error, resetErrorBoundary }) => (
    <div>
      <p>Failed to load: {error.message}</p>
      <button onClick={resetErrorBoundary}>
        Retry
      </button>
    </div>
  )}
  onReset={() => {
    // Clear any state that might have caused the error
    queryClient.clear()
  }}
>
  <DataComponent />
</ErrorBoundary>

Auto-Reset (on prop/state change)

const UserPage = () => {
  const { userId } = useParams()

  return (
    <ErrorBoundary
      resetKeys={[userId]}
      {/* Auto-resets when userId changes */}
      {/* e.g. navigating to a different user */}
      fallbackRender={({ error }) => (
        <p>Couldn't load user: {error.message}</p>
      )}
    >
      <UserProfile userId={userId} />
    </ErrorBoundary>
  )
}

What Error Boundaries DON'T Catch

Error boundaries only catch errors during rendering (in the component function body and lifecycle). They do NOT catch:

Scenario Caught? Solution
Error during render (JSX / component body) Yes Error boundary handles it
Error in useEffect Yes Error boundary handles it
Error in event handlers (onClick, etc.) No try/catch + local error state
Async code (setTimeout, Promises) No try/catch in async functions
Server-side rendering No Server-side error handling
Errors in the error boundary itself No Wrap in another boundary (parent)
Handling event handler errors (try/catch)
const DeleteButton = ({ id }: { id: string }) => {
  const [error, setError] = useState<string | null>(null)

  const handleDelete = async () => {
    try {
      setError(null)
      await fetch(`/api/items/${id}`, { method: 'DELETE' })
    } catch (err) {
      setError(err instanceof Error ? err.message : 'Delete failed')
    }
  }

  return (
    <div>
      <button onClick={handleDelete}>Delete</button>
      {error && <p style={{ color: 'red' }}>{error}</p>}
    </div>
  )
}

Reusable Error Fallback Components

Create reusable fallback components for different error contexts.

Three levels of error fallbacks
// Full page error — outermost boundary
const FullPageError = ({
  error,
  resetErrorBoundary,
}: {
  error: Error
  resetErrorBoundary: () => void
}) => (
  <div style={{ textAlign: 'center', padding: '60px' }}>
    <h1>Something went wrong</h1>
    <p>{error.message}</p>
    <button onClick={() => window.location.reload()}>
      Reload page
    </button>
  </div>
)

// Section error — around a major section
const SectionError = ({
  error,
  resetErrorBoundary,
}: {
  error: Error
  resetErrorBoundary: () => void
}) => (
  <div style={{ padding: '20px', border: '1px solid red' }}>
    <p>This section encountered an error.</p>
    <button onClick={resetErrorBoundary}>Retry</button>
  </div>
)

// Inline error — around a card or widget
const InlineError = ({ error }: { error: Error }) => (
  <span style={{ color: 'red', fontSize: '12px' }}>
    Failed to load
  </span>
)

Using Them Together

const App = () => (
  {/* Safety net — catches anything that escapes */}
  <ErrorBoundary FallbackComponent={FullPageError}>
    <Layout>
      {/* Section boundary — sidebar can fail independently */}
      <ErrorBoundary FallbackComponent={SectionError}>
        <Sidebar />
      </ErrorBoundary>

      {/* Main content boundary */}
      <ErrorBoundary FallbackComponent={SectionError}>
        <main>
          {/* Each card can fail independently */}
          {cards.map((card) => (
            <ErrorBoundary key={card.id}
              FallbackComponent={InlineError}>
              <Card data={card} />
            </ErrorBoundary>
          ))}
        </main>
      </ErrorBoundary>
    </Layout>
  </ErrorBoundary>
)

Error Logging

Use the onError prop to log errors. In development, log to console. In production, send to an error monitoring service.

const logError = (error: Error, info: { componentStack?: string | null }) => {
  // Development: console
  console.error('Error boundary caught:', error)
  console.error('Component stack:', info.componentStack)

  // Production: send to monitoring service
  if (import.meta.env.PROD) {
    // Sentry.captureException(error, { extra: info })
  }
}

<ErrorBoundary
  onError={logError}
  FallbackComponent={SectionError}
>
  <MyComponent />
</ErrorBoundary>

Summary

Error boundary rules
  • Use react-error-boundary (not raw class components)
  • Place boundaries around independent sections, not just the whole app
  • Use fallbackRender for dynamic error UI with reset buttons
  • Use resetKeys to auto-reset on navigation or prop changes
  • Use onError to log to monitoring services
  • Error boundaries don't catch event handlers or async code — use try/catch for those
  • Keep fallback UIs simple — a broken fallback gives you nothing
