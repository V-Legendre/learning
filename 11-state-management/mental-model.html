11 — State Mental Model

Understanding the different types of state and when to use each tool.

Server State vs Client State

The single most important distinction in React state management. These are fundamentally different problems with different solutions. Mixing them up is the #1 source of state management pain.

Server State

Data that lives on a server.

  • Fetched asynchronously (API calls)
  • Can be stale — someone else may have changed it
  • Needs caching, deduplication, background refetching
  • Has loading/error/success states
  • Shared across many components

Tool: TanStack Query

Examples: user profile, product list, search results, dashboard data

Client State

Data that exists only in the browser.

  • Synchronous — always up to date
  • Never stale — you own it completely
  • No caching needed
  • Immediate updates, no loading states
  • Often local to one component or section

Tool: useState, Zustand

Examples: dark mode toggle, sidebar open/closed, form inputs, modal visibility

Python Analogy

Server state is like your database + cache layer. Think PostgreSQL with Redis in front. The data is "the truth" but your local copy might be stale. You need invalidation strategies, TTLs, and retry logic.

Client state is like in-memory app state or session data. A Python dict you keep in your FastAPI/Flask app's memory. You own it, it's always fresh, and it disappears when the process dies.

The Mistake
// DON'T: Storing server data in client state
const [users, setUsers] = useState([])

useEffect(() => {
  fetch('/api/users')
    .then(r => r.json())
    .then(setUsers)
}, [])
// No caching, no loading state, no error
// handling, no refetch on focus, no dedup

Use TanStack Query for server data. It handles all the complexity for you.

The State Decision Tree

When you need state in a React app, walk through this decision tree top to bottom. Pick the simplest tool that fits.

Is it from an API?
Server data, fetched async
→
TanStack Query
Caching, loading, error, refetch
Used by one component?
Toggle, form input, local UI
→
useState
Simple, local, component-scoped
Complex local logic?
Multiple related state transitions
→
useReducer
State machine, action/dispatch
Shared by a few nearby components?
Parent + 1-2 children
→
Lift State Up
Move state to common parent, pass via props
Rarely-changing global data?
Theme, auth, locale
→
React Context
Provider/consumer, avoid for frequent updates
Complex shared client state?
Shopping cart, multi-step form, notifications
→
Zustand
External store, selectors, middleware
Common mistake: Putting everything in a global store. Most state is local. In a typical app, 80%+ of state is local useState and server state handled by TanStack Query. Only a small fraction needs Zustand.

Quick Reference Table

Tool When Scope Examples

URL as State

Search params, filters, pagination, and sorting should live in the URL — not in component state. This makes the state shareable, bookmarkable, and the back button works correctly.

In component state
// State lost on page refresh
// Can't share a link to this view
// Back button doesn't work
const [search, setSearch] = useState('')
const [page, setPage] = useState(1)
const [sort, setSort] = useState('name')
In the URL
// /products?search=shirt&page=2&sort=name
// Shareable, bookmarkable, back button works
const [searchParams, setSearchParams] =
  useSearchParams()

const search = searchParams.get('search') ?? ''
const page = Number(searchParams.get('page') ?? 1)
const sort = searchParams.get('sort') ?? 'name'
Rule of thumb: If a user would want to bookmark or share the current view, that state belongs in the URL. Think search results pages, filter configurations, and paginated lists.
Python Analogy

URL state is like query parameters in a REST API endpoint. In Flask/FastAPI:

# GET /products?search=shirt&page=2
@app.get("/products")
def list_products(
    search: str = "",
    page: int = 1,
    sort: str = "name"
):
    ...

Same idea in the browser: the URL drives what data the page shows.

React Router equivalent
// Read from URL
const [params] = useSearchParams()
const search = params.get('search') ?? ''

// Write to URL
const updateSearch = (value: string) => {
  setSearchParams(prev => {
    prev.set('search', value)
    prev.set('page', '1') // reset page
    return prev
  })
}

Key insight: If you find yourself reaching for Zustand or Context for most of your state, you're probably overcomplicating things. Start with useState and TanStack Query. Only reach for more when you genuinely need it.

Why Context Is Not a State Manager

React Context is often misused as a state manager. It works fine for rarely-changing data (theme, auth, locale), but has a fundamental problem with frequent updates:

The Re-render Problem
// Every component consuming this context re-renders
// when ANY value in it changes
const AppContext = createContext({
  theme: 'dark',
  user: null,
  notifications: [],   // changes frequently!
  sidebarOpen: false,
  cartItems: [],        // changes frequently!
})

// Component only reads theme, but re-renders when
// notifications or cartItems change too
const Header = () => {
  const { theme } = useContext(AppContext)
  // ^^^ re-renders on EVERY context change
}

Zustand solves this with selectors — components only re-render when their selected slice changes.

Summary: The Mental Model

Think of state as three buckets
1
Does this data come from a server?

Use TanStack Query. Don't put API data in useState or Zustand.

2
Is it only used by this component (or nearby children)?

Use useState/useReducer. Lift state up if needed. Keep it local.

3
Is it truly global client-only state?

Use Zustand for complex state, Context for simple rarely-changing values, URL for shareable state.

Start simple. Begin with useState everywhere. Only "promote" state to a higher-level tool when you feel the pain. Premature state management abstraction is just as bad as premature optimization.
