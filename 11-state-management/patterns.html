11 — Advanced State Patterns

Optimistic updates, derived state, URL state, and store composition patterns.

Optimistic Updates

Update the UI immediately, then send the request to the server. If it fails, revert. This makes the app feel instant even on slow connections.

1
User clicks
e.g. "Like" button
→
2
Update UI immediately
Show liked state
→
3
Send API request
POST /api/like
→
4
On error: revert
Restore previous state
Python Analogy

Optimistic updates are like writing to cache first, then persisting to the database. If the DB write fails, you roll back the cache.

# Python: write-through cache pattern
def toggle_like(post_id):
    # 1. Update cache immediately
    cache.set(f"liked:{post_id}", True)
    try:
        # 2. Persist to database
        db.execute("INSERT INTO likes...")
    except Exception:
        # 3. Rollback cache on failure
        cache.delete(f"liked:{post_id}")
TanStack Query pattern
const likeMutation = useMutation({
  mutationFn: (postId: string) =>
    fetch(`/api/posts/${postId}/like`, {
      method: 'POST',
    }),

  // Before request: optimistically update
  onMutate: async (postId) => {
    // Cancel in-flight queries
    await queryClient
      .cancelQueries({ queryKey: ['post', postId] })

    // Save previous state for rollback
    const previous = queryClient
      .getQueryData(['post', postId])

    // Optimistically update
    queryClient.setQueryData(
      ['post', postId],
      (old) => ({ ...old, liked: true })
    )

    return { previous }
  },

  // On error: revert to saved state
  onError: (_err, postId, context) => {
    queryClient.setQueryData(
      ['post', postId],
      context?.previous
    )
  },

  // Always refetch after error or success
  onSettled: (_data, _err, postId) => {
    queryClient.invalidateQueries({
      queryKey: ['post', postId],
    })
  },
})
The three callbacks:
  • onMutate — runs before the request. Save old state, update optimistically. Return context for rollback.
  • onError — runs on failure. Use context to revert.
  • onSettled — runs after success or failure. Invalidate queries to get fresh server data.
When to use optimistic updates: For actions where the success rate is high (>99%) and immediate feedback matters. "Like" buttons, toggling favorites, reordering lists. Don't use for critical actions like payments.

Derived State

Derived state is data computed from other state. The rule: don't store it — compute it. Storing derived values creates synchronization bugs.

Storing derived state (BUG)
// DON'T: total can get out of sync
interface CartState {
  items: CartItem[]
  total: number  // derived! don't store
}

addItem: (item) => set((s) => ({
  items: [...s.items, item],
  // Forgot to update total? Bug!
  total: s.items.reduce(...) + item.price
}))
Computing derived state
// DO: compute on the fly
interface CartState {
  items: CartItem[]
  // No total stored!
}

// Option 1: In the store with get()
getTotal: () =>
  get().items.reduce(
    (sum, i) => sum + i.price * i.qty, 0
  )

// Option 2: In component with useMemo
const items = useCartStore((s) => s.items)
const total = useMemo(
  () => items.reduce(
    (sum, i) => sum + i.price * i.qty, 0
  ),
  [items]
)
Python Analogy

Derived state is like a SQL VIEW or a Python @property:

# Don't store total in the table
# Compute it with a VIEW:
# SELECT SUM(price * qty) FROM cart_items

# Or in Python:
class Cart:
    @property
    def total(self) -> float:
        return sum(
            i.price * i.qty
            for i in self.items
        )
Common derived values
  • Total price = sum of (price * quantity)
  • Filtered list = items matching a filter
  • Item count = array.length
  • Is empty = items.length === 0
  • Selected item = items.find(id === selectedId)
  • Form validity = all fields pass validation

Use useMemo if the computation is expensive. For simple derivations, just compute inline.

URL as State (Deep Dive)

For filters, sorting, pagination, and search — use the URL. This gives you shareability, bookmarkability, and proper back/forward navigation for free.

Complete URL state pattern with TanStack Query
import { useSearchParams } from 'react-router-dom'
import { useQuery } from '@tanstack/react-query'

interface Filters {
  search: string
  category: string
  sort: 'name' | 'price' | 'rating'
  page: number
}

// 1. Read filters from URL
const useFilters = (): Filters => {
  const [params] = useSearchParams()
  return {
    search: params.get('search') ?? '',
    category: params.get('category') ?? 'all',
    sort: (params.get('sort') as Filters['sort']) ?? 'name',
    page: Number(params.get('page')) || 1,
  }
}

// 2. Update filters in URL
const useUpdateFilters = () => {
  const [, setParams] = useSearchParams()

  return (updates: Partial<Filters>) => {
    setParams((prev) => {
      const next = new URLSearchParams(prev)
      Object.entries(updates).forEach(([key, val]) => {
        if (val) next.set(key, String(val))
        else next.delete(key)
      })
      // Reset to page 1 when filters change
      if (!('page' in updates)) next.set('page', '1')
      return next
    })
  }
}

// 3. Use filters in a query
const ProductList = () => {
  const filters = useFilters()
  const updateFilters = useUpdateFilters()

  const { data, isLoading } = useQuery({
    // Query key includes filters — auto-refetches on change
    queryKey: ['products', filters],
    queryFn: () => fetchProducts(filters),
  })

  return (
    <div>
      <input
        value={filters.search}
        onChange={(e) =>
          updateFilters({ search: e.target.value })}
      />
      {/* URL updates, query refetches, UI updates */}
    </div>
  )
}
Note: When filters are in the URL and the query key includes the filters, TanStack Query automatically refetches when the URL changes. No manual coordination needed.

Zustand + TanStack Query

These two tools complement each other. The key rule: never duplicate server data in Zustand.

Data Type Tool Example
Server data TanStack Query Product list, user profile, comments
Client-only UI state Zustand Shopping cart, sidebar open, theme
Shareable/bookmarkable URL params Search, filters, pagination
DON'T: Duplicate server data
// Zustand store with server data — BAD
const useProductStore = create((set) => ({
  products: [],        // server data!
  isLoading: false,    // server state!
  error: null,         // server state!
  fetchProducts: async () => {
    set({ isLoading: true })
    try {
      const data = await fetch('/api/products')
      set({ products: data, isLoading: false })
    } catch (err) {
      set({ error: err, isLoading: false })
    }
  },
}))

No caching, no dedup, no background refetch, no stale-while-revalidate. You're reimplementing TanStack Query poorly.

DO: Each tool does its job
// TanStack Query: server data
const { data: products } = useQuery({
  queryKey: ['products'],
  queryFn: fetchProducts,
})

// Zustand: client-only state
const useCartStore = create(() => ({
  items: [],  // client state only!
  addItem: ...,
}))

// Component combines both:
const ProductCard = ({ product }) => {
  const addItem = useCartStore((s) => s.addItem)
  return (
    <button onClick={() =>
      addItem(product)}>
      Add to Cart
    </button>
  )
}

Slice Pattern

When a single store grows large, split it into slices — separate files that each define a portion of the state. Then compose them into one store.

src/stores/
  useAppStore.ts    // combines slices
  slices/
    authSlice.ts    // user, login, logout
    uiSlice.ts      // sidebar, theme
    notificationSlice.ts // toasts 
// slices/authSlice.ts
import type { StateCreator } from 'zustand'

export interface AuthSlice {
  user: User | null
  login: (user: User) => void
  logout: () => void
}

export const createAuthSlice: StateCreator<
  AuthSlice & UISlice,  // full store type
  [],
  [],
  AuthSlice             // this slice's type
> = (set) => ({
  user: null,
  login: (user) => set({ user }),
  logout: () => set({ user: null }),
})

// slices/uiSlice.ts
export interface UISlice {
  sidebarOpen: boolean
  toggleSidebar: () => void
}

export const createUISlice: StateCreator<
  AuthSlice & UISlice, [], [], UISlice
> = (set) => ({
  sidebarOpen: false,
  toggleSidebar: () =>
    set((s) => ({ sidebarOpen: !s.sidebarOpen })),
})

// useAppStore.ts — combine slices
import { create } from 'zustand'
import { createAuthSlice, type AuthSlice } from './slices/authSlice'
import { createUISlice, type UISlice } from './slices/uiSlice'

export const useAppStore = create<AuthSlice & UISlice>()(
  (...args) => ({
    ...createAuthSlice(...args),
    ...createUISlice(...args),
  })
)
Tip: Slices within one store can access each other's state via get(). But if the domains are truly independent, prefer separate stores over slices. Slices are for when domains need to interact.

State Normalization

Flatten nested data structures. Use IDs to reference entities, like rows in a database table. This prevents deep nesting, makes updates O(1), and avoids duplication.

Nested (denormalized)
// Deeply nested — hard to update
interface State {
  posts: {
    id: string
    title: string
    author: {
      id: string
      name: string  // duplicated!
    }
    comments: {
      id: string
      text: string
      author: {
        id: string
        name: string // duplicated!
      }
    }[]
  }[]
}

// Updating author name = find every
// occurrence and update. Easy to miss one.
Flat (normalized)
// Like database tables
interface State {
  users: Record<string, User>
  posts: Record<string, {
    id: string
    title: string
    authorId: string    // FK
    commentIds: string[] // FK[]
  }>
  comments: Record<string, {
    id: string
    text: string
    authorId: string    // FK
  }>
}

// Update author name = one operation
// state.users[id].name = "New Name"
Python / SQL Analogy

This is exactly how you'd model it in PostgreSQL:

-- Separate tables with foreign keys
CREATE TABLE users (
  id UUID PRIMARY KEY,
  name TEXT
);
CREATE TABLE posts (
  id UUID PRIMARY KEY,
  title TEXT,
  author_id UUID REFERENCES users(id)
);
CREATE TABLE comments (
  id UUID PRIMARY KEY,
  text TEXT,
  post_id UUID REFERENCES posts(id),
  author_id UUID REFERENCES users(id)
);
TypeScript equivalent
// Record<string, T> = dict[str, T] in Python
// It's a hash map keyed by ID

type NormalizedState = {
  entities: {
    users: Record<string, User>
    posts: Record<string, Post>
  }
  ids: {
    users: string[]  // preserves order
    posts: string[]
  }
}

// Lookup is O(1): state.entities.users[id]
// List render: state.ids.users.map(id =>
//   state.entities.users[id])
When to normalize: When you have entities referenced by other entities (users, posts, comments). For simple lists without relationships, flat arrays are fine. Don't over-engineer.

Pattern Summary

Pattern Problem It Solves Key Idea
Optimistic updates Slow-feeling UI on mutations Update UI first, revert on error
Derived state Sync bugs from redundant data Compute, don't store
URL as state Non-shareable, non-bookmarkable views Put filters/search/pagination in URL
Zustand + Query separation Duplicating server data in client stores Query for server data, Zustand for UI state
Slice pattern Large, unwieldy stores Split by domain, compose into one store
Normalization Deeply nested data, duplicate entities Flatten like DB tables, reference by ID
